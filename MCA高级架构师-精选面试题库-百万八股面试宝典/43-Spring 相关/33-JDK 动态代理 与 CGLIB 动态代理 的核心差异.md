#### 1. 代理机制本质

- **JDK 动态代理** 基于接口，使用 `java.lang.reflect.Proxy` 和 `InvocationHandler`，代理的目标必须实现至少一个接口。
- **CGLIB 动态代理** 不依赖接口，通过继承方式生成目标类的子类，并在运行期修改字节码，生成代理对象。

#### 2. 适用场景

- JDK 代理：目标实现接口时使用。轻量且无需额外依赖，是默认优选机制。
- CGLIB 代理：目标类无接口时自动采用，或显式设置 `proxy-target-class=true` 强制使用。

#### 3. 性能比较

- **创建性能**：JDK 代理速度快，占内存少；CGLIB 创建稍慢，生成的代理类较大。
- **调用性能**：CGLIB 方法调用优秀，因为直接通过子类调用，无反射开销；JDK 代理调用需反射，略慢。

#### 4. 限制与注意事项

- **JDK 代理**：只能代理接口方法，调用 `InvocationHandler.invoke(...)`，无类代理能力。
- **CGLIB 代理**：通过继承实现，仅支持非 `final` 类与非 `final` 方法；内部方法调用无法触发增强。

#### 5. Spring 默认行为

- 若目标实现接口 → 使用 JDK 动态代理；
- 若目标无接口 → 使用 CGLIB；
- Spring Boot 默认启用 CGLIB（classpath 中引入，并且通常自动配置为 `proxy-target-class=true`）。

### 总结建议S

|  |  |  |
| --- | --- | --- |
| **特性** | **JDK 动态代理** | **CGLIB 动态代理** |
| 基础机制 | 接口 + 反射 | 字节码增强 + 子类继承 |
| 是否依赖接口 | 是 | 否 |
| 创建开销 | 较低 | 较高 |
| 调用性能 | 略逊（反射调用） | 较优（直接调用） |
| 继承支持 | 不支持子类策略 | 支持，但类与方法不可为 final |
| Spring 默认 | 接口存在时优选 | 无接口时或显式配置使用 |

JDK 代理接口优先、CGLIB 处理类，性能对比和适用场景各有侧重。Spring 默认按需选择，Spring Boot 常用 CGLIB。 B
