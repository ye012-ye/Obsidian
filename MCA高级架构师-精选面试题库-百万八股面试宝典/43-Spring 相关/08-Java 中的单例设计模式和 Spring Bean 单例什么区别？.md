在 Java 和 Spring 中，“单例”的含义和作用域截然不同，具体体现在以下几点：

### 一、Singleton 模式（GoF 单例）

- **目的**：确保一个类在整个 JVM 中仅存在一个实例，常用于数据库连接、日志管理器、资源集中控制等场景。
- **实现方式**：私有构造器 + `private static` 实例 + `public static` 获取方法，通常结合双重检查锁定实现线程安全。
- **特点**：

- 生命周期与 ClassLoader 绑定，一个 ClassLoader 只加载一个实例。
- 无法创建多个实例，除非使用反射或反序列化破坏。M
- 属于设计层面的逻辑约束，由类自我管理创建和访问权限。S

### 二、Spring Bean 单例（Singleton 作用域）

- **目的**：在一个 Spring 容器（ApplicationContext）内，对应一个 Bean 标识只创建一个对象，用于容器内共享资源。
- **实现方式**：由 IoC 容器控制，默认作用域，不需要开发者显式实现单例逻辑。
- **特点**：

- 作用域局限于容器：容器A 和容器B 中同一个 Bean 类可各自拥有一个实例；
- 可以为同一个类配置多个 Bean（不同 ID），每个 ID 对应一个对象；
- 更灵活、易测试，也可配合原型作用域使用。B

### 对比

|  |  |  |
| --- | --- | --- |
| **方面** | **GoF 单例** | **Spring 单例 Bean** |
| 实例唯一性 | JVM + ClassLoader 全局唯一 | 每个容器、每个 Bean ID 唯一 |
| 创建方式 | 静态工厂或懒加载 + private 构造 | 由容器反射或配置文件自动创建 |
| 生命周期 | 与 ClassLoader 同步销毁 | 随容器启动销毁 |
| 测试 & 可扩展性 | 不易测试、强耦合 | 易 mock 和替换 |
| 同一类多实例情形 | 不支持 | 支持（不同 ID） |
