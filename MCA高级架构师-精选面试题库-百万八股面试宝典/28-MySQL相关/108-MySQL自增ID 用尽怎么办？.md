如果你为一个表设计了 `AUTO_INCREMENT` 整型字段，使用例如 `INT UNSIGNED` （最大值 2³² − 1）或 `BIGINT`（最大值 2⁶⁴ − 1），那么在插入操作中，当自增计数器到达类型上限时，下一次插入将会失败。MySQL 不会自动回绕，而是尝试分配下一个值时，发现已是最大值，就报主键冲突错误（`Duplicate entry 'max' for key 'PRIMARY'`），导致插入操作被拒绝。这种情况并不会重置自增值，后续插入依然无法进行，所以一旦出现，就需要手动干预。M

​

为了避免类似问题，一般有两种方式应对：其一，预防式处理，在表建好之后就考虑可能的上限风险，将 ID 类型设为合适大小，比如从 `INT` 升级为 `BIGINT UNSIGNED`，这样在常规业务中几乎不会遇到上限问题。其二，当确实快到达上限时，可以通过 `ALTER TABLE ... MODIFY id BIGINT UNSIGNED AUTO_INCREMENT` 来扩容数据类型，但这种操作会导致表结构重建，对大表影响较大。S

​

另一方面，对于那些没有显式主键、依赖 InnoDB 内部 row\_id 的表，如果 row\_id 用尽（这是一个 6 字节字段，最大为 2⁴⁸ − 1），InnoDB 会发生 wrap-around：当内部生成的 row\_id 达到上限后，将从 0 开始继续分配。新插入行如果与已有行 row\_id 相同，将覆盖旧数据，这显然带来严重数据丢失风险。这种循环覆盖行为远比自增主键冲突更危险，因此强烈建议明确指定主键而不是依赖隐式 row\_id 。B
