MVCC（Multi-Version Concurrency Control 多版本并发控制）是一种通过维护多个数据版本，支持高并发读取同时保障数据一致性的机制。其核心原理及应用如下：M

### 1. ​多版本数据结构

- 每行数据不覆盖式更新，而是在更新时新增版本，旧版本保留，并带有版本字段（如 `xmin/xmax` 或事务 ID 时间戳）。
- 读取时，根据事务自身的时间戳选取最合适版本，可实现一致的“快照读”。

### 2. 并发读写不互阻

- **读不阻塞写**：读请求查看旧版本，不需等待写锁；
- **写不阻塞读**：写操作产生新版本，旧版本继续服务读事务。S  
  这种机制显著降低了锁竞争，使并发性能大幅提升。

### 3. ​快照隔离与事务可见性

- 每事务开启即确定“快照时间点”（start timestamp），仅能看到在此之前提交的版本；
- 提交时，通过冲突检测确保不违反隔离规则（如快照隔离）。
- 在 PostgreSQL 中，重复读隔离下，操作期间的视图保持稳定；MySQL InnoDB 支持 Snapshots with Consistent Reads 。

### 4. ​版本垃圾回收机制

旧版本不会一直存在，需定期触发 **垃圾回收（VACUUM、UNDO purge）**，否则会造成存储膨胀与性能问。B

### 5. 优点与局限

|  |  |
| --- | --- |
| 优点 | 局限 |
| 读写并发性能优，减少锁等待 | 写写冲突需回滚重试 |
| 支持快照隔离，一致性保障好 | 存储占用高，需回收机制 |
| 降低死锁几率、读性能优 | 写偏斜等隔离异常可能存在 |

系统如 MySQL InnoDB、PostgreSQL、Oracle 等广泛使用 MVCC，实际实现略有差异，但都基于上述核心机制。
