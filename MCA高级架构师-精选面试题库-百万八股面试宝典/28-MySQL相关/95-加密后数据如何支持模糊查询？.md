以下是几种常见解决方案：

### 1. **维护明文–加密映射表（映射查询）**

为频繁模糊搜索的字段（如手机号），维护一个映射表：明文 + 加密值。查询时先在映射表 `LIKE '%123%'` 获取主键，再从主表查加密字段。  
**优点**：易实现，查询效率高。  
**缺点**：字段明文仍以明文形式存在，安全性有限。

### 2. **数据库解密后模糊（内存解密）**

将所有加密数据取出在应用层解密后进行模糊匹配。  
**优点**：实现简单。  
**缺点**：对大数据量场景全表扫描，极易 OOM，性能瓶颈严重。

### 3. **使用加密函数临时解密模糊查询**

利用数据库支持的加密函数（如 AES\_DECRYPT）对查询字段进行解密后模糊：

```sql
SELECT *  
FROM users  
WHERE AES_DECRYPT(encrypted_mobile, key) LIKE '%123%';
```

**优点**：无需额外字段，语句直接。  
**缺点**：无法利用索引浏览，性能较差，依赖 DB 支持。

### 4. **分片加密 + 拼接索引（推荐）**

将字段按固定长度拆分（如每4字符），对每片分别加密后拼装为一串，并为其创建索引。  
查询时只需将搜索片（如 "1234"）加密，比对拼接后的密文字段：

```sql
WHERE encrypted_shards LIKE '%<cipher of 1234>%'
```

**优点**：可支持索引，加快查询。  
**缺点**：占用空间更大、实现较复杂，对非整片（如 "123"）查找可能失败。

### 5. **可搜索加密 & 同态模糊索引**

- **可搜索加密**（Searchable Encryption）结合 N‑gram、LSH 或Trie结构，实现模糊匹配而不解密。学术方案已有，但实现复杂、资源开销大。
- **MongoDB Queryable Encryption** 是业界首个落地框架，支持加密字段查询，兼顾性能与安全。
