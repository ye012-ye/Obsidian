MySQL 默认隔离级别为 RR，虽然能解决 **不可重复读** 和 **幻读**，但在高并发环境下也带来性能问题。大厂改为 RC，主要出于以下考虑： M

### 主要原因

1. **减少死锁与锁争用**  
   RR 级别使用 **间隙锁 (gap lock)**，在批量删除或范围操作时极易导致死锁。改为 RC 后，InnoDB 不再使用 gap lock，显著降低了死锁率和锁等待时间。S
2. **降低 MVCC 版本维护成本**  
   RR 需为整个事务维护一致性快照，造成长期版本依赖；RC 则为每条语句生成快照，减少 undo 版本存储压力，减少 I/O 和内存占用 。
3. **提升并发吞吐量**  
   RC 下读操作不会持久保持锁，也不增加对版本的占用，可快速读取提交数据，提高并发性能，适用于读多写少的互联网业务场景。B
4. **业务容错可接受**  
   多数互联网业务（例如电商、推荐系统）允许出现 **不可重复读**，着重性能优先。对关键事务可临时使用更高隔离级别，整体上采用 RC 更灵活有效。
