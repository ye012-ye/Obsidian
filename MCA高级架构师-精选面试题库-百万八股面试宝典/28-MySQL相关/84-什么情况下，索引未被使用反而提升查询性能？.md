尽管索引通常能加速查询，但在特定场景下，查询优化器反而会跳过索引，选择全表扫描，这样反而更高效。以下是几种常见情形：m

### 1. 小表或全表读取

对于极小的数据表，索引查找反而要额外进行随机 I/O，速度可能不如一次顺序读取。查询优化器会直接全表扫描，因为成本更低。例如：

```sql
SELECT * FROM tiny_table WHERE col = 'x';
```

### 2. 读取大比例数据

当查询需要扫描表中较大比例的记录（如超过30%–50%），通过索引查出的结果依然要频繁回表读取数据页，全表扫描一次性顺序读反而更快 。S

### 3. 索引选择性低

如布尔或性别字段，仅区分两三个值，索引覆盖行数多且回表成本高，优化器会倾向于全表扫描。

### 4. 回表开销大

对于多列、随机访问的索引，读取数据时会导致缓存频繁替换。尤其是非聚簇索引，回表带来的随机 I/O 可能使索引查找比全表扫描慢一倍以上 。

### 5. 优化器误判或统计失真

少数情况下，索引统计信息不准或优化器估算有误，选择索引反而比实际慢。禁用指定索引或使用查询提示反而能提升性能。B
