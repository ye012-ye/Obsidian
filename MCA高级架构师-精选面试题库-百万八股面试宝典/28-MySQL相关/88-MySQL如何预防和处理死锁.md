MySQL的InnoDB存储引擎会自动检测死锁，并回滚其中一个事务，但要从根本减少死锁发生，我们可以从以下几个方面着手：

### 1. 固定锁顺序

确保多个事务访问相同表或相同行时，按照一致顺序加锁。  
**示例**：若事务A与B都更新表T1和T2，应统一执行 `UPDATE T1 → UPDATE T2`，避免出现“A锁T1再锁T2”与“先锁T2再锁T1”的循环等待。

### 2. 缩小事务范围与粒度

保持事务短小，在必要时才打开事务，减少锁持有时间。避免在事务中执行长逻辑或用户交互操作。M

### 3. 优化索引 & 查询

确保 `WHERE` 条件、`JOIN`、`ORDER BY` 涉及的列都有合适索引。这样查询只锁必要行，减少锁冲突。还可使用 `SELECT … FOR UPDATE` 或事务隔离级别为 READ COMMITTED 来降低锁级别。

### 4. 限制锁范围

尽量仅锁所需行，避免使用 `LOCK TABLES`。若确实需要，用行级锁而非表级锁，并及时提交事务。S

### 5. 设定合理超时并自动重试

配置 `innodb_lock_wait_timeout`，避免长时间等待。应用层应捕获死锁异常（如 Error 1213），对被回滚事务进行适当延迟后重试。

### 6. 监控日志与死锁追踪

开启 `innodb_print_all_deadlocks` 日志设置，收集所有死锁信息（`SHOW ENGINE INNODB STATUS`）。根据日志分析频繁死锁的 SQL、表、锁类型，从根源调优。B
