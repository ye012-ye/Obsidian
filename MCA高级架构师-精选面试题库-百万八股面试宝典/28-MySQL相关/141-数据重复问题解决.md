问题： 为何 MySQL 唯一索引允许多个 `NULL` 值，从而在含 `NULL` 的组合键上仍会出现“重复”数据？你会如何规避这一问题？M

在 MySQL 中，`UNIQUE` 索引在含有 `NULL` 的列上表现异常：允许多条包含 `NULL` 的记录共存，因此即使组合键里其中一项为 `NULL`，也不会触发唯一性约束。主要原因及应对策略如下：

# 原因分析：

1. SQL 标准将 `NULL` 视为“未知值”，不同于常规值。
2. MySQL（常见存储引擎，如 InnoDB）允许多个 `NULL`。也就是说，只要索引列中出现 `NULL`，该列就不会触发重复错误。
3. 这是 ANSI/ISO SQL 标准中认可的行为（多个 `NULL` 不被视为相等）S

​

# 解决方案：

1. **强制非 NULL**  
   将关键字段设置成 `NOT NULL`，并提供默认值（如 `''`、`0` 或特殊标志）。这样所有重复都会被拦截。
2. **替代值方式**  
   使用 `IFNULL` 或触发器，将 `NULL` 转为确定值后，再建唯一索引，保证约束有效。
3. **虚拟列 + 索引**  
   示例：添加一个虚拟列

```sql
ALTER TABLE t 
  ADD not_del GENERATED ALWAYS AS (IF(del_time IS NULL, 1, NULL)) VIRTUAL, 
  ADD UNIQUE idx_name_notdel(name, not_del);
```

这样，对非删记录唯一，对已删记录允许重复。

4. **逻辑删除状态 + 标识**

- 方法一：给逻辑删除表多次删除递增状态（`delete_status` 从 0、1、2 增长），每次都不同，保证组合唯一。
- 方法二：添加时间戳或 `delete_id` 字段，组合唯一索引含其值，确保每次删除都不冲突。B
