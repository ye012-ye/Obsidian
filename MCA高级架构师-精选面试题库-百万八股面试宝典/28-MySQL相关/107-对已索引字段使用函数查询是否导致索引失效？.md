在 MySQL 中，并非所有对索引列使用函数的场景都会造成索引失效，但多数情况下涉及函数的过滤条件会让索引无法发挥效能。其原因在于索引中保存的是原始数据，若查询条件中使用了函数，DB 引擎无法直接匹配这些已变更的值，从而退化为全表扫描。

# 为什么函数会让索引失效

- 索引是按照列原始值建立的 B‑Tree 结构，函数变换后，数据库就找不到匹配路径，必须遍历所有行并执行函数判断，性能下降显著 。
- 举例来说，`WHERE YEAR(created_at)=2023` 即使 `created_at` 有索引，也会失效。理想写法应为：

```plsql
SELECT ... FROM table
WHERE created_at BETWEEN '2023-01-01' AND '2023-12-31';
```

此方式可直接利用索引范围扫描。

# 能保留索引的情况

- 当函数出现在 SELECT 列表、排序或聚合中，而非 WHERE/ON 中时，不影响索引使用
- MySQL 8.0.13+ 支持函数索引，可为表达式如 `LOWER(username)` 建立索引，使查询 `WHERE LOWER(username) = 'abc'` 利用索引。

# 优化方案

1. **范围替代函数**

- 改写为范围查询，如 `date_column BETWEEN x AND y`，而非 `YEAR(date_column)=y`。

2. **虚拟列 + 索引**

- 在 MySQL 5.7+ 使用虚拟生成列：

```sql
ALTER TABLE users 
ADD COLUMN username_lower VARCHAR(255) 
  AS (LOWER(username)) STORED, 
ADD INDEX (username_lower);
```

3. **函数索引（MySQL 8.0.13+）**

- 如：

```sql
ALTER TABLE items ADD INDEX ((MONTH(created_at)));
```

4. **尽量前置处理**

- 将可变换操作推到应用层，减少 SQL 中的函数依赖。
