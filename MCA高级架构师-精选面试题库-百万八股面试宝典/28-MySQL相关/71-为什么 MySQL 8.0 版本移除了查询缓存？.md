MySQL 自 5.6 起默认禁用查询缓存，并于 8.0 正式移除其功能。核心原因包括以下几点：M

### 1. 可扩展性差与全局锁争用

查询缓存依赖全局锁机制，在写入、更新、删除时必须清空缓存导致严重锁竞争。多核、多线程环境下会成为瓶颈，不利于高并发性能提升。

### 2. 缓存失效频繁

只要底层表有更新，所有相关缓存立即失效。对于写操作频繁的表，缓存效果几乎无效，且不断失效与重建，消耗资源 。S

### 3. 仅接收精确字节匹配

查询缓存仅缓存字节级完全一致的 SELECT 查询，空格、注释、不同字段顺序都会导致缓存不命中，严重限制实用性。

### 4. 不支持非确定性与复杂查询

包含 RAND()、NOW()、临时表、变量或者 JOIN 等复杂语句无法缓存。因此缓存命中场景局限严重。B

### 5. 现代替代方案更灵活

随着硬件升级及应用层缓存的普及，Redis、Memcached、ProxySQL 或 ORM 缓存已成为更优替代方案。MySQL 也推荐使用代理层或应用层缓存，而非内部查询缓存。
