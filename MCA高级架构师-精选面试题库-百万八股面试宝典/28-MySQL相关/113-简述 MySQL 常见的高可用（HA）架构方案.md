在生产环境中，MySQL 高可用性通常通过以下几种架构来实现，各具特点M：

### 1. 主从复制 + 故障转移工具

- **方式**：主库执行写操作，数据通过 binlog 复制到从库。从库可作为读副本或备用主库。
- **组件**：常配合 MHA（Master High Availability）、MHA4MySQL 或 orchestrator 实现主节点故障时自动提升从库为主库。
- **适用场景**：适合架构简单、预算有限的中小型系统，读写分离常用且从库可用于容灾。

---

### 2. 半同步复制

- **方式**：增强主从复制，主库提交事务时等待至少一个从库确认，降低数据丢失风险。
- **特点**：写延迟略有增加，但在异常断电或主库宕机时能保守最新数据。
- **适用场景**：对数据一致性敏感，同时容忍小幅写延迟的系统。S

---

### 3. MySQL Group Replication（MGR）

- **方式**：MySQL 官方提供的多主或单主组复制技术，基于 Paxos/Mencius 协议。
- **特点**：自动选主、故障检测、强一致性。支持单主写、多主写或读写分离配置。
- **适用场景**：面向高可靠性、自动 failover 的应用，特别是云、容器化环境。

---

### 4. InnoDB Cluster（结合 Group Replication + MySQL Router/Shell）

- **方式**：整套官方 HA 套件，包括 MGR、MySQL Shell、MySQL Router。实现自动选主、负载均衡、连接自动路由。
- **特点**：部署和管理简易，支持自动故障恢复。
- **适用场景**：适合希望减少运维复杂度，使用容器或云服务快速构建 HA 架构的团队。

---

### 5. XtraDB/Galera Cluster 或 MySQL NDB Cluster

- **方式**：

- **Galera**（或 Percona XtraDB Cluster）：同步多主复制，事务提交需所有节点达成一致。
- **MySQL NDB Cluster**：共享无结构、自动分片、高吞吐群集。

- **特点**：

- Galera 支持多主写，高一致性，但写性能受制于最慢节点。
- NDB Cluster 可线性扩展、容忍多个节点故障，适合电信级高可用。

- **适用场景**：

- Galera：电商、金融等高可用场景。
- NDB：对极端低延迟、无单点故障和在线扩容有严格需求的环境。B

---

### 6. ProxySQL + VIP/Keepalived + HA 方案组合

- **方式**：应用通过 ProxySQL 连接数据库集群，结合 VIP 或 DNS 实现故障透明切换。ProxySQL 支持读写拆分、健康检查。
- **适用场景**：中大型系统架构，读写分担明确、希望应用层无感知切换的场景。
