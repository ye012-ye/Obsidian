在实际生产环境中，即使对比物理内存大得多的表（如200 GB）进行全表扫描，也不会导致数据库服务器 OOM。原因如下：M

### 1. **“边读边发”机制，不累积大结果集**

MySQL 并不会一次性把所有查询结果加载到内存，而是逐行读入发送缓冲区（`net_buffer_length`，默认约16 KB），当缓冲区满了就立即发送给客户端（包括写入文件或从客户端取走）。如果客户端接收慢，MySQL 会等待而不是继续累积，因此不会因扫描结果过大而占满服务器内存。

### 2. **InnoDB Buffer Pool 不会被全表扫描打爆**

Buffer Pool 是缓存页的“共享池”，大小由 `innodb_buffer_pool_size` 决定。扫描大表确实S会引入大量新页面并可能淘汰未使用的页，但 **扫描过程不会无限制地增加内存使用**。InnoDB 使用“young vs old”分区＋插入时间控制（`innodb_old_blocks_time`）来防止扫描访问刷新常用缓存，从而减轻对业务查询的冲击。

### 3. **扫描虽然有副作用，但不能致内存溢出**

真正可能耗尽内存的操作是设置过大 buffer pool、设置过高连接数和 per-connection buffer 等。B但仅仅是执行全表扫描，不会导致 MySQL 占用超出 buffer pool 限定的内存，也不会触发 OOM。
