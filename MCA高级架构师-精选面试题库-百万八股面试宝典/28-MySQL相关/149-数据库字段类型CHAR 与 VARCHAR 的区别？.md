`CHAR` 与 `VARCHAR` 都是用于存储字符串，但它们在存储机制、空间效率、性能表现以及应用场景上存在显著差异：

### 1. **存储机制**

- **CHAR**：固定长度，定义为 `CHAR(n)` 时，无论实际字符多少，都会**填充空格**至 n 长度；存储空间恒定为 n 字节。
- **VARCHAR**：可变长度，若定义为 `VARCHAR(n)`，则仅存储实际字符长度，加上 1~2 字节长度信息 。

- 当字符数 ≤ 255 时，用 1 字节长度前缀；超出则使用 2 字节。

### 2. **尾部空格处理**

- 存储 `CHAR` 会在末尾自动**右填空格**。取出时默认剔除这些空格。
- `VARCHAR` 不填补空格，除非数据库版本特殊，否则保留用户输入所有字符 。

### 3. **空间占用**

- 若字段长度固定不变，`CHAR` 可简化读写操作；但长度波动大时，会造成磁盘空间浪费。
- `VARCHAR` 根据实际长度存储，节省磁盘空间，尤其适合文本长度不稳定的字段。

### 4. **性能表现**

- 在全行长度固定的表中，`CHAR` 字段访问稍优，因为无需动态计算长度，也利于行内定位 。
- 然而多数现代数据库针对 `VARCHAR` 也有强优化，表设计不当时性能差异非常小 。

### 5. **长度限制**

- `CHAR(n)` 限制通常在 255 字节内（如 MySQL），其他数据库限制略有差异 。
- `VARCHAR(n)` 最大长度随 DBMS 版本不同而不同，例如 MySQL 可上限为 65 535 字节（整行共用），SQL Server 为最多 8000 字节，PostgreSQL 更高等。

​

综上所述, 如果字段长度稳定，使用 `CHAR` 能带来更快访问速率并结构清晰；对于长度不固定、希望节省存储的字段，建议选择 `VARCHAR`。
