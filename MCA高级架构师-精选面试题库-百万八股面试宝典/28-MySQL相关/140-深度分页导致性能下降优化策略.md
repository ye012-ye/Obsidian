在 MySQL 中，深度分页（如 `OFFSET 1000000,10`）性能急剧下降，主要是因为：MySQL 会扫描并跳过前面所有符合条件的行才返回后续记录，导致大量 I/O 开销和 CPU 消耗。

有如下几点优化策略：

### 一、使用索引子查询 + JOIN（Indexed Subquery + JOIN）

**思路**：先用索引快速定位所需页的主键列表，再用主键去主表查询完整数据。

- 举例：

```sql
SELECT * FROM t 
JOIN (
  SELECT id FROM t ORDER BY id LIMIT 1000000, 10
) AS sub USING(id)
ORDER BY id;
```

- **适用场景**：需要指定任意页、但仍有索引覆盖排序列时，可显著减少扫描行数 。

---

### 二、游标/Keyset 分页（Cursor/Keyset Pagination）

**思路**：不使用 OFFSET，而是记录上次最后一条记录的排序键（如 `id` 或 `(create_time, id)`），下次查询时加 `WHERE id > last_id`。

- 优点：查询每页开销为 O(log N)，性能稳定。
- 缺点：无法跳转指定页，只能前后翻页，更适用于“下一页/上一页”或无限滚动应用。

---

### 三、延迟 JOIN（Deferred Join）

**思路**：与索引子查询类似，但强调“先按索引定位，再 JOIN 主表”，减少回表次数。

- 示例与第①类似，但强调整合排序设计，适用于多列排序索引策略优化 。

---

### 四、限流与缓存策略

- **前端限制**：限制最大页数（如不支持跳到第100万页），提示“请缩小查询范围”。
- **缓存**：将热门页结果缓存到 Redis/Memcached，避免频繁深度查询；适合热门分页场景。

---

### 五、业务层优化策略

- **预计算/归档**：对历史数据做归档或做物化快照，用专门小表分页。
- **客户端分页**：一次拉取一定范围的数据，在客户端做翻页，适合数据自增长、实时性要求不高的场景。
