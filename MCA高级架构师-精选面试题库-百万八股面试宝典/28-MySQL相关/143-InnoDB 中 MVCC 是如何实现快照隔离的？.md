InnoDB 利用 MVCC（多版本并发控制）在读取数据时避免加锁，通过“快照读”实现每个事务看到一致的数据视图。下面从关键机制逐步解析：

### 1. 隐式字段

每行数据额外包含三个系统字段：

- `DB_TRX_ID`：最近修改此行事务的编号。
- `DB_ROLL_PTR`：指向上一版本的 undo 日志位置。
- `DB_ROW_ID`：隐含行标识，用于无主键时生成聚簇索引。  
  这些字段组成版本链，便于重建历史数据M视图。M

---

### 2. Undo 日志

当事务更新或删除数据时，旧版本被写入 undo 日志，并通过 `DB_ROLL_PTR` 链接成链表。

- **Update** 操作：锁定行 → 旧值写入 undo → 修改数据库 → 更新指针 → 提交释放锁。
- **Delete** 操作：行只是标记删除，旧值保存至 undo。  
  Undo 日志使得快照读能回溯到符合事务视图的版本。S

---

### 3. 读视图（Read View）及快照版本判断

事务第一次执行快照读时会创建一个 Read View，包含：

- 当前活跃事务 ID 列表 (`trx_list`)
- 活跃事务中最小 ID (`up_limit_id`)
- 即将分配的下一个事务 ID (`low_limit_id`)
- 本事务 ID (`create_trx_id`)

**可见性规则**：快照读查看行上 `DB_TRX_ID`，通过以下顺序判断是否可见：

1. 若等于本事务 ID，直接可见；
2. 若小于 `up_limit_id`，表示早已提交，可见；
3. 若大于等于 `low_limit_id`，是后续事务，不可见；
4. 若在 `trx_list` 中是未提交事务，不可见；否则可见。  
   不可见时通过 `DB_ROLL_PTR` 找到旧版本，直至满足可见性。B

---

### 4. 隔离级别下的快照行为区别

- **READ COMMITTED（RC）**：每次快照读都生成新的 Read View，可以读到他人新提交数据。
- **REPEATABLE READ（RR）**：事务生存期间只在首次快照读时生成一次 Read View，后续读使用同一视图，保证一致性，可防不可重复读。

---

MVCC 通过上述机制实现非阻塞读写，提升并发性能。同时可有效避免不可重复读、幻读等问题。但写–写冲突仍需锁机制配合解决，如与悲观锁或乐观锁组合使用。
