MySQL 实现多表连接的主要算法包括以下三类，每种在不同场景下有显著性能差异。

### 1. 嵌套循环连接（Nested Loop Join, NLJ）

- **原理**：外层循环遍历一张表（外表），内层对另一张表（内表）进行全表扫描，也可借助索引进行索引嵌套循环（INLJ），或使用缓冲批量读取（BNLJ）以减少 I/O
- **适用场景**：外表较小且内表有合适索引，OLTP 查询场景较多使用这种方式。
- **优势**：实现简单，对小表性能优良。M
- **劣势**：数据量大时性能退化严重，I/O 代价高。

---

### 2. 哈希连接（Hash Join）

- **原理**：针对等值连接，先从小表构建哈希表（build），然后探测大表（probe）匹配键值
- **适用场景**：两表较大且没有合适索引时非常高效；MySQL 从 8.0.18 起引入此优化
- **优势**：等值连接性能优，尤其无索引、数据庞大的场景；查询速度快。
- **劣势**：需要足够内存；不适用于非等值连接（如范围连接）。S

---

### 3. 排序合并连接（Sort-Merge Join）

- **原理**：先对两表连接列排序，然后依次扫描进行合并匹配
- **适用场景**：两表连接键已有索引或已排序，且连接列涉及范围、排序、分组等操作时效果显著。
- **优势**：适用等值和范围连接，高效处理大量数据，尤其是在索引或排序已准备好的场景。
- **劣势**：排序阶段可能大量 I/O；对排序支撑要求较高。B

​

对比如下：

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 算法 | 连接条件 | 内存需求 | I/O 开销 | 适合场景 |
| 原始 NLJ | 任意 | 低 | 高 | 小表连接、小事务 |
| INLJ / BNLJ | 存在索引或批量情况 | 中等 | 中 | 外表小、内表大且有索引或批量优化 |
| Hash Join | 等值连接 | 高 | 低–中 | 大表等值连接，无索引最佳 |
| Sort‑Merge Join | 等值或范围连接 | 中等 | 中高 | 已排序/索引，或关联范围、排序规则 |

**推荐选择策略**

- OLTP 负载（小表 + 索引）：使用 **INLJ** 或 **简单嵌套循环**。
- 大表等值连接且缺少索引：优选 **Hash Join**。
- 范围查询或排序查询：优先考虑 **Sort‑Merge Join**（尤其有索引时）。
