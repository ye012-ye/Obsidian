在 MySQL 中使用 `IS NULL` 条件时，InnoDB 会将其视为等值条件进行优化，与 `= constant` 类似。这意味着，如果字段上有索引，MySQL 可以利用 B+ 树完成索引扫描，从而直接定位到存储 `NULL` 的记录，而不是进行全表扫描。但是，当该字段被定义为 `NOT NULL` 时，`IS NULL` 条件永远不成立，优化器会识别这个“永假”条件，将其从执行计划中移除，跳过索引扫描并直接返回结果 。M

同样，`IS NOT NULL` 表达式也并不总是绕过索引。MySQL 的优化器会根据字段中非空值的分布情况以及估算的成本来决定是否使用索引。若列中绝大多数行都是非空，则索引命中率低，反而全表扫描效率更高；优化器因此会选择不使用索引。但如果非空值很少，过滤选择性高，那么索引扫描反而更有利，优化器会优先选择索引路径 。S

例如，对于字段允许 `NULL` 的场景，执行 `SELECT … WHERE col IS NULL` 时，只要该列上有索引，MySQL 就能通过索引快速定位 NULL 值行，即使该表较大也能实现快速过滤。但如果你执行 `SELECT … WHERE col IS NOT NULL`，并且绝大多数行都匹配，那优化器会判断索引带来的回表成本高于全表扫描效率，因此直接扫描整个表，而不是通过索引。

此外，InnoDB 也支持 `col IS NULL OR col = const` 这类混合条件的索引优化，执行计划中会标记为 `ref_or_null`，意味着先读取等值记录，再搜索 NULL，仍能高效利用索引。

综上所述，`NULL` 和 `NOT NULL` 并不会绝对导致索引失效。MySQL 会根据字段属性、数据分布和查询选择性，动态决策是否使用索引。当字段为 `NOT NULL` 且查询是 `IS NULL` 时，优化器会识别并略过；而在 `IS NOT NULL` 场景下，则会灵活权衡全表扫描与索引访问的优劣。B
