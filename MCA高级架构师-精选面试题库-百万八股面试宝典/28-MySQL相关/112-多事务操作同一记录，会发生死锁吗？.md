在 InnoDB 中，即使多个事务仅针对同一行记录进行操作，也可能出现死锁。这与其内部锁机制相关：M

### 1. 锁类型与顺序冲突

- **相同记录上的共享锁（S）与排它锁（X）升级冲突**  
  当事务先获取了 S 锁（如 `SELECT … FOR UPDATE` 或查询判断），随后尝试升级为 X 锁时，如果另一个事务已持有 S 锁，可能发生等待—同时该事务在等待升级，形成死锁。

### 2. 不同索引路径导致锁顺序不一致

- 若 WHERE 条件使用不同索引定位同一行，不同事务可能按不同索引顺序加锁，造成锁等待环形依赖，进而引发死锁 。

### 3. 插入意向锁（Insert Intention Lock）与自增锁

- 对同一行或相邻 gap 范围内的插入操作，会先获取插入意向间隙锁（Gap + record lock），与另一个正在插入或删除该行的事务发生冲突，出现死锁。
- 使用 `AUTO_INCREMENT` 时，会获取全表锁（auto-inc table lock），多事务并发插入也可能导致循环等待而死锁。

# 解决策略：

|  |  |
| --- | --- |
| 策略 | 说明 |
| 确保统一锁顺序 | 所有事务操作相同行时，尽量固定索引与执行顺序，避免交叉加锁。 |
| 避免先读后写 | 若无需读判断，可使用直接 `UPDATE`  ，避免先获取 S 锁再升级为 X 锁造成冲突。 |
| 缩短事务时长 | 尽快 `COMMIT`  ，缩短锁持有时间，减少锁等待概率 。 |
| 捕获死锁并重试 | InnoDB 会自动检测并回滚某事务（错误码 1213），应用需捕获异常，重试逻辑确保最终成功 。 |
| 优化插入逻辑 | 对插入大量数据，配置合适 `innodb_autoinc_lock_mode`  ，或改用批量插入、去重机制减少 lock contention。 |
