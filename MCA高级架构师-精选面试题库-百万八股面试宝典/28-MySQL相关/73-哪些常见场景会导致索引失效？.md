### 一、索引失效常见原因

1. **模糊 LIKE 查询以** `%` **开头**

```plsql
WHERE name LIKE '%abc'
```

这类查询无法利用 B-Tree 索引起始定位，需要完整扫描，建议改为 `abc%` 或使用全文索引。M

2. **在索引字段上使用函数或表达式**

```plsql
WHERE YEAR(created_at) = 2023
```

索引列被函数转换导致失效，推荐改为：

```plsql
WHERE created_at BETWEEN '2023-01-01' AND '2023-12-31'
```

3. **索引字段参与运算或隐式类型转换**

```plsql
WHERE id + 0 = 100
```

或

```sql
WHERE varchar_col = 123
```

会阻断索引匹配，应直接比对，避免变形。

​

4. **复合索引不满足最左前缀原则**  
   `(a,b,c)` 索引，仅 `WHERE b = ?` 无法使用，必须从 a 列开始匹配如 `WHERE a=? AND b=?`。

​

### 二、优化建议

- **规避前缀** `%abc`**：**  
  尽量写作 `abc%` 或使用全文索引、覆盖索引等方式提升性能。
- **避免函数与运算：**  
  将条件改写为直接列比对或范围查询。
- **确保复合索引的合理顺序：**  
  设计时遵循最左前缀原则，切忌单列出现在中间。B
- **查看执行计划与统计信息：**  
  使用 `EXPLAIN` 判断索引是否被考虑，并用 `SHOW INDEX`、`ANALYZE TABLE` 更新统计信息。
- **维护索引可见性与类型一致性：**  
  不要默认索引为 INVISIBLE，JOIN和WHERE字段类型必须统一。
- **慎用低选择性索引：**  
  可结合高选择性列创建复合索引，也可考虑过滤索引或删除无效的索引。
