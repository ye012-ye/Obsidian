以下为系统化排查与优化方案：M

### 1. 更新统计信息

优化器依赖统计决定成本估算。若数据分布发生变化（如数据量剧增、列偏斜），统计失效会误选索引。

- MySQL：执行 `ANALYZE TABLE t;` 更新分区统计。
- PostgreSQL：`VACUUM ANALYZE t;` 或调高 `default_statistics_target`。  
  最佳实践是在业务低峰执行以避免锁表干扰 。

### 2. 修正查询写法

部分写法会导致索引失效：

- 隐式类型转换 (如 `varchar_col = 123`)
- 函数操作列 (如 `YEAR(ts) = ...`)
- 通配符开头的 `LIKE '%...%'` 或 `OR` 条件  
  需改写为显式类型匹配、范围表达式、拆分查询或 UNION，以恢复索引使用。S

### 3. ​ 优化索引结构

确保索引设计符合查询模式：

- 复合索引顺序应先等值再范围列
- 覆盖索引避免回表
- 删除冗余索引，减少优化器干扰

### 4. ​调整优化器参数或提示

当统计与设计仍无法改善时：

- 可使用 MySQL `FORCE INDEX` 或 PostgreSQL `USE INDEX` 提示强制使用目标索引
- 或调整参数：如降低 MySQL `max_seeks_for_key`，PostgreSQL 调整 `random_page_cost` 以影响成本模型

### 5. ​使用函数索引或拆表处理

- 创建虚拟列 + 函数索引（MySQL/G. PostgreSQL）
- 在业务层拆分复杂查询为多个简单查询，逐步拼接结果B
