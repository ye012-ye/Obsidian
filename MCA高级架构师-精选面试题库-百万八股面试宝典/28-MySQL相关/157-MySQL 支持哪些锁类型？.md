MySQL提供多种锁机制保障数据一致性与并发控制，主要有以下几种：M

### 1. 表级锁（Table-level Lock）

在 MyISAM 和 InnoDB（通过 `LOCK TABLES`）中生效，会将整个表加读锁或写锁。读锁允许并发读取，写锁则完全独占，直到事务提交释放。

### 2. 行级锁（Row-level Lock）

**2.1 共享锁（Shared/S 锁）**：用于读操作（如 `SELECT ... LOCK IN SHARE MODE`），允许多个事务同时读取，但阻止写操作。  
**2.2 排他锁（Exclusive/X 锁）**：用于写操作（如 `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`），阻止其他事务的读写冲突请求。

### 3. 意向锁（Intention Locks）

在 InnoDB 中运行行锁时，先在表级打意向锁，表明事务意图在行上施加 S 或 X 锁。意向共享锁（IS）和意向排他锁（IX）用于协调多事务同时在同一表中不同粒度加锁，提升效率并避免冲突。S

​

### 4. 间隙锁（Gap Lock）

针对聚簇索引中两个记录之间的“间隙”加锁，防止其他事务在该区间插入新行，常用于 `REPEATABLE READ` 隔离级别以防止幻读。

​

### 5. 临键锁（Next-Key Lock）

结合记录锁和间隙锁，即锁定具体记录及其左侧间隙，用于应对既要防止记录修改，又防止其他事务在相邻插入的场景，是 InnoDB 默认的锁机制。

​

### 6. 插入意向锁（Insert Intention Lock）

当事务计划在某个间隙插入数据时，会设置轻量级的插入意向锁，多个事务可并发意向不同记录插入，但防止与已有 Gap 或 Next-Key 冲突。

​

以上总结对比如下：B

|  |  |  |
| --- | --- | --- |
| 锁类型 | 作用范围 | 应用场景 |
| 表级锁 | 全表 | DDL 操作、备份、大批量修改等 |
| 行级锁（S/X） | 单行 | 精细化并发控制 |
| 意向锁 | 表级 | 标记行锁意图，协调并发 |
| 间隙锁 | 行间区间 | 防止幻读 |
| 临键锁 | 记录+区间 | 防止幻读及行改插入冲突 |
| 插入意向锁 | 区间内插入 | 优化并发插入冲突处理 |
