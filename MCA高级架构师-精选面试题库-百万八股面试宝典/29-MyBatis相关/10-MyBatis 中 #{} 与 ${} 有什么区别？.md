在 MyBatis 中，`#{}` 与 `${}` 都用于结合参数生成 SQL，但它们的行为和风险截然不同。下面解释：M

#### 1. 参数绑定方式与安全性

使用 `#{}` 时，MyBatis 会将其替换为 `?` 占位符，并通过 JDBC 的 `PreparedStatement` 设置参数值，完成预编译并自动处理特殊字符，从而有效防止 SQL 注入。这是推荐的安全方式。

相反，`${}` 会在 SQL 构建阶段直接用参数值替换标记，不进行转义或预编译。如果参数来源不受控，极易导致 SQL 注入。

#### 2. 使用场景对比

- `#{}`：适用于所有值类型参数，如数字、字符串、日期或布尔值。在绝大多数普通查询中应首选。
- `${}`：仅用于 SQL 结构元素的动态拼接，例如表名、字段名、排序字段等。这类内容无法由 JDBC 占位符处理，因此必须拼接至 SQL 中。

例如：

```xml
<!-- 安全：参数绑定 -->
<select id="find" resultType="User">
  SELECT * FROM users WHERE id = #{userId}
</select>

<!-- 动态列名：结构拼接 -->
<select id="findSorted" resultType="User">
  SELECT * FROM users ORDER BY ${orderField}
</select>
```

S

#### 3. 性能与处理机制

`#{}` 由于使用预编译，SQL 可复用，效率高。每次方法调用只需传入参数，不重新编译 SQL。而 `${}` 生成的是完整 SQL 字符串，每次都编译执行，性能略低。

### ​总结建议

- **尽可能用** `#{}`：安全、性能好、代码健壮。
- 只有当需要动态拼接“结构级别”内容（如表名、列名、排序字段）时，才使用 `${}`，且务必确保参数来源可信、受控。
