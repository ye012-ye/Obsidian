MyBatis 和 JPA 代表两种不同的数据持久化方式：SQL 驱动 vs 实体驱动，各有侧重。

M

#### ​1. SQL 控制与抽象层级

MyBatis 用明确的 SQL 映射 Java 方法，开发者手写查询和优化，保留对 SQL 的完全掌控，更适用于复杂或性能敏感场景。JPA 基于实体类与数据库表的映射，自动生成 SQL，通过 JPQL 查询实体，开发更简洁但 SQL 控制有限，排查复杂 SQL 较为困难。

#### ​2. ORM 实现 vs 数据映射

JPA 是 ORM 规范（如 Hibernate 实现），强调自动级联、状态跟踪、懒加载等功能，适用于将整个领域模型持久化。MyBatis 则为半 ORM，只将 Java 方法与 SQL 映射，POJO 无需维护持久化状态，领域模型更纯粹且不被侵入。

S

#### ​3. 灵活性与表现力

MyBatis 可以无缝对接存储过程、视图、数据库特性，并支持动态 SQL、ResultMap、缓存机制，特别适应非规范化、遗留结构数据库。JPA 更适合规范化、可变较少的领域模型，但在复杂查询或多表联查时可能需写原生 SQL。

B

#### ​总结对比：

|  |  |  |
| --- | --- | --- |
| **特性** | **MyBatis** | **JPA (Hibernate)** |
| SQL 控制 | 手写原生 SQL，灵活可控 | 自动生成 SQL，控制度低 |
| ORM 支持 | 半 ORM，只映射方法与 SQL | 全 ORM，支持实体状态管理与关系映射 |
| 查询复杂度 | 支持复杂 SQL 和存储过程，轻松调优 | 复杂 JPQL 或 Native SQL 支持有限 |
