NGINX 采用的是一种 **事件驱动 + 异步非阻塞 I/O** 的架构，结合 master-worker 模型，能够高效应对海量连接。以下是Nginx架构特点：M

### 1. Master–Worker 模式

- **主进程（master）**：负责读取配置、绑定端口、启动/监控 worker 进程等。
- **子进程（worker）**：每个绑定于单个 CPU 核心，独立处理所有网络事件，避免了线程/进程频繁创建和销毁的开销。

### 2. 事件循环 + Reactor 模式

- 每个 worker 进程运行一个核心 **事件循环**，通过系统调用如 `epoll`（Linux）或 `kqueue`（BSD）监听 socket 可读写事件。
- 当有连接事件触发（如 accept、read、write），由事件循环统一调度处理，不占用阻塞等待。

### 3. 非阻塞 I/O

- worker 进程不会因某个连接等待 I/O 而被阻塞，而是注册相关事件后继续处理其他连接，一旦事件就绪则通过回调处理 。S

### 4. 资源节省与低开销

- 使用少量固定线程／进程，但却能处理数万乃至十万的并发连接，极大节约了内存与上下文切换成本。
- worker 进程多为单线程模型，绑定到 CPU 核心后可保持缓存命中率高，切换开销低。

### 5. 高并发与可扩展性

- 单 worker 可支持数十万连接并处理请求，而 NGINX 可通过增加 worker 数（通常等于核心数）实现 **线性扩展**，满足 C10k、C100k 级别高并发需求。
- 在负载激增时仍能快速响应，没有线程耗尽或响应阻塞问题。

### 总结：

NGINX 的事件驱动架构通过 master-worker、非阻塞 I/O 和 Reactor 模式结合，显著降低资源开销、减少上下文切换，并允许单个进程高效管理成千上万连接。这种设计模式使其在高并发环境中具有极高的性能优势和响应效率，是现代互联网流量处理的优秀选择。
