线程池大小的设置，应根据任务类型与系统资源综合考虑。在 Java 中，可参考如下理论与实践指导：

**1. CPU 密集型任务**  
这类任务持续占用 CPU，线程数过多会增加上下文切换开销。建议设置为：

```java
int cores = Runtime.getRuntime().availableProcessors();
corePoolSize = maximumPoolSize = cores + 1;
```

这样可保证 CPU 充分工作，同时预留一个备用线程提高容错能力。M

**2. IO 密集型或混合型任务**  
这类任务大部分时间可能处于等待状态（如网络或磁盘等待），可采用以下经验公式：

```plain
线程数 ≈ CPU 核心数 ×（1 + 等待时间 / 计算时间）
```

其中等待时间是任务阻塞占用时间，计算时间是实际占用 CPU 的时间。若等待与运行等时，则线程数约为 `2 × cores`；若等待比运行时间大，则更适合增加线程数。S

**3. 核设置原则**

- 核心线程数（`corePoolSize`）一般设为稳定处理任务所需的线程数（可等于或略少于最大值）。
- 最大线程数（`maximumPoolSize`）可设置为理论值稍高的上限，用于处理峰值负载。如果核心与最大相同，则线程池行为固定，适用于任务稳定的场景。B

### 示例

```java
int cores = Runtime.getRuntime().availableProcessors();
// 计算密集任务
ThreadPoolExecutor cpuPool = new ThreadPoolExecutor(
    cores + 1, cores + 1, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>());

// IO 密集任务（等待时间约为计算时间）
int optimal = (int)(cores * (1 + 1)); // = 2 * cores
ThreadPoolExecutor ioPool = new ThreadPoolExecutor(
    optimal, optimal * 2, 60, TimeUnit.SECONDS, new ArrayBlockingQueue<>(queueCapacity));
```

### 总结

- 对于 CPU 密集型任务，将线程数设为 CPU 核心数加一可以在保证利用率与降低切换开销之间取得平衡。
- 对于 IO 或混合任务，使用 `cores × (1 + waitTime / computeTime)` 公式计算初步线程数，并通过压测进一步调整。
- `corePoolSize` 设置稳定负载所需线程数，`maximumPoolSize` 设置峰值容纳容量。二者可相同（固定线程池），也可不同（弹性扩容）。
- 最终配置应基于实际性能测试，避免盲目设置过多线程导致资源浪费或上下文切换过度。
