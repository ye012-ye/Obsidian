Java 的线程调度基于操作系统的线程模型，通常采用 **基于优先级的抢占式调度（fixed‑priority preemptive scheduling）**。线程具有 1–10 的优先级，JVM 通常选择优先级最高的线程执行，若优先级相同，则采用类似 FIFO 的机制轮流调度。

### 一、调度机制说明

- Java 线程映射为 **本地操作系统线程**（native threads），完全由操作系统负责调度。（1:1 模型）
- JVM 提交到 OS 的线程在就绪状态时，调度器按照优先级抢占 CPU，优先级高的线程有较大机会被执行。相同优先级线程间通常以先进先出（FIFO）或轮转方式共享 CPU。（若线程主动 `yield()`、`sleep()`、或被阻塞 lock/IO，则 CPU 会分配给其他线程）
- 这是 **抢占式调度** —— 操作系统可在任意时间点中断当前线程并切换到高优先级线程；不同于协作式调度，Java 不依赖线程自己让出 CPU。

### 二、优先级的作用与局限

- 优先级提供一种调度暗示，但 **具体效果取决于底层操作系统实现**。不同平台的调度策略不同，JVM 会尽量尊重优先级，但不能完全依赖这个特性。
- 如果高优先级线程持续就绪，则可能导致低优先级线程无法执行（**线程饥饿**）。一些系统通过“aging”机制动态提升低优先级线程优先级以缓解饥饿问题。
- JVM 规范并不强制保证线程调度的公平性与精确行为开发者需避免依赖调度细节，只能作为“指导”，应使用同步机制保证线程安全。

### 三、简化示例说明

```java
Thread t1 = new Thread(task1);
t1.setPriority(Thread.MAX_PRIORITY); // 10
Thread t2 = new Thread(task2);
t2.setPriority(Thread.MIN_PRIORITY); // 1
t1.start();
t2.start();
// 理论上 t1 优先执行，但 t2 仍有机会执行，具体取决于底层调度器。
```

即使优先级不同，t2 会在 t1 阻塞、休眠或主动让出 CPU 后执行；如果 t1 一直占用 CPU，t2 可能长时间等待。
