Java 自 Java 19 起引入虚拟线程（Project Loom 提供，正式支持于 Java 21），它为并发编程带来了里程碑式的改进，尤其在 Spring Boot 应用场景中展现出明显优势。

M

### 一、虚拟线程的底层原理与资源优势

虚拟线程由 JVM 管理而非直接映射操作系统线程，其栈空间极小、创建和销毁非常轻量，可支持成千上万甚至数百万个线程运行于相对较少的操作系统线程之上。此时 JVM 会将阻塞操作（如 I/O、`Thread.sleep`、阻塞队列等待等）“卸载”到操作系统线程上，释放原本绑定的虚拟线程去调度其他任务，从而极大提高吞吐量和资源利用效率。与此同时，当虚拟线程执行同步代码或 native 方法时，会短暂停留在宿主线程上，但整体影响较小。

S

### 二、在 Spring Boot 中的应用及典型性能表现

Spring Boot 3.2 及 Spring Framework 6 已内建对虚拟线程的支持，可通过配置 `spring.threads.virtual.enabled=true` 或显式提供虚拟线程执行器来启用这一特性。具体配置方式包括使用 `Executors.newVirtualThreadPerTaskExecutor()` 或自定义 `TaskExecutor`。

在一些测试中，针对典型的异步 `@Async` 调用（模拟 50ms 的 I/O 等待），传统平台线程调度数万个任务耗时长达数分钟，而改用虚拟线程后的测试仅需几秒钟完成数万次并发执行，性能提升可达几十倍。

B

### 三、适用场景与限制条件

- **最适合场景**：I/O 密集型应用（如 Web 应用、数据库访问、缓存操作、HTTP 调用等），因为线程阻塞时间多，可充分利用虚拟线程的资源优势，提高并发度和降低延迟。
- **不擅长的场景**：CPU 密集型任务（如复杂计算、批量加密等），虚拟线程并不能绕过 CPU 资源限制，因此收益有限甚至可能与平台线程相当。某些情况下任务池、同步锁或 ThreadLocal 使用也会影响性能和可预测性。
