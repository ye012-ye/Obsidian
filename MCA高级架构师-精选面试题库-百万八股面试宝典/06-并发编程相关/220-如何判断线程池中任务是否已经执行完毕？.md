Java 中判断线程池任务执行完成的方式主要有以下几种，每种方法适用于不同场景：

### 1. `shutdown()` + `isTerminated()` / `awaitTermination()`

调用 `ExecutorService.shutdown()` 后，不再接受新任务。随后可以使用 `isTerminated()` 判断所有任务是否完成，或调用 `awaitTermination(timeout, unit)` 阻塞等待线程池关闭完成。注意，`isTerminated()` 在调用 `shutdown()` 前始终返回 `false`。

**优点**：简单直接。  
**缺点**：需要关闭线程池，适合一次性任务，不适合长生命周期共享线程池。

M

### 2. 使用 `getCompletedTaskCount()` 与 `getTaskCount()` 对比

通过 `ThreadPoolExecutor.getCompletedTaskCount()` 获取已完成任务数量，并与 `getTaskCount()`（提交+执行的总任务数）进行对比。当二者相等时，说明所有任务完成。

**优点**：无需关闭线程池。  
**缺点**：适用于任务数量固定、不再提交新任务的场景，否则结果不准确。

### 3. 使用 `CountDownLatch`

初始化 `CountDownLatch` 的计数值为任务数量，在每个任务执行完成时调用 `countDown()`。主线程使用 `await()` 等待计数器归零，表示所有任务完成。

**优点**：清晰、优雅，适合已知任务总数的场景。  
**缺点**：需要预知任务总数并在任务中显式 `countDown()`，若异常未处理会导致阻塞。

S

### 4. 手工维护计数器

通过线程安全计数（如 `AtomicInteger` 或加锁变量）记录完成任务数量，并在主线程轮询判断是否达到预期数量。

**优点**：灵活，可定制复杂场景。  
**缺点**：实现繁琐，容易出错，需确保线程安全。

### 5. 使用 `Future` 对象检查任务完成状态

提交任务时使用 `submit()` 返回 `Future<?>`，保留所有 `Future` 列表。主线程通过 `isDone()` 非阻塞检查每个任务是否完成，或调用 `get()` 阻塞等待完成。

使用 `invokeAll()` 提交一批 `Callable`，该方法阻塞直到所有任务执行完成，返回 `List<Future<T>>`，每个 `Future` 的 `isDone()` 均为 `true`。

**优点**：适用于需要结果处理或单任务精确控制场景。  
**缺点**：每个任务都需要维护 `Future`，会有额外资源开销。

B

### 对比

|  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- |
| **方法** | **是否关闭线程池** | **需知道任务总数** | **可返回任务结果** | **非阻塞支持** | **适用场景** |
| shutdown + isTerminated/awaitTermination | 是 | 否 | 否 | 部分（await 可阻塞） | 一次性任务批量执行 |
| getCompletedTaskCount + getTaskCount | 否 | 否 | 否 | 是 | 固定任务提交、不新增任务场景 |
| CountDownLatch | 否 | 是 | 否 | 是 | 已知任务数量、需要阻塞主线程等待 |
| 手工计数器管理 | 否 | 是 | 否 | 是 | 灵活场景、任务状态复杂 |
| Future / invokeAll | 否 | 可选 | 可得 | 是 | 需要任务结果、组合控制或回调场景 |
