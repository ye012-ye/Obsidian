在 Java 并发中，`ReentrantLock` 提供了两种获取锁的方式：阻塞获取（`lock()`）与非阻塞获取（`tryLock()`）。这两者的行为和适用场景存在明显差异。

### 一、行为差异说明

- 使用 `lock()` 时，线程会一直等待，直至成功获取锁，过程中持续阻塞，不返回值。该方法适用于必须确保执行临界区的场景，无论等待多久。
- 使用 `tryLock()` 时，线程尝试即时获取锁，如果当前锁被其他线程持有，则立即返回 `false`，不会阻塞线程；若获取成功，返回 `true` 并进入临界区。适用于希望非阻塞执行或避免等待过长的场景。

此外，也可以使用带超时的版本 `tryLock(long timeout, TimeUnit unit)`，可等待指定时间后放弃锁申请。

### 二、示例对比代码

```java
ReentrantLock lock = new ReentrantLock();

// 使用 lock()
lock.lock();
try {
    // 临界区逻辑
} finally {
    lock.unlock();
}

// 使用 tryLock()
if (lock.tryLock()) {
    try {
        // 临界区逻辑
    } finally {
        lock.unlock();
    }
} else {
    // 获取锁失败后可执行备用逻辑
}
```

典型示例如下：在 `tryLock()` 获取失败的情况下可以不执行阻塞等待，而是选择跳过或执行其他路径。反之，`lock()` 保证进入临界区但可能会一直阻塞等待。

### 三、适用场景与优势

- `lock()`**：** 保证锁定成功、执行关键业务，适用于必须完全控制并发访问的场景。
- `tryLock()`**：** 提供 **非阻塞尝试**，避免线程长时间等待或死锁风险，可用于超时控制、调度优先处理等场景。即使锁被占用，线程流程继续也不会被挂起。
