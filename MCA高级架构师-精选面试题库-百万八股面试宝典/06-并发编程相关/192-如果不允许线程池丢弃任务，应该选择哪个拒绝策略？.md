在 Java 的 `ThreadPoolExecutor` 中，默认的拒绝策略是 `AbortPolicy`，当线程池无法接收新任务时，会抛出 `RejectedExecutionException`。然而，在某些场景下，我们希望确保任务不被丢弃，而是能够被执行。此时，可以选择 `CallerRunsPolicy` 作为拒绝策略。

### 1. `CallerRunsPolicy` 的行为

`CallerRunsPolicy` 是 `RejectedExecutionHandler` 的一种实现，当线程池无法接收新任务时，它会由提交任务的线程（即调用 `execute` 方法的线程）来执行该任务。如果线程池已关闭，则任务被丢弃。这种策略提供了一种简单的反馈控制机制，能够减缓任务提交的速度，从而避免系统过载。

### 2. 使用 `CallerRunsPolicy` 的示例代码

以下是使用 `CallerRunsPolicy` 的线程池示例：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, // 核心线程数
    4, // 最大线程数
    60, // 线程存活时间
    TimeUnit.SECONDS, // 时间单位
    new LinkedBlockingQueue<>(2), // 任务队列
    Executors.defaultThreadFactory(), // 线程工厂
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
);
```

在这个示例中，当线程池和任务队列都满了时，新提交的任务会在调用线程中执行。

### 3. `CallerRunsPolicy` 的优缺点

**优点：**

- 确保任务最终被执行，避免任务丢失。
- 通过让调用线程执行任务，减缓了任务提交的速度，有助于控制系统负载。

**缺点：**

- 可能导致调用线程阻塞，影响调用方的性能。
- 在高并发场景下，可能导致调用线程的响应时间增加。

### 4. 适用场景

`CallerRunsPolicy` 适用于以下场景：

- 任务非常关键，不能丢失。
- 调用方可以接受在当前线程中执行任务的场景。
- 希望通过减缓任务提交速度来控制系统负载。
