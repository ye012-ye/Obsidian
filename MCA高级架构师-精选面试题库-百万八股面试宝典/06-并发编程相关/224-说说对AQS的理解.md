在 Java 并发框架中，**AQS（AbstractQueuedSynchronizer）** 是个构建自定义同步器和一系列核心同步组件（如 `ReentrantLock`、`Semaphore`、`CountDownLatch`、`ReadWriteLock` 等）的基础工具。

### 1. 核心组件与数据结构

AQS 主要维护两个核心成员：一个由 `volatile int state` 表示的同步状态，以及一个用于线程排队的 **FIFO 双向链表**（`Node` 队列），其中每个节点封装了一个线程和状态信息。链表结构包含 `head` 和 `tail` 指针，实现 CLH 样式的阻塞队列机制。

### 2. 独占 vs 共享模式

AQS 支持两种模式：

- **独占模式**（exclusive）：如 `ReentrantLock`，仅允许一个线程持有同步状态；
- **共享模式**（shared）：如 `Semaphore`、`ReadWriteLock` 的读锁，允许多个线程同时获取资源。

线程尝试获取同步状态，会调用模板方法 `acquire(...)` 或 `acquireShared(...)`，内部通过 `tryAcquire(...)` 或 `tryAcquireShared(...)` 控制获取逻辑，由具体子类实现。

### 3. 获取与等待机制

如果调用 `tryAcquire` 成功，线程直接获得锁（`state` 修改成功）；否则，AQS 会将线程包装成 `Node` 并入队尾，随后调用 `acquireQueued(...)` 阻塞当前线程。等待锁释放后，使用 `unpark()` 唤醒队首线程重新尝试获取锁。

### 4. 释放机制

调用 `release(...)` 或 `releaseShared(...)` 时，会执行 `tryRelease(...)` 或 `tryReleaseShared(...)` 来更新 `state` 并判断是否释放资源。若成功，唤醒同步队列中后继节点（如头节点后继）继续获取资源。

### 5. Condition 支持

AQS 内部定义了 `ConditionObject`，用于实现类似 `Object.wait/notify` 的条件等待机制。`ConditionObject` 拥有专属等待队列，等待线程需要通过 `signal()` 转移回同步队列等待再次获取锁。

### 总结：

- AQS 是 Java 并发的底层同步框架，具备通用性和可扩展性；
- 通过 `state`、`Node` 队列、CAS 操作等组合提供高效的同步控制机制；
- 可支持独占与共享两种资源获取模式；
- 子类通过实现模板方法轻松定制自定义锁或同步器；

整体机制基于模板范式，由 AQS 管理线程排队、阻塞、唤醒等流程，把同步策略抽象出来，构建了 JUC 中众多稳定高效的同步组件。
