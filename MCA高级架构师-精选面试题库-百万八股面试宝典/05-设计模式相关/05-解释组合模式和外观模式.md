#### 一、组合模式（Composite Pattern）

组合模式将对象组织成树形结构，使客户端能够**统一处理单个对象与组合对象**。它通过定义一个统一的 `Component` 接口，`Leaf` 实现基本对象功能，而 `Composite` 则维护一个子组件集合，并将操作递归分派给子组件，从而形成节点与子树的统一接口。

- **简化管理**：客户端只与 `Component` 接口交互，不需要区分叶节点还是复合节点，调用接口即能递归处理整个层次结构。
- **适用场景**：适合处理“整体-部分”层级关系，如文件系统（文件和文件夹）、UI 组件树、组织结构图等场景，可以在统一结构中自由组合与扩展。

#### 二、外观模式（Facade Pattern）

外观模式提供一个简化接口，隐藏底层各组件或子系统的复杂交互，为客户端提供一个高层、统一且清晰的入口。客户端无需了解多个子系统的内部调用顺序与接口细节，只需调用外观对象即可完成复杂流程。

- **简化使用**：通过一个 `Facade` 类封装多个子系统接口，使复杂操作一步调用完成。例如启动计算机只需调用 `Computer.start()`，底层依次调用 CPU、硬盘、内存等组件的方法。
- **适用场景**：适合系统内部子系统较多、接口复杂、流程繁琐的情况，如第三方库封装、模块初始化、系统入口封装等场景。

S

#### 三、对比与协调作用

|  |  |  |  |
| --- | --- | --- | --- |
| **模式** | **结构形式** | **客户端与子系统关系** | **简化方式** |
| 组合模式 | 树形结构，统一接口 | 客户端直接操作统一接口 | 递归处理层级结构 |
| 外观模式 | 单一入口封装多个子系统 | 客户端通过 Facade 与子系统通信 | 隐藏复杂流程与接口逻辑 |

- **组合模式**侧重对象内部结构的管理与统一操作，让客户端能忽略叶与组合的区别。
- **外观模式**则侧重屏蔽子系统复杂性，提供清晰、简化接口供外部系统调用。

B
