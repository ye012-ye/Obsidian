装饰器（Decorator）和代理（Proxy）模式均属于结构型设计模式，二者在接口与类结构上相似，但本质目的与应用场景截然不同。

M

#### 一、设计意图对比

- **装饰器模式** 的意图是**动态地为对象添加新职责或功能**，不改变原对象结构。它通过包装已有对象、实现相同接口，从外层注入新的行为逻辑。
- **代理模式** 的核心目的是**控制对真实对象的访问**，通过代表对象“代理”真实服务。可用于访问安全控制、延迟加载、远程调用等场景。代理提供替代接口但不改变对象核心行为。

#### 二、结构差异与依赖方式

- **装饰器** 必须持有一个已存在的被装饰对象实例，通过包装方式叠加功能，运行时可以多个装饰器链式组合，功能可动态扩展。
- **代理** 则可选择自行创建真实对象，也可以接收传入的对象。通常仅单层代理，不推荐链式组合。代理关注控制和管理调用流程，而不是功能增强。

S

#### 三、功能扩展 vs 访问控制

- **装饰器用于功能增强**：比如在图形界面组件上动态叠加滚动条、边框、缓存、日志等装饰功能，而不影响原始接口调用。
- **代理用于访问控制或资源管理**：常见有远程代理（远程方法透明调用）、虚拟代理（延迟实例化大对象）、保护代理（权限验证）、缓存代理等用途。

#### 四、使用情况

|  |  |  |
| --- | --- | --- |
| **模式** | **目的** | **常见场景示例** |
| 装饰器 | 动态添加新功能 | Java `BufferedReader`  添加缓冲、UI 组件装饰样式等 |
| 代理 | 控制访问、增强调用流程 | Spring AOP 事务代理、安全控制、远程方法调用、延迟加载等 |

B

#### 五、总结与区分口诀

- **Decorator** 是“白盒包装”，目标是功能增强，并允许多个装饰层叠加。
- **Proxy** 是“黑盒代理”，重在透明控制访问流程，通常只代理对象而不并列多个。运行时可自行创建被代理对象。
