策略模式通过以下方式实现算法的灵活替换：

#### 算法封装（Strategy 接口）

将可替换的算法抽象成统一接口，例如：

```java
public interface Strategy {
    Result execute(ContextData data);
}
```

每种算法作为独立实现类（ConcreteStrategy），封装在单独模块中。

#### 上下文注入与组合方式

Context 类不直接实现算法，只持有一个 Strategy 类型的引用，通过构造器或 setter 注入具体策略。Context 调用时将任务委派给当前策略实例。

```java
paymentContext.setStrategy(new PayPalPayment());
paymentContext.execute(amount);
```

这种组合方式（而非继承）实现模块解耦。

#### 运行时动态切换

客户端可根据用户选择、配置或业务规则，在运行时替换策略实例，而无需修改 Context 或重新编译。  
策略对象可通过工厂方法或依赖注入框架动态创建并注入使用。

#### 模式优势

- 遵循开闭原则：新增算法只需添加新的 ConcreteStrategy，实现接口即可，不改动 Context 代码。
- 消除条件判断逻辑：Context 内无需 if‑else/switch 分支，算法切换统一由策略对象处理。
- 算法与 Context 解耦：可以独立测试、复用策略类，提升系统灵活性。
