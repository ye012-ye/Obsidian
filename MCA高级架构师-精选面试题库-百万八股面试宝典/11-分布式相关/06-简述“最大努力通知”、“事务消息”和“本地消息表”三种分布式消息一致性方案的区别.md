在微服务或分布式系统中处理跨服务一致性问题时，常见三种方案如下：

M

#### 1. 最大努力通知（Best‑Effort Notification）

这种方式由发送方“尽最大努力”发消息给被通知方，但**不保证消息一定送达**。它通常包括多次重试机制和一个查询接口，由接收方主动确认处理结果。

该方案实现简单，适用于对一致性要求不高的场景，例如支付成功后的通知、发短信邮件等场景。M  
**优点**：开发成本低，业务侵入性小；  
**缺点**：存在消息丢失和重复的风险，无法保证强一致性，一致性依赖接收方主动补偿。

#### 2. 事务消息（Transactional Message）

发送时，先向 MQ 发“半消息”（half‑message），确认消息入库成功后执行本地业务，最后提交或回滚消息。MQ 会定期回查半消息状态，由生产者决定消息的最终投递或丢弃。

这种模式由消息系统保证可靠性，适用于对一致性要求较高但不追求强一致性的场景，如积分发放、订单状态同步。S  
**优点**：消息投递原子、可靠；无需接收方主动查询；  
**缺点**：依赖 MQ 支持事务消息，业务侵入增加，需实现回查接口及状态管理。

S

#### 3. 本地消息表（Local Message Table / Outbox Pattern）

在执行本地事务时，将消息持久化到本地消息表和业务数据一起提交。由后台定时任务扫描消息表，将消息发送到 MQ，发送成功后更新状态或删除记录。

该方式可应对 MQ 不支持事务消息的情况，适合需要轻度一致性的场景，如订单生成并通知库存系统。B  
**优点**：无需 MQ 支持事务消息，业务与消息共事务；  
**缺点**：需要额外的消息表设计、轮询任务和消息状态管理，业务代码复杂度上升。

### 对比

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **方案** | **消息保证** | **实现复杂度** | **业务侵入性** | **适用场景** |
| 最大努力通知 | 最弱，可能丢失 | 低 | 低 | 邮件/短信通知、回调通知等 |
| 事务消息 | 强，依赖 MQ | 中等 | 中等 | 支付后状态同步、积分发放 等 |
| 本地消息表 | 强，脱离 MQ 限制 | 中高 | 较高 | 数据一致性重要，但 MQ 不支持事务时 |

B
