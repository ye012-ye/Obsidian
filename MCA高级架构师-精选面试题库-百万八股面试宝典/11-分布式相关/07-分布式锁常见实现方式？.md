在分布式系统中，分布式锁有三种常见实现方式：**数据库锁**、**Redis锁** 和 **Zookeeper锁**。它们各有适用场景与特点。

M

#### 1. 数据库锁

通过在数据库中创建一张专门的锁表（含 `lock_key`, `lock_owner`, `expire_time` 等字段），使用 `INSERT / UPDATE / DELETE` 的原子操作来获取和释放锁。

这种方式的优势在于工具简单，容易维护，且能借助数据库事务保证互斥与可重入性。不过其劣势也很明显：在高并发场景下数据库 IO 成本高，性能瓶颈明显；主动阻塞（如轮询等待）和公平性机制也实现复杂。M

S

#### 2. Redis 分布式锁

利用 `SETNX + EXPIRE` 或更先进的 Redlock 算法实现；Java 世界中还常用 Redisson 库封装的一整套锁机制（支持可重入、公平锁、看门狗续期等功能）。

- **优点**：延迟低，性能高，且客户端 API 简洁；Redisson 提供了可重入和自动续期功能。
- **缺点**：在主从复制或哨兵切换场景下，可能出现锁误判或失效，造成安全隐患。Redlock 算法虽试图解决分布式一致性，但在节点异常、网络延迟下仍有挑战。

#### 3. Zookeeper 分布式锁

基于 ZooKeeper 的临时顺序节点机制实现，客户端在 `/lock` 下创建 `ephemeral-sequential` 节点，并监听前一个顺序节点的删除事件。通过节点最小序号判断谁持有锁，形成公平阻塞机制。

- **优点**：强一致性、故障自动恢复、锁自动失效、天然公平；适合对正确性要求高的场景。
- **缺点**：引入了 ZooKeeper 服务，系统复杂度和运维成本上升；锁粒度通常在节点级别，且公平性机制带来一定延迟。

### 对比

|  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- |
| **实现方式** | **一致性** | **可用性/性能** | **自动续期** | 设置复杂度 | 适用场景 |
| 数据库锁 | 强 | 较低 | 手动或轮询 | 低 | 资源有限、高一致性需求 |
| Redis锁 | 弱（单节点）/较强（Redlock） | 高 | 看门狗自动续期 | 中 | 高并发环境，性能优先 |
| Zookeeper锁 | 强 | 中 | 自动（节点消失释放锁） | 高 | 高可靠性、多节点协调 |

### 选择建议

- 追求 **极简与可靠性**，且性能要求不高时，可选择数据库锁。
- **高性能或轻量级锁机制**倾向 Redis，尤其在本地缓存+Redisson 的常见业务场景中。
- 若对**一致性、公平性、可靠性**要求高（如分布式协调、Leader 选举等），建议使用 Zookeeper。
