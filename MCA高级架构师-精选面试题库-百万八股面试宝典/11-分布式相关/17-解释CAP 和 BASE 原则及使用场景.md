# **CAP 原则**

CAP 定理指出在分布式系统中遇到网络分区时，系统只能保证其中两项：

- **C（一致性）**：所有副本在读操作时都能返回最新写入。
- **A（可用性）**：每个请求都能获得响应（成功或失败），不能被无限延迟。
- **P（分区容忍性）**：系统在任意节点或网络断连后仍能继续运作。

**CAP 中的取舍方式**

- **CP 优先一致性**：如金融系统，遇分区时牺牲可用性，保证数据准确。（如 MongoDB、HBase）
- **AP 优先可用性**：如社交平台或缓存服务，允许短暂不一致，提高响应速度。（如 Cassandra、DynamoDB）
- **CA 在分布式场景中不可行**，因为必须容忍网络分区。

# **BASE 原则**

BASE 是对 AP 模式下一致性策略的一种总结：

- **Basically Available（基本可用）**：系统对请求返回肯定响应，但可能是过时或降级内容。
- **Soft state（软状态）**：系统允许存在中间状态，副本之间可短暂不一致。
- **Eventually Consistent（最终一致）**：经过一段时间后，所有副本收敛至一致状态。

### 应用对比

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **模式** | **一致性** | **可用性** | **延迟** | **系统场景示例** |
| **CP** | 强一致性 | 可用性低 | 延时高 | 银行转账 |
| **AP + BASE** | 最终一致 | 高可用 | 延迟低 | 社交点赞、留言分发 |

- 对于银行交易等对正确性要求极高的业务，优先选择 **CP 模式**，可接受部分不可用或延迟。
- 对于社交平台中点赞、评论等场景，适合 **AP + BASE** 模式，通过异步复制提高响应速度，允许短时不一致但最终达到一致。

B

**总结**

- CAP 原则帮助我们在分布式系统中明确面对网络分区时的一致性与可用性权衡。
- BASE 原则则是 AP 模式下一种容忍不一致的设计思想，通过弱一致性保障系统高可用与高性能。
