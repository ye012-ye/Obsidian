在存在网络故障（分区）时，分布式系统不能同时满足三项关键属性：一致性（C）、可用性（A）和分区容错性（P）。解释如下：M

首先，P（网络分区）是不可避免的现实，即系统可能会面临节点间通信中断的情况。在这种情况下，系统必须在以下两条路径中做出选择：

- **牺牲一致性（C），保持可用性（A）**  
  此时各节点即使数据不一致，也会继续响应请求以保证系统可用。这种设计允许返回过期数据，但系统整体可用，如 Cassandra 中常见的 AP 模式。M 若等待同步会造成可用性下降。
- **牺牲可用性（A），保持一致性（C）**  
  系统在一致性策略下拒绝处理请求，直到恢复最新数据。虽然保障了读写一致性，但在分区期间会阻塞响应，如 CP 系统（如 MongoDB 默认配置）。B

正常情况下（无网络分区），系统可以同时满足 C 和 A，实现 CA。然而，一旦 P 出现，就必须在 C 与 A 之间进行权衡。

本质上，这体现了“二选一”的设计取舍原则：

- **CP 类型**：优先一致性与分区容忍 → 在分区期间牺牲可用性。
- **AP 类型**：优先可用性与分区容忍 → 在分区期间牺牲强一致性，通常采用最终一致。
- **CA 类型**：只在完全没有分区的前提下才可能实现一致性与可用性，但不是真正的分布式方案。

CAP 定理告诉我们：在 分区 容忍 是必须的前提下，与其试图兼顾三项，不如根据业务场景主动权衡：

- 金融级系统通常选择 **CP** → 即使牺牲部分可用性，也要保证一致性。
- 社交、日志、电商库存等场景更倾向 **AP** → 接受短暂不一致，提升响应可用性。

总结一句：当网络分区发生时，系统只能在“一致性”和“可用性”之间二选一，因此不能同时满足 CAP 三者。
