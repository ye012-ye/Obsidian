Seata 是一款开源、易用且高性能的分布式事务解决方案，主要用于在微服务架构中协调多个数据库或服务间的事务，确保全局一致性。其核心架构包括三部分角色：TC（Transaction Coordinator，事务协调者）、TM（Transaction Manager，事务管理器）和 RM（Resource Manager，资源管理器）。TC 单独部署，TM 和 RM 嵌入应用中，协同作用完成全局事务的管理。

其有三种事务模式：AT、XA和Saga，下面分别说明：

### AT 模式（Automatic Transaction）

在 AT 模式中，当业务服务操作数据库时，Seata 会自动截取 SQL 并生成 undo-log（快照），第一阶段本地事务立即提交，同时注册分支事务给 TC，并加全局行锁；进入第二阶段时，若所有分支准备成功，TC 通知提交并删除快照，否则通过快照回滚数据。  
这种模式兼顾性能与一致性，全局行锁时间短，适合作为默认事务模式，但适用于关系型数据库场景，且提供的是最终一致性而非强一致性。

### XA 模式（Two‑Phase Commit）

Seata 对 XA 规范进行了封装，采用经典的两阶段提交协议。在第一阶段，RM 执行 prepare 并锁定资源，但不提交；第二阶段由 TC 决定是否 commit 或 rollback，然后各 RM 执行。XA 模式提供强一致性（ACID），实现无侵入，但由于持久锁定资源，性能与扩展性较差，且依赖数据库对 XA 的支持。

S

### Saga 模式

Saga 是一种补偿式长事务模式，将全局事务拆分为多个本地事务，每个事务执行成功后才继续下一步。如某一步失败，会执行对应的补偿事务逆向撤销之前操作。Seata 使用基于状态机的 Orchestration 实现 Saga，以 JSON 描述流程并自动执行补偿或正向重试。适合流程长、容忍最终一致性、不需全局锁定的业务场景，但增加补偿逻辑开发复杂度，并不保证隔离性。

### 模式对比总结

|  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- |
| **模式** | **一致性** | **性能** | **资源锁定** | **事务类型** | **适用场景** |
| XA | 强 | 较差 | 长时持锁 | 2阶段提交 | 要求 ACID、数据库支持 XA |
| AT | 最终 | 较优 | 短时行锁 | 自动快照回滚 | 关系型 DB、注重性能、简化开发 |
| Saga | 最终 | 高 | 无锁 | 多本地事务 + 补偿机制 | 流程复杂、需跨服务、有补偿业务能力 |
