#### 1. 两阶段提交（2PC/XA）

- **原理**：协调者（TM）在提交前发送 `Prepare` 请求给所有参与者（RM），各自执行本地事务并返回是否可提交。若全部同意，则发 `Commit`，否则发 `Rollback`。
- **优点**：标准 ACID，跨库强一致。
- **缺点**：阻塞严重，协调者单点故障，性能损耗高，多存日志。
- **场景**：跨多个数据库强一致需求，如银行柜台后台。

#### 2. 三阶段提交（3PC）

- **原理**：在 2PC 基础上增加 `PreCommit` 阶段（可提交），让系统进入一致的预备状态，从而在协调者失败时能更安全地决定。
- **优点**：减少长期阻塞。
- **缺点**：网络延迟高，实际环境中协议不完备，假设时延可控，限制较大。
- **场景**：对一致性要求高、对协调者挂掉容忍但可控延迟环境。

#### 3. TCC（Try–Confirm–Cancel）

- **原理**：每个参与操作实现三阶段：`Try` 预留资源，成功后 `Confirm` 提交，失败时 `Cancel` 回滚。需幂等性和空回滚控制机制。
- **优点**：资源锁粒度小、性能好、跨服务强一致。
- **缺点**：开发侵入性强，业务耦合高，实现复杂。
- **场景**：库存/订单锁定类业务，对性能敏感但需精准控制。

#### 4. Saga 模式

- **原理**：事务拆为多个本地事务，通过正向顺序执行，若失败则按反序执行补偿操作。有两种协调方式：中心化 Orchestration 或去中心 Choreography。
- **优点**：无全局锁，非阻塞，高可用。
- **缺点**：补偿事务需业务实现；不支持隔离，需做并发控制。
- **场景**：长事务或异步流程，如订单处理、跨服务编排。

#### 5. 本地消息表 / MQ 事务消息

- **原理**：将业务操作与消息写入同一本地表，事务提交后定时发送消息。MQ 通常实现类事务（如 RocketMQ 半消息 + 回查机制）。
- **优点**：业务与消息一致性高，依赖少，性能好。
- **缺点**：实现复杂，需要出账表清理、状态管理等，需开发支持。
- **场景**：异步消息驱动应用，如电商通知/日志系统。

M

### 对比总结

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **方案** | **一致性** | **性能** | **实现复杂度** | **场景** |
| 2PC/XA | 强 | 差 | 高 | 严格一致 |
| 3PC | 强 | 更差 | 更高 | 特殊高一致 |
| TCC | 强 | 较高 | 较高 | 锁控制 |
| Saga（编排/协调） | 最终一致 | 高 | 中高 | 跨服务流程 |
| 本地消息表 / MQ事务 | 最终一致 | 高 | 中 | 异步场景 |

S

### 选型建议

- 对跨多库多服务且需强一致时，优先使用 **2PC/XA**。若业务分布多且需性能与可用性，则倾向于 **TCC**。
- 在异步或容忍延迟一致性情况下，推荐**Saga**或**本地消息表+MQ事务**。
- 每种方案都有边界，实际中可混合使用，根据实际事务类型做权衡，保障可用性与可靠性。
