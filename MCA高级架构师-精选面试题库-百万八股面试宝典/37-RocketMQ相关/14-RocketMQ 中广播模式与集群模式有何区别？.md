RocketMQ 支持两种消费模式：**集群消费模式** 与 **广播消费模式**。这两者在消息分发机制、消费进度管理、失败处理能力和适用场景等方面存在本质区别。M

### 一、消息分发机制

- **集群模式**：属于同一个 Consumer Group 的消费者实例之间实施负载均衡，每条消息只分配给组内的 **一个消费者**。
- **广播模式**：无论多少消费者属于同一组，**每个消费者**都会收到并消费同一条消息的副本，实现 **消息全量广播**。

### 二、消费进度管理

- **集群模式**：消费进度（offset）由 Broker 统一管理，存储在服务器端；消费者重启后可从上次提交的进度继续消费。
- **广播模式**：消费进度由客户端自行维护，Broker 不参与管理；消费者重启后通常从**最新消息**开始，历史消息容易丢失。

### 三、失败重试与功能支持

- **集群模式**：支持消费失败自动重试、顺序消费、消息偏移重置、查看堆积消息等功能。
- **广播模式**：不支持消息重试，也不支持顺序消费、偏移重置或查看堆积状态，可靠性较弱。S

### 四、适用场景对比

- **集群模式**：适合主要业务场景，如订单处理、库存扣减、任务分发等，要求每条消息只处理一次、高可靠且支持扩展。
- **广播模式**：适合通知广播、配置下发、日志同步等，要求每个节点独立接收同一消息，弱一致性可接受的场景。

### 特性对比

|  |  |  |
| --- | --- | --- |
| **特性** | **集群消费（Clustering）** | **广播消费（Broadcasting）** |
| 消息分发 | 消息组内共享，每个消息由一个实例消费 | 消息全量发送至组内每个实例 |
| 消费进度管理 | Broker 统一管理 | 客户端自行管理 |
| 消费失败重试 | 支持 | 不支持 |
| 顺序消息消费 | 支持 | 不支持 |
| 消息偏移重置 | 支持 | 不支持 |
| 适用场景 | 高可靠、高并发处理业务流程 | 通知、日志、配置下发等广播型需求 |
