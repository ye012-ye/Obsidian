在微服务架构中，**客户端负载均衡**和**服务端负载均衡**是两种常见的请求分发方式，各自在实现机制、优势和适用场景上存在显著差异。

M

### 1. 划分职责与调用流程

**服务端负载均衡**（Server-side）：  
客户端只需调用统一的负载均衡入口（如 Nginx、HAProxy、云厂商 LB），由这个中间层将请求转发给后端各实例。客户端无需感知实例地址，降低调用复杂度。

**客户端负载均衡**（Client-side）：  
客户端主动向服务注册中心（如 Eureka、Nacos）获取可用实例列表，然后根据本地策略（如轮询、随机、响应时间）选择实例发起请求，无需经过额外中间代理。

### 2. 架构复杂度与故障承担

服务端 LB 在中间集中管理，客户端逻辑简单，但需部署和维护专门的负载均衡器，且成为潜在单点故障。

客户端 LB 分布式执行决策，没有单一故障点，但客户端需集成服务发现和负载算法组件，增加部署复杂性。

### 3. 性能影响与网络路径

服务端 LB 增加了一次请求跳转，可能导致额外延迟。对应节点配置可做流量限速、安全防护等控制。

客户端 LB 客户端直接请求目标实例，减少中间跳转，延迟低；客户端还可动态调整策略，如避开高延迟或异常实例。

### 4. 扩展场景与治理能力

服务端 LB 通常适合对外流量入口、统一 API 网关场景，支持复杂路由、SSL 终止、防 DDOS 等治理功能。

客户端 LB 常与 Spring Cloud Ribbon 或 LoadBalancer 结合使用，适用于微服务间调用，通过动态实例更新、健康检测实现灵活路由和容错。

### 5. Spring Cloud生态中体现

- **服务端 LB 示例**：Spring Cloud Gateway、Netflix Zuul 等，通过统一入口处理请求，再分发给服务实例。
- **客户端 LB 示例**：Spring Cloud LoadBalancer（替代 Ribbon）集成 Eureka/Nacos，实现客户端直接调用实例并执行负载算法。

### ​总结对比

|  |  |  |
| --- | --- | --- |
| **特性** | **服务端负载均衡（Server-side）** | **客户端负载均衡（Client-side）** |
| 请求路径 | 客户端 → 负载均衡器 → 服务实例 | 客户端 → 服务实例 （中间无需代理） |
| 复杂度与运维 | 集中配置、统一管理，单点易控 | 部署分散，每个客户端需要实现服务发现和 LB 逻辑 |
| 延迟与跳数 | 增加一次跳转 | 直接调用实例，延迟较低 |
| 容错与扩展 | 容易集中监控、防护 | 可动态感知实例状态、智能选择目标 |
| 架构耦合 | 客户端无需感知实例细节 | 客户端耦合服务发现机制，需动态更新策略 |
