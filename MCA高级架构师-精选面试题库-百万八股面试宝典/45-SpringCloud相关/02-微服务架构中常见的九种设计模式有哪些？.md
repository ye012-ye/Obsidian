以下九种设计模式来自 Microsoft Azure Architecture Center，适用于提升微服务系统的可靠性、可维护性和渐进演进能力：M

### 1. **Ambassador 模式**

对应客户端服务部署一个代理（通常以 sidecar 形式运行），负责日志采集、监控、安全认证、路由等公共职责。它可以以语言无关方式增强客户端或遗留系统的能力，无需修改应用代码。适用于增强不可控或无法改造的客户端组件环境。

### 2. **Anti‑corruption Layer（防腐层）**

在新系统与旧有系统之间建立适配层（或 façade），将遗留系统的接口、数据模型转换为新版系统使用的形式，从而避免新系统被旧系统的设计限制。适用场景包括对接遗留系统和平滑迁移。

### 3. **Backends for Frontends（BFF）**

为不同类型的客户端（如 Web、Mobile、SPA 应用）分别设计专属后端服务，从而客户端获取的数据更贴合自身需求。可简化接口设计，提升性能与灵活性。S

### 4. **Bulkhead（舱壁隔离）**

为不同服务或工作负载分配独立的资源池（如线程池、连接、CPU），避免其中一个应用或服务占满资源，造成系统级雪崩。增强微服务体系的弹性和故障隔离能力。

### 5. **Gateway Aggregation**

API 网关层汇集多个微服务的调用并聚合响应，客户端只发起一次请求即可获取多个服务数据，降低通信冗余和请求延迟。适用于聚合多个服务接口以提高客户端效率。

### 6. **Gateway Offloading**

将共用功能（如 SSL/TLS、认证、限流、监控等）从业务服务卸载到 API 网关，减轻各微服务压力并实现统一治理。适合统一处理跨服务横切关注点。

### 7. **Gateway Routing**

通过统一入口路由请求到不同服务，隐藏多个后端接口细节，简化客户访问。适合统一管理版本、动态路由与灰度策略等场景。

### 8. **Sidecar 模式**

将辅助功能（例如日志代理、监控代理、配置管理等）部署为与服务一起运行的独立进程或容器，可封装非业务逻辑职责，提升服务模块化程度。Sidecar 也常用于 Ambassador 模式的实现。B

### 9. **Strangler Fig（绞杀者模式）**

通过逐步拦截旧系统特定功能，并将其替换为新微服务的方式来迁移单体系统。旧模块在新服务稳定后逐个剔除，最终完成拆分演进。适用于系统平滑升级与渐进重构。

### 对比

|  |  |  |
| --- | --- | --- |
| **模式** | **主要功能** | **适用场景** |
| Ambassador / Sidecar | 提供语言无关的客户端代理功能 | 增强遗留或多语言客户端功能 |
| Anti‑corruption | 隔离遗留系统模型与新系统 | 平滑对接与迁移 |
| BFF | 客户端专属后端接口开发 | 多客户端类型场景 |
| Bulkhead | 资源隔离、防止级联故障 | 高可用、高隔离需求系统 |
| Gateway 聚合/卸载/路由 | 接口聚合、功能卸载、统一路由 | API 网关集中治理 |
| Strangler Fig | 渐进替换旧功能模块 | 拆分单体系统、系统重构 |
