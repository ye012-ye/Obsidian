在 Seata（Simple Extensible Autonomous Transaction Architecture）中，目前支持四种分布式事务模式，分别是 AT、TCC、SAGA 和 XA。以下针对它们进行简要说明：

M

### **AT 模式（Auto‑Transaction，默认模式）**

AT 模式通过对业务方 JDBC SQL 的代理拦截与快照机制，实现事务的自动补偿。系统会自动记录执行前后的数据快照（before image 与 after image），并写入 undo log 表。在全局事务提交阶段，如果需要回滚，Seata 会根据快照内容自动恢复数据状态。

**特点与适用场景**  
适用于基于关系型数据库且业务侵入性低的常规微服务系统。AT 模式无需修改业务逻辑，接入成本低，性能适中。主要用于电商、CRM 等中小规模系统。

S

### **TCC 模式（Try–Confirm–Cancel）**

TCC 模式依赖业务代码显式实现三个阶段：Try（资源预留）、Confirm（最终提交）、Cancel（取消补偿）。Seata TM 调度各服务的 Try、Confirm 或 Cancel 方法以保证最终一致性。

**特点与适用场景**  
优点是控制粒度细、性能高、资源锁持有时间短；缺点在于业务侵入性强，需要开发者手动编写补偿逻辑。适用于核心业务场景、高并发但业务逻辑复杂、需要精确事务控制的系统。

### **SAGA 模式**

SAGA 模式基于事件驱动或状态机执行流程，每个本地事务提交成功后，如果后续某一步失败，则触发补偿事务按反向顺序逐步取消已执行的事务。Seata 提供基于状态机的 Saga 引擎支持编排执行流程与补偿策略。

B

### **XA 模式（基于两阶段提交协议）**

XA 模式利用数据库原生支持的 XA 标准协议，采用两阶段提交协议管理全局与分支事务。Seata 提供 DataSource XA 代理，通过 XAConnection/XAResource 与 TC 协调提交或回滚，确保强一致性。

**特点与适用场景**  
强一致性（CP 模式），事务隔离性强；但性能较 AT/TCC 差，易阻塞。适用于金融、核心业务系统，对一致性要求高但事务规模不大的场景。

### **对比**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **事务模式** | **一致性级别** | **侵入性** | **性能表现** | **典型场景** |
| AT | 最终一致性 | 低 | 较高 | 常规业务系统、快速接入 |
| TCC | 最终一致性 | 高 | 性能优 | 核心模块、高并发控制要求 |
| SAGA | 最终一致性 | 中 | 一致性视设计复杂度 | 多阶段业务、跨服务补偿 |
| XA | 强一致性 | 低 | 较低 | 金融类或交易一致性要求高场景 |
