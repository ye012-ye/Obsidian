OAuth 2.0 是一种业界标准的**授权框架**，可以让用户将其存储在某个服务上的受保护资源授权给第三方应用使用，而无需泄露用户名和密码。该协议广泛用于现代 Web、移动和 API 场景中，专注于简化开发者授权流程，同时保证安全性。

### 一、核心角色解析

在 OAuth 2.0 中主要包括四类角色：

- **资源所有者（Resource Owner）**：通常指最终用户，是拥有受保护资源的实体。
- **客户端（Client）**：希望访问资源的第三方应用。
- **授权服务器（Authorization Server）**：负责验证资源所有者身份并签发访问令牌。
- **资源服务器（Resource Server）**：存储受保护资源的服务，通过令牌验证访问权限。 M

### 二、基本授权流程（以授权码方式为例）

流程如下：

1. 用户（资源所有者）通过客户端发起授权请求，跳转至授权服务器。
2. 授权服务器验证用户身份并请求用户同意授权。
3. 用户同意后，授权服务器生成授权码并重定向回客户端。
4. 客户端携带授权码与客户端凭据向授权服务器交换访问令牌。
5. 客户端使用访问令牌访问资源服务器，获取受保护资源。 S

### 三、常见授权模式（Grant Types）

根据不同应用场景，OAuth 2.0 提供多种授权流程，主要包括：

- **授权码授权（Authorization Code）**：适用于服务端 Web 应用，安全性高，涉及用户交互与令牌交换。
- **客户端凭证（Client Credentials）**：适用于服务间调用，不涉及用户身份，由客户端凭证获取访问令牌。
- **资源所有者密码凭证（ROPC）**：客户端直接使用用户名密码获取令牌，适用于受信任应用，但安全性较低。
- **隐式流（Implicit）**：适用于单页应用，令牌通过重定向直接返回，较简洁但安全性较弱。
- **刷新令牌（Refresh Token）**：可通过刷新令牌机制续期访问令牌，避免频繁授权。

### 四、OAuth 2.0 的优势与安全注意

OAuth 2.0 的优势包括：

- 无需共享密码即可授权访问；
- 支持不同客户端类型（Web、移动、桌面、设备）；
- 用户可撤销权限，并可限定访问范围与时效性（scope）。

但需要关注安全实践，如：

- 始终使用 HTTPS 保护敏感数据传输；
- 推荐在原生或浏览器应用中使用 PKCE（Proof Key for Code Exchange）防止授权码劫持；
- 客户端密钥需妥善保管，避免泄露导致滥用。 B
