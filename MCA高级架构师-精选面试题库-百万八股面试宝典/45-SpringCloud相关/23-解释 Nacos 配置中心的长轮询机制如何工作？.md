Nacos 的长轮询机制是一种“客户端主动拉取 → 服务端延迟响应”的 hybrid 模式，兼顾 Pull 模型简洁性与 Push 模型及时性。

M

### 客户端请求行为

客户端定期向服务端发起 HTTP 请求（Pull 模式），用于监听特定 `(dataId, group, namespace)` 的配置变更。如果服务端判断配置未变更，则不会立即响应，而是挂起该请求约 **29.5 秒**，等待期间不关闭连接。

### 服务端处理逻辑

服务端接收请求后，使用 Servlet 3.0 的 `AsyncContext` 异步处理该连接，将 请求加入 `allSubs` 队列，并安排定时任务：30 秒后统一检查并回应挂起请求。

如果在队列挂起期间，相关配置发生了变化，服务端会通过队列中的连接立即将变更响应给客户端（相当于一次 Push），随后连接关闭；否则超时后返回空响应，客户端重新发起下一轮请求。

S

### 优势与潜在问题

此机制减少短轮询频率，降低服务器压力，并且在配置更新时几乎实现实时通知。然而：

- 依赖大量长连接，可能占用服务端线程和内存；
- 高并发下频繁短连接可能引入大量 `TIME_WAIT`，加剧 GC 和资源消耗。

B

### Nacos 2.x 中的演进

Nacos 2.x 已逐步引入长连接（如 gRPC 或 RSocket）替代短轮询模型。客户端与服务端建立持久连接，配置变更时由服务端直接推送通知，客户端收到变更通知后按需拉取更新；此方式提升通信效率和实时性，大幅减少连接开销。

通过上述机制，Nacos 在 Spring Cloud 集成下，客户端通过长轮询（Nacos Config 模块）结合 `@NacosValue` 等注解可以实现热更新，无需重启服务，确保配置管理展示出高效、实时且资源友好的特性。
