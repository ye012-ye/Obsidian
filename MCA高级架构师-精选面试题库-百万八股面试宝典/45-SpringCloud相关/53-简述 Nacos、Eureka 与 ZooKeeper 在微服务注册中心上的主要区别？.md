**Nacos** 是一个功能丰富的平台，集成了服务发现、配置管理和动态 DNS。它支持 AP 模式（优先可用性）或 CP 模式（一致性强），底层使用 Raft 协议来保证数据一致性（如开启 CP 模式），同时也允许 AP 模式操作。在服务发现方面，Nacos 2.x 采用基于 TCP 的长连接推送机制，通知服务变更更及时。Nacos 还支持 namespace 分隔多环境和灵活的健康检查与负载策略配置。M

**Eureka** 提供纯粹的注册与发现功能，遵循 CAP 的 AP 原则，强调高可用性而不是一致性。各 Eureka 节点之间通过互相注册的对等架构实现同步。只要有一个节点可用，微服务就能继续通信。Eureka 客户端周期性发心跳维护注册表，若过期，则实例被注销。此外，Eureka 有自我保护机制防止因网络抖动时批量下线实例。S

**ZooKeeper** 除了注册中心，还兼具分布式协调功能，如命名服务、配置同步等。它严格遵循 CP 架构，优先保证数据一致性，但在网络分区期间可能会牺牲可用性。客户端通过 TCP 长连接与服务端通信，适合用于需要强一致性场景。不适合对可用性有极端要求的服务发现场景。B

## 对比总结

|  |  |  |  |
| --- | --- | --- | --- |
| **项目** | **Nacos** | **Eureka** | **ZooKeeper** |
| 功能 | 注册、配置、DNS 等 | 仅注册与发现 | 注册 + 分布式协调 |
| 架构类型 | 支持 CP 或 AP 模式 | AP 架构 | CP 架构 |
| 一致性 | 可配置 Raft 保证一致性 | 不保证强一致性 | 强一致性 |
| 数据同步 | 2.x 使用 TCP 推送 | 客户端定时轮询拉取 | TCP 长连接推送 |
| 健康检查 | 支持多种协议与元数据配置 | Client 心跳机制 | 客户端长连接检测 |
| 适用场景 | 微服务 + 配置治理 | 高可用注册中心 | 配置管理及协调任务场景 |
