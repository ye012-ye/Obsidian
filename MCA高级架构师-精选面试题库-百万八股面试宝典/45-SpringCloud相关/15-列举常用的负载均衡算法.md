以下算法广泛应用于客户端或服务端的负载均衡器中（如 Gateway、Ribbon、Nginx 等）：M

### 1. 轮询（Round Robin）

每次将请求顺序分配给下一个服务器，遍历完成后重新从第一台开始。适合服务器性能一致、请求均匀的场景，配置简单但不考虑实际负载。

### 2. 加权轮询（Weighted Round Robin）

基于轮询算法，但为不同服务器设定权重，让高性能服务器承担更多请求。适合服务器性能不一的场景。

### 3. 随机算法（Random）

将请求随机分配给服务器，适用于性能相近的均衡场景，算法简单但可能波动较大。

### 4. 加权随机（Weighted Random）

在随机分配的基础上加入权重，让高性能节点获得更高概率分配。适用于混合性能环境，兼顾随机性与性能分配。

### 5. 最少连接（Least Connections）

将请求发送给当前连接数最少的服务器，适用于请求处理时长差异较大的场景（如长连接、大资源任务）。

### 6. 加权最少连接（Weighted Least Connections）

结合服务器权重和当前连接数选择节点，兼顾容量与实时负载，适合高并发、复杂应用场景。S

### 7. IP/URL 哈希（Hash）

根据客户端 IP 或请求 URL 计算哈希值，决定请求目标实例，实现会话粘性或缓存一致性。常见于需要访问黏性的设计。

### 8. 响应时间最小（Least Response Time）

将请求分配给最近响应最快的服务器，动态感知节点性能，适合延迟敏感型系统。

### 9. 一致性哈希 / Rendezvous 哈希

适合缓存或分布式存储场景，可在节点增删时最大限度减少数据重新映射。Rendezvous Hash 实现更灵活高效，也用作请求分发规则。

### 10. 自适应或资源感知算法（Adaptive / Resource‑based）

通过监控服务器 CPU、内存、负载等指标，动态调整权重与选择策略，适用于资源负载高度动态变化场景。B

### ​应用场景对照（以 Ribbon 为例）

Ribbon（Spring Cloud 客户端负载均衡器）默认支持多种规则，包括：

|  |  |
| --- | --- |
| **策略** | **功能描述** |
| `RoundRobinRule` | 轮询分配，简单稳定 |
| `RandomRule` | 随机分配 |
| `WeightedResponseTimeRule` | 基于响应时间加权 |
| `AvailabilityFilteringRule` | 剔除不可用或高延迟实例后再轮询或随机 |
| `ZoneAvoidanceRule` | 避开高延迟或同 zone 性能差实例作分发策略 |

可以通过配置或自定义 Rule 实现特定需求，比如 `ConsistencyHashRule`、`CustomRule` 等。
