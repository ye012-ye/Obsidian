### 一、常见的限流算法

1. **固定窗口计数（Fixed Window Counter）**  
   将时间划分为固定区间（如每分钟），统计窗口内请求数，超过上限则拒绝。该方式简单易实现，但可能在窗口边界产生突发请求峰值。
2. **滑动窗口（Sliding Window Counter / Log）**

- **Log 版本**：保留每次请求时间戳，滑动时间窗口内统计记录数量，精确控制请求率。
- **Counter 版本**：保持多个子窗口的计数，通过加权累加实现滑动效果，相比 Log 更高效。

3. **令牌桶（Token Bucket）**  
   按固定速率向桶内生成令牌，请求需消耗令牌才能通过，桶空则拒绝，支持突发流量处理能力。
4. **漏桶（Leaky Bucket）**  
   请求进入队列，以固定速率处理，队列满则丢弃。适合平抑突发流量，输出节奏稳定。

各算法在吞吐量控制、突发容忍、实施复杂度和延迟方面各有权衡，实际选型需要根据系统负载和业务流量特性决定。

### 二、网关实现限流的方式（以 Spring Cloud Gateway 为例）

1. **嵌入式插件/过滤器限流**  
   Spring Cloud Gateway 内置 `RequestRateLimiter` 过滤器，使用 RedisRateLimiter 提供 Token Bucket 算法实现限流。开发者在配置中设置 `replenishRate`（令牌生成速率）、`burstCapacity`（桶容量）和 `requestedTokens`，即可定义请求速率与突发允许量。
2. **KeyResolver 定义限流维度**  
   通过实现自定义 `KeyResolver`，可以基于客户端 IP、用户 ID、API 密钥或路由标识等维度做资源隔离限流。例如根据 JWT 中的用户 ID 或客户端 IP 设置不同限流策略。
3. **插件化可扩展策略设计**  
   在高级设计中，可将限流算法模块化，通过策略接口支持多算法（Token Bucket、Leaky Bucket、Sliding Window 等）的动态切换，并将限流规则托管于配置中心（如 Config Server 或数据库），实现热更新。
4. **分布式存储支持**  
   Redis 常被用作分布式存储后端，支持原子递增、事务操作和 TTL 特性，不仅保证多实例间限流协调一致，也支持不同时间窗口的计数策略。
