**隔离机制差异：** Hystrix 通过线程池（bulkhead pattern）或信号量隔离来保护调用。当使用线程池隔离时，每个资源可能配置独立线程池，提升隔离效果但也带来线程上下文切换开销。Sentinel 则采用轻量级的信号量隔离：通过控制每个资源的并发线程数来限制访问，不创建独立线程池，减少性能开销同时防止资源滥用 。

M

**流量控制与熔断策略：** Sentinel 提供丰富的限流与降级策略，包括基于 QPS、并发线程数、响应时间、异常比例等多维规则，支持流量预热、排队、链路级保护、热点参数限流等高级控制方式。而 Hystrix 主要以异常比例为触发机制的熔断器，并不具备灵活的流量控制能力 。

**动态规则配置和监控：** Sentinel 支持通过控制台或动态规则来源（如 Nacos、Apollo、ZooKeeper、Redis 等）实时修改规则配置，同时提供链路透视的调用监控和仪表盘实时展示。Hystrix 的配置通常与 `@HystrixCommand` 注解或 Java 属性绑定，动态性较弱，尽管也提供监控仪表盘，但分析维度较 Sentinel 单一 。

S

**API 与生态整合：** Hystrix 使用 `HystrixCommand` 或 `@HystrixCommand` 注解封装业务逻辑及回退处理，绑定了 RxJava 执行模型。Sentinel 的 `@SentinelResource` 注解与编程 API（SphU、规则加载）资源定义与规则配置分离，使用更灵活。Sentinel 提供与 Spring Cloud、Dubbo、gRPC 等生态集成支持，支持多语言环境；而 Hystrix 已进入维护模式，不再活跃迭代 。

## 对比总结表

|  |  |  |
| --- | --- | --- |
| **特性维度** | **Hystrix** | **Sentinel** |
| 隔离方式 | 线程池隔离（可选信号量隔离） | 信号量隔离（线程计数模式） |
| 流量控制策略 | 重点关注异常率触发的熔断 | 支持 QPS、线程数、响应时间、异常数、多种链路与热点规则 |
| 动态配置能力 | 注解或属性固定配置，变化不灵活 | 支持控制台控制与外部数据源实时推送动态规则 |
| 监控与仪表盘 | 提供仪表盘视图，统计偏重延迟、错误报告 | 提供实时链路监控、局部调用路径、资源降级与预警控制面板 |
| 生态支持 | 基于 Spring Cloud Netflix，维护模式已停止 | Spring Cloud Alibaba、Dubbo、gRPC 等多生态支持，仍在活跃发展 |
