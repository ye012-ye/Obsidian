Seata 是一个开源的分布式事务解决方案，旨在为微服务架构提供易用、高性能的事务一致性保障平台。它支持 AT、TCC、Saga 和 XA 四种事务模式，以覆盖不同业务场景的需求。

**核心架构角色：**

- **TC（Transaction Coordinator）**：全局事务协调器，负责生成唯一的全局事务 ID（XID），管理所有分支事务的提交或回滚决策。
- **TM（Transaction Manager）**：事务管理器，嵌入业务系统，通过注解或 API 启动并结束全局事务，调度提交或回滚流程。
- **RM（Resource Manager）**：资源管理器，代理数据操作，负责注册分支事务、记录状态与执行本地事务提交或补偿操作。

M

**以 AT 模式为例说明工作流程：**

1. TM 向 TC 开启全局事务，并获取 XID；业务调用链携带此 XID 传播。
2. RM 拦截业务 SQL 语句，解析“before image”与“after image”，将数据与回滚日志（undo log）插入数据库，并执行本地事务提交第一阶段。
3. RM 向 TC 报告分支事务状态。TC 根据所有分支交互结果决定是提交还是回滚。
4. 若提交，RM 异步清理 undo log 并结束事务；若回滚，RM 根据 undo log 自动生成补偿 SQL，将数据恢复至事务开始前状态，完成反向操作。

S

**其他事务模式适用场景简述：**

- **TCC（Try‑Confirm‑Cancel）模式**：业务需要显式实现三段式操作（Try、Confirm、Cancel），适合对回滚逻辑要求高的核心业务场景。
- **Saga 模式**：适合业务步骤多、长事务流程，采用补偿式机制，每个子交易失败时执行补偿操作以保持最终一致性。
- **XA 模式**：遵循数据库 XA 协议的传统两阶段提交，虽保证强一致性，但锁资源周期长，性能较差。

B

总结：Seata 通过引入 TC、TM、RM 三方协作机制，将分布式事务拆解为全局事务与多个分支事务，并基于事务模式的不同处理路径（如 AT 的 undo log 快照补偿、TCC 的显式接口补偿、Saga 的业务补偿执行）来保证一致性。同时，它在 AT 模式下实现了几乎零业务侵入、异步二阶段操作，提高性能扩展性，适合现代微服务环境使用。
