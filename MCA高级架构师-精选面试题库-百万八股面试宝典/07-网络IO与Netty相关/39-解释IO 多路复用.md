I/O 多路复用是一种允许程序通过单个线程同时监控多个 I/O 通道（如网络连接、文件描述符）并获取哪个就绪的技术。与传统为每个连接启动线程相比，它显著节省了资源开销，提升系统扩展能力和效率，用一句话来说，就是“一个线程，多路监听”。M

常见实现机制包括以下几种：

**1.** `select()` **/** `poll()` **模式**

- `select()` 利用固定大小的 `fd_set`，每次调用时用户态将所有文件描述符集合拷贝到内核态，内核遍历判断哪些准备好，完成后再用户态遍历结果，复杂度为 O(n) 且受最大描述符数量限制，仅适合连接数较少的场景。
- `poll()` 则使用动态的 `pollfd[]` 数组结构，免去了最大描述符数限制，但仍需重复传递整个数组，复杂度仍为 O(n)，性能略优但同样不适用于大量并发连接。

**2. Linux 下的** `epoll`

- 分三步进行：通过 `epoll_create` 创建内核上下文，通过 `epoll_ctl` 注册或取消感兴趣的 fd，通过 `epoll_wait` 阻塞等待事件发生。这样内核维护一个红黑树结构，仅在事件发生时通知，避免每次 O(n) 遍历，整体复杂度接近 O(1)，尤其适用于大量并发连接场景。
- 支持边沿触发（edge-triggered）和水平触发（level-triggered）两种模式，在高并发场景中可以选择更高效的策略处理就绪事件。

**3. BSD/macOS 的** `kqueue`

- 与 `epoll` 类似，利用内核事件队列高效管理 I/O 就绪通知，在 FreeBSD、macOS 等系统上常被使用。

S

部署要点：

- 在连接数较少时，`select`/`poll` 已能满足需求；但要处理高并发、大量连接，推荐使用 `epoll` 或 `kqueue`。
- `epoll` 与 `kqueue` 都将感兴趣的 fd 注册到内核中，仅在其状态变化时才唤醒用户态，显著减少系统调用与数据拷贝次数，提高单线程的处理能力。
- 使用时要注意触发模式的选择和对已就绪事件的完整处理逻辑，否则可能出现遗漏或重复触发问题。

通过这种方式，一个线程就能高效覆盖大量 I/O 通道，适用于高性能网络服务器、实时通讯等场景。B
