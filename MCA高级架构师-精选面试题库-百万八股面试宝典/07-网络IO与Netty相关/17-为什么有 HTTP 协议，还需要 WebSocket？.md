HTTP 是一种典型的请求—响应协议，属于半双工通信模式：客户端必须发起请求，服务器才能响应，然后连接关闭，通信中断。每次互动都要建立新连接（即使启用了 HTTP keep‑alive），并携带冗长的头部信息，效率低下，延迟较高。因此，HTTP 适合传输一次性资源（如网页、图片、文档），但并不适用于高频、实时双向交互的场景。

WebSocket 在 HTTP 基础上引入升级机制。客户端向服务器发出 HTTP 请求，并携带 `Upgrade: websocket` 头部。服务器应答后，双方切换到 WebSocket 协议，建立一个持久的、全双工的 TCP 通道，并通过轻量级帧交换消息。这样之后，客户端与服务器可以在任何时刻互相推送数据，不必频繁建立连接或等待请求。

M

由于使用了持久连接和小消息帧，WebSocket 可显著减少网络开销与传输延迟，尤其适用于对实时性要求高的应用，如多人在线游戏、即时聊天、协作编辑、实时行情、IoT 数据推送等。在这些场景下，服务器需要主动向客户端推送，而 HTTP 即使使用轮询技术，也无法做到及时、流畅。

当然，并非所有场景都适合 WebSocket。如果场合是偶发访问、大文件下载或资源请求（如访问文章、图片、文档）等，HTTP 更简单、兼容性更高、缓存机制成熟。在这些情况下，用 HTTP 更合适。

S

### **总结对比**

|  |  |  |
| --- | --- | --- |
| **感知差异** | **HTTP** | **WebSocket** |
| 通信模式 | 半双工、始终由客户端发起请求 | 全双工、客户端和服务器可任意发消息 |
| 连接状态 | 短连接或短期保持连接 | 一旦建立，长久在线只关闭时断开 |
| 性能与开销 | 每次请求都带大头部，频繁连接开销大 | 小帧通信，无需重复握手，延迟低、效率高 |
| 适用场景 | 网页资源加载、REST API | 实时互动场景如聊天、游戏、协同编辑 |

B

**结论：** 若应用侧重于单向、瞬时请求—响应或资源下载，HTTP 足够且成熟；而面对需实时、双向通信，高频数据交换的场景，WebSocket 是更合适的选择。
