现代系统中，为了实现评论、点赞、聊天、通知等实时消息推送，常见的方案主要有以下几种：

M

#### **1. 短轮询（Short Polling）**

客户端定时发 HTTP 请求询问是否有新消息。如每秒发送一次请求，获取最新状态或内容。优点是实现简单，兼容性好。缺点是频繁请求造成资源浪费、带宽压力增大，且延迟取决于轮询间隔。

#### **2. 长轮询（Long Polling）**

客户端发起请求后，服务器保持请求悬而未决，直到有新数据或超时后才返回，并在客户端收到响应后立刻重新发起请求。相比短轮询网络效率更高、延时更低，但仍会产生持续请求开销，资源占用相对较高。

S

#### **3. Server-Sent Events（SSE）**

SSE 建立一个持久的 HTTP 连接，服务器以 `text/event-stream` 格式单向推送事件。优势在于使用标准 HTTP，无需特殊协议或复杂实现；支持自动重连和事件 ID。适用于单向消息推送，如通知、统计更新等。缺点是不支持客户端向服务器主动发送消息、无法传递二进制数据，IE 浏览器兼容性较差。

#### **4. WebSocket**

WebSocket 在建立初期通过 HTTP 握手后升级为 TCP 全双工连接，可实现低延迟、双向通信，适用于实时聊天、协同编辑、在线游戏等场景。性能优越，但实现较复杂，需要考虑连接管理、状态同步、负载均衡等问题。

#### **5. MQTT**

一种轻量级的 publish/subscribe 协议，基于中心 broker（服务器）转发消息，特别适合资源受限或不稳定的网络环境（如 IoT、移动端）。支持 QoS 保证、多订阅、二进制消息传递。常用于设备群推，亦可以通过 WebSocket 作为传输层扩展浏览器支持。

B

### **技术选型对比**

|  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- |
| **方案** | **连接方式** | **通信方向** | **延迟** | **实现复杂度** | **适用场景** |
| 短轮询 | HTTP请求 | 双向请求型 | 中高 | 低 | 简单状态检查、兼容性高 |
| 长轮询 | 持久HTTP | 双向请求型 | 中 | 低中 | 临时实时、兼容老系统 |
| SSE | 持久HTTP | 服务端→客户端 | 低 | 低 | 通知、监控、日志推送 |
| WebSocket | TCP持久连接 | 双向 | 极低 | 中高 | 聊天、游戏、协作 |
| MQTT | TCP持久/WS | 发布/订阅 | 可低 | 中 | IoT、多客户端订阅 |

### **如何选择？**

- **若需双向低延迟通信，且对性能有较高要求，推荐** **WebSocket****；**
- **若只需服务端单向推送，开发简单且兼容浏览器，推荐** **SSE****；**
- **若设备端资源有限、有订阅分发需求，考虑** **MQTT****；**
- **对于兼容性需求强、实现简单的方案，可考虑** **长轮询****；**
- **短轮询仅适合不追求实时、实现特别简单的场景。**
