HTTP 接口（尤其是基于 REST 架构）主要采用文本协议，例如 JSON 或 XML，在 HTTP/1.1 或 HTTP/2上通信。它以资源为中心，通过一致的 URL 和 HTTP 方法（如 GET、POST、PUT、DELETE）来操作资源。这种方式标准化程度高、兼容性强，适合开发浏览器 API、外部开放接口或通用服务，具备良好的可读性和生态支持。REST 接口是无状态的，有助于水平扩展和负载均衡。

M

RPC 接口则偏重函数调用风格，客户端像调用本地方法一样远程调用服务端函数。RPC 通信常通过二进制协议（如 Protocol Buffers、Thrift）传输，依赖接口定义语言（IDL）进行类型定义和自动代码生成。这种方式通信更紧凑、高效，延迟更低，但调试可读性较差，浏览器原生不支持，需要额外支持。RPC 适用于微服务内部高频调用、低延迟、高吞吐的场景。

S

性能方面，RPC（例如 gRPC）通过 HTTP/2 实现多路复用、头压缩与二进制序列化，比传统 HTTP+JSON 接口能提供更高吞吐量和更低延迟，常用于内部服务通信、实时交互或对效率要求严格的系统。

在耦合方式和扩展性上，HTTP/REST 是无状态的资源导向，为系统演进提供了松耦合的优势，而 RPC 更偏程序调用，易造成较强依赖，但接口生成后一致性更好、强类型支持明确，有助于大型系统中快速构建稳定接口。

B

如果你的系统对外提供接口给浏览器或跨平台客户端使用，或对资源具过程化管理且关注可读性与兼容性，应使用 HTTP/REST；如果目标是构建高性能的内部服务调用、微服务之间频繁交互且要求低延迟与强类型契约，那么基于二进制协议的 RPC（如 gRPC）通常是更优选择。两者也可混合使用：对外暴露 REST，对内服务互调采用 RPC，以兼顾灵活性与性能。
