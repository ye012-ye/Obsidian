在 Java 网络编程中，分别采用了三种不同 I/O 模型，它们在阻塞类型、线程使用、性能与复杂度上各有优劣：

#### 一、BIO（Blocking I/O）

这是最传统的一种模式，每个客户端连接对应一个线程，完全阻塞模式。当线程执行 `accept()` 或 `read()` 时，会一直等待直到操作完成。这种模型逻辑简单，但在连接数增长时，会因为大量线程阻塞，导致资源浪费与性能下降。 S

**适用场景**：连接数量少、持续时间长、逻辑简单，如小规模 HTTP 服务或命令行客户端。

#### 二、NIO（Non-blocking I/O，多路复用）

NIO 引入了 **非阻塞通道(Channel)、缓冲区(Buffer)、选择器(Selector)**。线程无需为每个连接开独立线程，而是注册通道事件到 Selector，然后通过 `select()` 等接口检测哪些通道准备好 I/O，再执行实际的读写。虽然 `read()` 仍是阻塞，但是在确定就绪后进行，从而不浪费资源。 M

**优点**：用少数线程处理大量连接，提升高并发性能，适合聊天室、消息服务器等短连接场景。  
**缺点**：编程复杂度高，需要手动管理状态与事件注册。

#### 三、AIO（Asynchronous I/O，异步非阻塞）

Java 7 引入更先进的异步通道（`AsynchronousSocketChannel`）。调用 I/O 方法后立即返回，操作交给操作系统异步执行，完成后通过回调或 Future 通知程序处理结果。这是纯异步模式，线程不阻塞，也无需频繁轮询，效率最高。 B

**适用场景**：连接数多且处理复杂或耗时，如大文件传输、图像处理等高并发、高吞吐应用。但编码更繁琐，需设计回调或链式处理逻辑。

### 总结对比表

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **模型** | **阻塞方式** | **线程模型** | **关键机制** | **应用场景** |
| BIO | 阻塞 | 每连接一线程 | - | 小规模群连接 |
| NIO | 同步非阻塞 | 少量线程 + Selector | 多路复用 | 高并发短连接 |
| AIO | 异步非阻塞 | 事件回调机制 | 操作系统异步通知 | 高吞吐、长连接 |
