在 I/O 操作中，同步和异步是两种根本不同的调度方式，它们的差异在于发起请求的线程是否等待操作完成。

# 一、概念

**同步 I/O：阻塞等待完成**  
同步 I/O 模式下，当前线程发起 I/O 操作（如读写文件或网络）后，将被阻塞，直到该操作结束并返回结果。执行流程是线性的：调用—等待—结果被处理。这种方式的优点是编程直观、逻辑清晰，易于调试。但在执行耗时 I/O（如大文件读写或慢网络）时，线程一直被阻塞，导致资源浪费和响应变慢。

**异步 I/O：发起即返回，结果延后处理**  
在异步 I/O 模式中，线程发出 I/O 请求后立即返回，然后继续执行其他任务。操作完成时，系统会通过回调函数、事件通知、完成端口等机制告知应用程序。这样即使 I/O 花费较长时间，主线程依然可以并行处理其他任务。异步 I/O 常用于高并发场景，能够显著提高吞吐和资源利用率。

# 二、对比

|  |  |  |
| --- | --- | --- |
| **特性** | 同步 I/O | 异步 I/O |
| **阻塞行为** | 请求阻塞，线程无法继续执行 | 请求非阻塞，线程可继续执行其他任务 |
| **代码风格** | 线性、直观，易于理解与调试 | 依赖回调/事件机制，逻辑复杂，可读性较低 |
| **资源利用** | 阻塞期间 CPU/线程资源闲置，效率低 | 支持并发，提升系统吞吐和资源利用率 |
| **适用场景** | CPU 密集型或快速 I/O 场景 | 高并发 I/O、网络服务器、数据库、大文件处理场景 |

# **三、优缺点**

- **同步 I/O**

- **优点**：实现逻辑清晰，开发调试成本低；适用于小批量、快速完成的 I/O。
- **缺点**：在长时间 I/O 操作时会引起线程阻塞，影响应用响应性能。

- **异步 I/O**

- **优点**：非阻塞、高并发、可并行处理多个I/O，提高系统吞吐；底层可借助 DMA 或中断机制进一步提升效率。
- **缺点**：实现更复杂，需管理状态、回调或事件，调试难度提升；如果 I/O 操作非常快，异步机制带来的开销可能反而适得其反。

# **四、如何选择？**

若应用场景中 I/O 操作简单且快速、无需高并发，使用同步 I/O 更直接；但对于数据库、网络服务器或大数据处理等涉及长时间或高并发 I/O 的环境，异步 I/O 能更有效提高性能和资源利用。开发者需要权衡二者实现难度和性能收益，选用最合适的 I/O 模型。
