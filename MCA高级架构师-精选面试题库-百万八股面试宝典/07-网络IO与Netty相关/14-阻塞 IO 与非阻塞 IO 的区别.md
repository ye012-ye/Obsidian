在 I/O 操作中，阻塞 I/O（Blocking I/O）与非阻塞 I/O（Non-blocking I/O）最大的区别在于调用线程是否会被挂起等待操作完成。

M

#### **阻塞 I/O**

阻塞 I/O 会使线程等待 I/O 操作完成后再继续执行。如果应用执行 `read()`、`write()` 等调用，线程会被阻塞直到数据准备好或操作完成。这种方式编程直观、易读，适合对并发性能要求不高的简单场景，如批量文件处理或命令行工具。但当 I/O 操作耗时较长时，线程将被挂起，导致响应变慢，资源闲置率高。

#### **非阻塞 I/O**

非阻塞 I/O 则会立即返回，不会阻塞线程。通常返回一个指示当前 I/O 是否可用的状态，应用需要使用轮询机制（如 `select()`、`poll()` 或事件驱动）来检测 I/O 是否就绪。这种模式让线程可以同时处理多个 I/O 操作，提升资源利用率和并发能力。但它的编程复杂度更高，需要管理轮询、状态检查与错误重试等。

#### **异步 vs 非阻塞**

值得注意的是，非阻塞 I/O 并不等同异步 I/O。非阻塞只代表调用返回快，实际完成需要额外机制；异步 I/O则是由系统或框架完成 I/O 后主动通知应用，比如通过回调或事件机制。两者可结合使用，但概念上不同。

### **优缺对比**

|  |  |  |
| --- | --- | --- |
| **特性** | **阻塞 I/O** | **非阻塞 I/O** |
| **线程是否被挂起** | 是，直到 I/O 完成 | 否，立即返回，需后续状态检查 |
| **编程复杂度** | 低，代码清晰易写 | 高，需要轮询、事件驱动管理 |
| **资源利用** | 效率低，线程在等待期间资源闲置 | 效率高，可管理多个通道，适合高并发 |
| **响应能力** | 延迟高，对实时场景不友好 | 延迟低，可快速响应多个 I/O 事件 |
| **适用场景** | 简单 I/O、低并发、一次性任务 | 网络服务器、GUI 事件处理、实时应用等 |

B

### **使用建议**

- 如果应用场景中 I/O 操作简单、任务数少，偏好清晰的编程模式，阻塞 I/O 是合理选择。
- 对于需要同时处理多个 I/O 事件、提高资源利用率与响应速度的场景，如网络服务端、事件驱动应用，非阻塞 I/O 更具优势，尽管编程复杂度更高。
