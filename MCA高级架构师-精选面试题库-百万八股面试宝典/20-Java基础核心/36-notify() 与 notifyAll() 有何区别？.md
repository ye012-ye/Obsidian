在 Java 中，`notify()` 和 `notifyAll()` 都是 `Object` 类的方法，用于唤醒因调用 `wait()` 而挂起的线程，但应用场景和行为存在明显差异：M

### ​1. 唤醒范围不同

- `notify()`：只随机唤醒一个正在等待该对象监视器的线程。其他等待线程仍处于阻塞状态。线程选择取决于线程调度器，具有不定性。
- `notifyAll()`：唤醒所有等待该锁的线程，唤醒后它们会竞争该对象的锁，最终只有一个线程获得锁继续执行，其他线程可能继续等待或退出等待状态。

### ​2. 性能与资源开销

- `notify()` 优点在于只唤醒一个线程，开销小；缺点是可能唤醒不合适的线程，导致资源空转。
- `notifyAll()` 虽然会触发更多线程竞争锁（开销较大），但能保证不会遗漏任何应当被唤醒的线程，避免死锁或逻辑阻塞。

### ​3. 适用场景

- **使用** `notify()`：当多个等待线程中只有一个能继续执行，例如生产者-消费者场景中生产方 `notify()` 消费者 。
- **使用** `notifyAll()`：当多个线程能够并行响应唤醒且彼此独立，或等待条件复杂（有多种等待条件），需确保所有满足条件的线程都能被唤起，例如任务完成后通知多个线程。S

### ​4. 使用注意

- 两者必须在同步块或同步方法内部调用，否则会抛 `IllegalMonitorStateException`。
- 唤醒线程后，仍需再次检查等待条件（通常使用 `while` 循环），确保逻辑正确──因为即使被唤醒，条件可能不再满足，必须重检。

### 总结

|  |  |  |  |
| --- | --- | --- | --- |
| **方法** | **唤醒线程** | **性能开销** | **适用场景** |
| `notify()` | 唤醒一个 | 低 | 只需唤醒一个线程继续任务时 |
| `notifyAll()` | 唤醒所有 | 高 | 多线程需并行响应，或等待条件较复杂时 |

若只唤醒一个线程符合逻辑且开销可控，则可使用 `notify()`；如果担心遗漏线程或多线程需要并行动作，则应使用 `notifyAll()`，以确保线程间通信可靠、无遗漏。B
