基于角色的访问控制（RBAC）是现代权限管理的常见模式，其核心思路是：将权限赋予角色，由用户担任角色来获得权限，从而简化权限管理。下面简要分析：M

1. **核心组件**

- **用户（User）**：系统中的主体，可以登录操作。S
- **角色（Role）**：代表一组权限的集合，例如“系统管理员”、“普通用户”。
- **权限（Permission）**：对系统资源进行操作的授权，比如“读取订单”、“删除用户”。
- **会话（Session）**：用户可以激活其被授权的角色，从而在该会话中行使角色权限。

2. **基本模型（RBAC0）**用户——> 多个角色 ——>多个权限，用户登录后通过会话激活角色，使得权限控制清晰可管理且灵活。
3. **扩展形式**

- **RBAC1（角色继承）**：支持角色之间的继承关系。上级角色自动拥有下级角色权限。适用于层级明确的管理结构。
- **RBAC2（职责分离约束）**：引入静态/动态约束，如互斥角色、角色切换控制，防止权限滥用或权限冲突。
- **RBAC3（集成模型）**：结合角色继承与职责约束，是最完整的 RBAC 实现形式。B

4. **实际应用案例**

- **企业权限结构**：总部主管角色继承区域主管权限，同时受制于财务与审批职责的分离。
- **大型系统用户管理**：数千用户中批量授权，先创建角色并关联权限，再批量绑定用户或用户组，有效减少重复设置。
- **场景切换**：用户可临时激活不同角色，如同时担任“测试”和“开发”角色时，通过会话激活其中一种，在特定任务中行使对应权限。

5. **使用优势**

- 管理方便：新增用户只需关联角色，无需逐条配置权限。
- 灵活扩展：配合继承与约束，可以应对复杂权限需求和组织结构调整。
- 提升安全性：职责分离避免敏感操作集中于单一角色，降低风险。

整个模型清楚地划分了 **用户、角色、权限** 的关系，同时通过扩展形式应对层级与安全性的实际需求，使得大中型系统权限架构既灵活又易维护。
