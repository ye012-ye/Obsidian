Redis 使用**分离链表（Separate Chaining）**策略来解决哈希冲突。当多个键经过哈希函数映射到同一个槽时，它们不会被丢弃，而是以链表（或类似结构）的形式链接在该槽中。

### ​实现细节分析

1. **dictEntry 链表节点**  
   Redis 的每个哈希桶中存储的是 `dictEntry` 结构，每个节点包含 key、value 和指向下一个节点的 `next` 指针。
2. **动态扩容与渐进式 rehash**

- 当键数量与桶数量的比（负载因子）超过 1 时，Redis 会扩大哈希表大小（通常按 2 的幂增长）。
- rehash 不是一次完成的，而是在之后的访问过程中逐步完成，这样可以避免停顿（阻塞时间短）。

3. **并行查询 rehash 中的表**  
   在 rehash 进行期间，对应的查找操作会依次在旧表 `ht[0]` 和新表 `ht[1]` 中查找目标 key，以保证正确性。
4. **链表性能与负载控制**  
   如果链表过长（即冲突过多），平均查找复杂度可能退化；Redis 通过自动扩容来控制负载因子，使链表长度维持在一个较低水平。

### ​总结说明

Redis 使用**链式法**处理哈希冲突，通过动态扩容和渐进 rehash 的方式平衡性能和效率。在 rehash 期间，读写操作能正确访问新旧结构，并最终迁移所有数据，既避免了阻塞也保证兼容性和稳定性。
