在大规模网站中统计 UV（Unique Visitors）或访问量时，Redis 提供了多种方案，适用于不同的业务要求。以下从准确性、内存开销与性能角度说明三种常用方式。M

### 1. Hash 或 Set 精确统计

可以使用 `HSET uri:date userId 1`（或 `SADD uri:date userId`）存储每个用户的访问标识，最后使用 `HLEN` 或 `SCARD` 获取当天访问人数。该方式精确性高，便于查询。但当用户数达到数亿级别时，会产生大量 key 和 field，占用内存巨大，导致查询和存储性能下降，成本不现实。适合用户规模中等、精确性要求高的场景。

### 2. Bitmap 位图统计

通过将用户 ID 映射到位图位置（例如哈希到某个整型 ID），使用 `SETBIT key offset 1` 标记已访问，最后通过 `BITCOUNT key` 得出访问人数。此方案使用空间非常节省，例如 1 亿用户约占 12 MB 内存（每位 1 bit），查询同样高效且支持快速判断单个用户是否访问过。缺点是对用户 ID 映射依赖较强，无法支持稀疏用户或字符串原 ID，且存在碰撞概率（不同用户哈希到同一位可能）。若用户稀疏或超过映射范围，会增加额外复杂度。

### 3. HyperLogLog 基数估算

当系统访问量非常大、且可容忍少量误差时，推荐使用 Redis 的 HyperLogLog。它是一种基数估计算法，使用固定约 12 KB 内存，无论元素数量多大，提供标准误差约 0.81%。通过 `PFADD key userId` 添加用户，最后用 `PFCOUNT key` 得出估算值，适合大规模 UV 或 PV 统计。优点是极低内存开销和良好性能；缺点是只能提供估算结果，且无法查询某单一用户是否访问过，不适用于需要精确判断的业务。

### 对比

|  |  |  |  |
| --- | --- | --- | --- |
| **方案** | **精度** | **内存消耗** | **适用场景** |
| Hash / Set | 完全准确 | 高 | 中小规模、精确统计 |
| Bitmap | 准确且检查单点 | 较低 | 用户 ID 映射明确、需要按用户查询 |
| HyperLogLog | 估算（误差约 1%） | 极低 | 海量用户 UV 统计，宽容误差 |

若业务需要既要低延迟又准确且能判断单一用户访问状态，可采用 Bitmap；如果只是统计 UV 总量，且规模极大，HyperLogLog 是最优选择；若对数据精度要求极高且规模适中，Hash/Set 模型仍然可靠。
