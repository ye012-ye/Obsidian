当 Redis 达到 `maxmemory` 限额时，会根据配置的 `maxmemory-policy` 触发淘汰机制，确保继续接收写入请求。核心策略如下：

M

### 1. **noeviction**

- **机制**：不删除任何数据，新写入将被拒绝，返回错误。
- **适用场景**：适合对数据完整性要求极高的系统。
- **风险**：应用可能因写入失败导致异常。

### 2. **volatile-**\* 系列（仅淘汰设有过期时间的 Key）

- **volatile-LRU**：优先淘汰最近最少使用的过期 Key，适合缓存场景。
- **volatile-TTL**：倾向删除 TTL 剩余时间最短的 Key，以提升空间释放速度。
- **volatile-random**：随机淘汰过期 Key，实现简单但无访问倾向性。
- **可选场景**：适用于大部分 Key 可设置 TTL 的应用。

### 3. **allkeys-**\* 系列（全键淘汰，无过期限制）

- **allkeys-LRU**：最常用的缓存淘汰策略，按最近最少使用原则淘汰所有 Key。
- **allkeys-LFU**：淘汰访问频次最低的 Key，适合热点频繁变更的场景。
- **allkeys-random**：随机删除 Key，实现简单无状态。
- **适用场景**：适合不能统一设置 TTL，或缓存中有重要长时有效数据的系统。

S

**建议：**

- 若 Redis 作缓存使用，推荐 `allkeys-LRU`。
- 对访问频率高度失衡的情况，可考虑 `allkeys-LFU`。
- 若结合 TTL 使用，推荐 `volatile-LRU`。
- 必须保障写入，业务异常优先，则选择 `noeviction`。

这些策略帮助 Redis 高效控制内存，兼顾性能与数据完整性，是运维和架构优化时需重点掌握的内容。
