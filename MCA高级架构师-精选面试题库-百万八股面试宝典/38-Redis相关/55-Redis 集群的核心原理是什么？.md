Redis 集群通过**数据分片 + 主从复制 + 自动故障转移**实现高可用性与水平扩展。其原理可以分为以下几个方面：

M

### 1. 数据分片（Hash Slot 概念）

Redis 将整个 key 空间划分为 **16 384 个哈希槽（hash slots）**，通过 `CRC16(key) % 16384` 将每个 key 映射到固定槽位。

每个集群节点（Master）负责一段连续槽的读写请求。当新的节点加入或下线时，集群会按槽重新分配，保持负载均衡。

### 2. 主从复制

- 每个 Master 通常配备一个或多个 **Slave 备份节点**，通过异步复制确保数据冗余与读扩展能力。
- Slave 可用于读取请求，减少主节点压力，提升读吞吐量。

S

### 3. 客户端路由机制

- 客户端初次连接集群后，会缓存各节点所负责槽的信息，并通过哈希计算，直接向正确节点发送命令，避免跨节点请求。
- 若请求命中了错误节点，该节点会返回 `MOVED` 或 `ASK` 重定向指令，指导客户端重路由。

### 4. 故障检测与自动故障转移

- 集群内部节点使用 **Gossip 协议** 定期交换状态，实现节点故障检测。
- 当 Master 被多数节点判定为“失联”后，会自动选举其某个 Slave 升为 Master，接手其槽位，实现无缝故障转移。

B

### 5. 架构扩展与弹性

- 通过 Hash Slot 机制，集群可连接数十至上百个 Master 节点，可扩展至数百 TB 数据容量。
- 增加节点仅涉及槽的重新分配，而不用修改客户端业务逻辑，支持横向扩展。
