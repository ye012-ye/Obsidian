Redis 提供三种持久化方式：RDB 快照、AOF 增量日志与混合持久化，各自的机制与适用场景如下所述。

M

### 1. RDB（快照持久化）

Redis 会按照配置规则（如 “保存 n 次写入后 m 秒执行”，或手动 `BGSAVE`/`SAVE`）自动或手动生成内存中的数据快照，并 fork 出子进程写入二进制文件。子进程处理持久化，不阻塞主线程，引入内存写时复制优势。​

- **优点**

- 占用空间紧凑，文件利于备份和迁移
- 重启恢复迅速

- **缺点**

- 存在写入窗口内数据丢失风险
- BGSAVE fork 在大数据量下可能造成短暂阻塞或延迟

S

### 2. AOF（追加日志持久化）

Redis 会将每个写操作追加到 AOF 文件，并按策略 `appendfsync`: `"always"`（同步每条操作）、`"everysec"`（每秒同步）或 `"no"`（依赖 OS）。日志可重写以裁剪冗余命令。恢复时通过重放命令重建数据集。

- **优点**

- 接近实时持久化（尤其 `always` 模式，写后立即落盘）
- 日志可修复，数据恢复更完整

- **缺点**

- 文件较大，恢复和 IO 负担较重
- “always” fsync 会显著降低写性能

B

### 3. 混合持久化（RDB + AOF）

从 Redis 4.0 起支持 AOF 文件中嵌入 RDB 前缀，即在 AOF 重写时将快照和增量日志结合，重启时优先加载 AOF 并回放后续日志，实现快速恢复与较强持久性平衡。

- **优点**

- 快速 重启 + 数据完整度更高

- **缺点**

- 配置复杂，仍有极小丢失窗口

### 4. 各机制比较与选用建议

|  |  |  |  |
| --- | --- | --- | --- |
| 特性 | RDB | AOF（everysec） | 混合持久化 |
| 写入性能 | 最优（BGSAVE 非阻塞） | 中等（fsync 每秒） | 较好（基本类似 AOF） |
| 重启恢复速度 | 快 | 慢 | 介于两者之间 |
| 数据完整性 | 可丢失最近几个秒的数据 | 丢失窗口 <1s（取决于 fsync 策略） | 更强持久性 |
| 文件体积 | 紧凑 | 最大 | 介于两者之间 |

- **RDB 独用**：适合缓存应用，对数据丢失容忍、重启速度优先
- **AOF 独用（everysec）**：适合要求尽量不丢数据的场景
- **混合模式**：兼顾性能与数据完整性，是一般推荐配置
