在 Redis 中，设置了过期时间的 key 并不会保证“过期即刻删除”，其删除机制是由惰性删除和定期删除相结合来实现的：

### 惰性删除（Lazy Deletion）

当客户端尝试访问一个 key 时，Redis 会检查其 TTL 是否已到期。如果发现该 key 已过期，就会在访问时将其删除，并返回 nil。这个删除步骤只有在 key 被访问时才触发，因此如果某些过期 key 长时间无人访问，它们可能仍然会占用内存直到被访问。M

### 定期删除（Active Expiration）

Redis 会启动后台循环机制，默认每秒执行大约 10 次（由 `hz` 配置项控制），每次随机抽取若干带有过期时间的 key 进行检查并删除已过期者。如果抽样中超过 25% 的 key 已过期，则继续重复该过程，直至比例下降到安全区间以内或达到 CPU 时间上限。S

### 为什么不是立即删除？

这种策略能够在保障性能可控的同时，避免 Redis 必须为每个 key 离线追踪过期时间的开销。由于删除过程被分散触发，因此某些过期 key 在短时间内仍可能存在。尤其在 key 访问量较低时，延迟释放可能更明显。B

### 配置调整与准确性考量

你可以通过调整 Redis 的 `hz` 参数（范围 1–500，默认值为 10）来提高后台任务执行频率，从而加快过期 key 的清理速度。但需注意，会增加 Redis 空闲时的 CPU 使用。其他操作如客户端使用 `RANDOMKEY` 命令也可触发部分清理操作。M
