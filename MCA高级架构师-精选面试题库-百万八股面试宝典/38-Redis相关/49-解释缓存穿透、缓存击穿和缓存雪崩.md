### 1. 缓存穿透（Cache Penetration）

**定义与场景**  
请求查询的 Key 不存在于缓存也不存在于数据库，恶意或异常请求持续冲击数据库，背景压垮数据库。这类问题通常由用户级攻击或非法请求引发。

**优化策略**

- 将空结果缓存在 Redis 中，并设置较短 TTL，避免频繁命中数据库。
- 使用布隆过滤器（Bloom Filter）预先判断 Key 是否可能存在，大批不存在的请求可直接拒绝或快速返回，节省数据库开销。

M

### ​2. 缓存击穿 / 缓存断裂（Cache Stampede / Breakdown）

**定义与场景**  
大量并发请求集中访问某一个热门 Key，在缓存失效后同时穿透缓存直击数据库，造成请求洪峰。

**优化策略**

- 对 Key 加锁（如 SETNX 或互斥锁），确保只有一个线程请求数据库并重建缓存，其他线程等待或直接返回旧缓存。
- 使用异步刷新机制，后台定期更新热门 Key 缓存，避免请求触发 expensive 查询。

S

### ​3. 缓存雪崩（Cache Avalanche）

**定义与场景**  
大量 Key 同时过期或缓存服务短暂宕机，导致请求集中打到数据库，引发雪崩式压力。

**优化策略**

- 为缓存设置随机 TTL，避免集中失效。
- 架构上加强缓存集群冗余，例如 Redis Cluster、Sentinel+主从复制等。
- 引入熔断、限流、防压机制，当检测到底层拥塞时暂缓或拒绝部分请求。

B

### 总结对比

|  |  |  |  |
| --- | --- | --- | --- |
| **问题类型** | **触发机制** | **关键风险** | **核心防御措施** |
| 缓存穿透 | 虚假请求绕过缓存打数据库 | 频繁无效数据库访问 | 缓存空值、布隆过滤器 |
| 缓存击穿 | 热点 Key 丢失缓存并发涌入 | 数据库瞬时洪峰请求 | 互斥锁（SETNX）、异步刷新机制 |
| 缓存雪崩 | 大批 Key 同时过期或缓存故障 | 多 Key 同时穿透到 DB | 随机 TTL、缓存多级架构、熔断限流机制 |
