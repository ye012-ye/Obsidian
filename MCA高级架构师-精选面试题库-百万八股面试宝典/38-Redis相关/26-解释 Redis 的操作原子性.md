Redis 承诺单个命令或脚本运行时具有原子性：每个操作要么全部执行成功，要么完全不执行。这种保障源自其单线程 I/O 模型与脚本与事务机制的支持。 Redis 的操作原子性集中体现在以下几个方面：

M

### 1. 单个命令的原子性

Redis 基于单线程架构执行请求，每条客户端命令都是按顺序、独立地执行的。因为不存在并行执行，不可能在命令执行过程中被打断或插入其他命令。举例来说，`INCR`、`DECR`、`SET` 等命令在执行期间不会被中断，因此天然具备原子性 。

### 2. MULTI/EXEC 事务机制

当需要对多个命令进行原子执行时，可以使用 `MULTI` + 一系列命令 + `EXEC`。Redis 会将这些命令序列化并一起执行，在提交时保证所有命令一次完成，其他客户端无法在中间插入命令。事务保证了执行顺序，但不支持中途回滚；如果没有 `WATCH` 操作，提交一定成功，不会失败。若有使用 `WATCH` 检测 key 更改，则可能触发事务放弃执行，客户端需重试。S

### 3. Lua 脚本的原子执行

Lua 脚本在 Redis 中作为一个整体执行，Redis 在执行期间不会调度其他命令，保持完全一致性和原子性。脚本里所有 `redis.call(...)` 的 Redis 操作都被视为一个不可中断的逻辑单元。这比事务更强大，可以在脚本中条件判断、读取和写入，再统一提交结果。即便脚本中包含多个读写操作，也不会被并发打扰。

### 使用场景与推荐

- 单一简单操作，如 `INCR`, `HSET`, `LPUSH` 等天然原子，可直接使用。
- 需要批量修改且不允许中间状态被其他客户端看到时，可使用 `MULTI/EXEC` 事务。
- 更复杂的条件逻辑或多个数据操作需组合执行时，推荐使用 Lua 脚本来保持原子性与效率，并避免网络往返。

### 原子性局限与注意点

- **非事务操作之间仍可能交叉执行**：Pipeline 并不提供原子性，不同客户端的 pipeline 命令可能交错执行，无法保证整体批量操作作为事务行为。
- **脚本执行时间要适度**：Lua 脚本在运行期间阻塞 Redis 主线程，如果脚本执行时间过长，可能影响其他客户端响应能力，造成性能瓶颈或阻塞。应避免执行过于复杂或耗时逻辑。
- **Cluster 模型下的约束**：MULTI/EXEC 与 Lua 脚本需确保所有涉及 key 位于同一 Redis 分片节点上，否则可能失败或报错；此时需合理设计 key 标签机制和请求路由。

### 总结:Redis 的原子性依赖于其单线程结构，使每条命令天然原子；事务（MULTI/EXEC）和 Lua 脚本进一步支持组合操作的原子执行。在日常使用中，对于独立简单命令原子性天然满足；对于复杂多步骤逻辑，建议优先使用 Lua 脚本确保正确性和性能。但需注意脚本执行时间、分片约束以及 pipeline 不具原子性等问题。
