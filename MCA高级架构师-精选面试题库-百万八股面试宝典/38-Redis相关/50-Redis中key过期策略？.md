Redis 采用了一种混合过期机制，即结合“惰性删除”和“定期删除”，在性能与内存回收间寻找平衡，针对访问和后台清理双重角度进行设计。

M

### 1. 惰性删除（Lazy/Passive Expiration）

当客户端请求访问某个带 TTL 的 Key 时，Redis 会首先判断是否已经过期，如果已过期，则立即删除并返回空值。这种策略不会频繁检查 Key，有效节省 CPU 资源。但缺点是，如果有大量过期 Key 始终无人访问，它们会长期滞留，占用内存并可能导致空间浪费。

S

### 2. 定期删除（Active/Active Expiration）

Redis 会定期（默认每 100ms）随机抽样已设置过期时间的 Key（每次 20 个），检查哪些已过期，并删除之。如果超过 25% 的抽样 Key 已过期，Redis 会继续抽样，直到比例降低。这种机制用于主动清理未被访问的过期数据，减少内存占用。不过它可能会消耗一定 CPU，并且无法保证“全部”过期 Key 都被及时清除。

B

### 3. Redis 6 优化（Radix Tree 辅助扫描）

从 Redis 6 开始，定期扫描逻辑得以优化，新增了一个 Radix Tree 索引用于存储即将过期的 Key。接下来扫描会优先检查这些即将过期的 Key，从而减少滞留，提升过期效率，同时减少过度抽样和 CPU 开销。
