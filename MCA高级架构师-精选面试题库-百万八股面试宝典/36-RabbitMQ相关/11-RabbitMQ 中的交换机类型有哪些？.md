在 RabbitMQ 中，交换机（Exchange）负责将生产者发送的消息路由到一个或多个队列。主要有四种标准类型：M

### 1. Direct 交换机

Direct 交换机通过绝对匹配 routing key 来分发消息：只有当消息的 routing key 与队列绑定时所设置的 binding key 完全一致，消息才会被路由至该队列。它通常用于一对一或一对多的精确路由场景，当多个队列绑定了相同的 key 时，也可用于将消息广播给多个目标。

### 2. Fanout 交换机

Fanout 交换机忽略 routing key，将接收到的每条消息广播给所有绑定了该交换机的队列。它适用于广播式场景，例如发布/订阅模型或通知系统。

### 3. Topic 交换机

Topic 交换机允许使用通配符（`*` 匹配一个单词，`#` 匹配零个或多个单词）对 routing key 进行模糊匹配。例如，绑定 `"order.*.created"` 可匹配 `"order.us.created"`，而 `"order.#"` 可匹配所有以 `"order."` 开头的 routing key。这种模式适合按主题分类、日志或分区路由的场景。

S

### 4. Headers 交换机

Headers 交换机基于消息头（headers）中的属性进行路由，忽略 routing key。绑定时可以设置多个键值对，并通过 `x-match=all` （全部匹配）或 `x-match=any`（任意匹配）来控制路由策略。这种方式灵活性高，适用于按多维属性筛选消息的场景。

​

**此外，还有两种常见但非“核心标准”交换机制：**

### 1. 默认交换机（Default Exchange）

RabbitMQ 内置一个无名 direct 交换机（名字为空字符串），自动将 routing key 与队列名称匹配。这使得生产者可以直接向某个队列发送消息而无需声明交换机。

B

### 2. 死信交换机（Dead Letter Exchange）

用于接收未被正常消费或被拒绝的消息。可将这些“死信”定向到另一个队列进行后续处理，如记录日志、重试或人工干预。

**总结：**

|  |  |  |
| --- | --- | --- |
| **交换机类型** | **路由方式** | **适用场景** |
| Direct | 精确匹配 routing key | 精准路由、一对一或一对多 |
| Fanout | 广播给所有绑定队列 | 广播、发布/订阅通知 |
| Topic | 支持通配符的模糊匹配 | 按主题筛选、多级分类 |
| Headers | 根据消息头属性匹配 | 多维属性筛选、复杂路由 |
| Default | 默认 direct，无需声明 | 简化单队列发送 |
| Dead Letter | 捕获未成功消费的消息 | 异常消息处理、重试机制 |

每种交换机类型各司其职，结合 binding 和 routing key，可构建灵活的消息分发拓扑，满足微服务、日志监控、任务分发等多种实时场景。
