在 Java 内存模型中，“运行时常量池”与“字符串常量池”属于两个不同层次的概念：

#### 1. 运行时常量池（Runtime Constant Pool）

- 来源于 class 文件的常量池表，JVM 在加载类或接口时，将该表内的编译期常量（如整数、浮点、字符串字面量、方法和字段引用等）加载到方法区或 Metaspace 中。M
- 它属于 *方法区*（逻辑上也是堆的一部分），由 JVM 管理，用于类级别的常量访问。

#### 2. 字符串常量池（String Constant Pool）

- 是 JVM 中专门处理字符串字面量的运行时结构。它只存储 `String` 对象，用于实现字符串重用。S
- 自 Java 7 起，这个池位于 *堆内存* 中，允许 GC 回收不再引用的常量字符串，从而缓解 PermGen 限制问题。

### 二者关系与区别

|  |  |  |
| --- | --- | --- |
| **特性** | **运行时常量池** | **字符串常量池** |
| 内容 | 含各种常量（数值、字段/方法引用、字符串字面量等） | 仅限 `String`  类型字面量 |
| 存放位置 | 方法区或 Metaspace（JDK 8+） | 堆内存 |
| 生命周期 | 类加载即存在，类卸载时释放 | 由 JVM 管理，GC 回收可达性为 null 的字符串 |
| 管理方式 | JVM 内部结构，支持符号引用和解析 | 实时缓存，线程安全，intern 实现共享 |

总结：

- **运行时常量池** 是 class 文件结构的一部分，面向所有类型常量，存放于 *方法区/Metaspace*。
- **字符串常量池** 是 JVM 在 *堆* 中特别为 `String` 常量维护的池，便于重用、节省内存，并支持 intern 操作。B
- 两者互补：前者用于解析 bytecode 的常量，后者则是 Java 程序运行时用于优化字符串实例共享的机制。
