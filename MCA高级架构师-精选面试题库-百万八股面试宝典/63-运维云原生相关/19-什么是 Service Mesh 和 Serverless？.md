云原生架构中，**Service Mesh** 和 **Serverless** 分别代表了微服务通信治理与无服务器执行模型中的两类关键技术，它们解决的问题不同，但常可协同使用。

M

### Service Mesh

Service Mesh 是构建微服务系统时的一个基础设施层，通常以 sidecar 代理（如 Envoy）形式部署在每个服务实例旁边，通过统一控制平面管理这些代理。它承担服务发现、流量管理、负载均衡、互信认证（如 mTLS）、故障重试/熔断以及可观测性等功能，无需对业务代码进行侵入性改造。

**优点**：

- 提供细粒度的流量控制，如 A/B 测试、金丝雀发布和流量分流；
- 加固服务间通信的安全性，实现零信任架构；
- 带来强监控和追踪能力，有利于故障诊断。

**缺点**：

- 增加基础设施复杂度和性能开销；
- 部署侧车代理后，每个服务实例都需要额外组件，管理成本提升。

**典型平台**：Istio、Linkerd、Consul Connect、AWS App Mesh、Traefik Mesh 等。

S

### Serverless

Serverless（或 Function as a Service）是一种事件驱动执行模型，开发者只需编写函数代码，部署到云平台（如 AWS Lambda、Azure Functions、Google Cloud Functions 等），平台负责自动调度、扩缩容和运行时管理，无需运维服务器资源。

**优点**：

- 极低启动门槛，开发者专注业务代码；
- 自动弹性扩缩容、按调用量计费，有效控制成本；
- 加快部署速度，适合快速迭代的小型服务或任务。

**缺点**：

- 冷启动延迟、状态管理困难；
- 易受平台限制与供应商锁定影响；
- 不适合长时任务或高性能计算场景。

B

### Service Mesh vs Serverless 对比分析

|  |  |  |  |
| --- | --- | --- | --- |
| **技术** | **关注层级** | **主要职责** | **典型适用场景** |
| Service Mesh | 微服务通信层 | 管理服务间流量、安全与可观测性 | 多微服务、高可靠性、复杂流量治理需求的系统 |
| Serverless | 业务执行层 | 事件驱动函数执行、资源自动分配与计费 | 无状态函数、小型事件响应任务、快速迭代需求 |
