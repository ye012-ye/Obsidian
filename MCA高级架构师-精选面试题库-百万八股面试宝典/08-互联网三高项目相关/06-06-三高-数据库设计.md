# 死锁

一个请求的时候，无所谓。

当同时 高并发请求来时，会发生多个 操作方因为竞争资源而互相等待，造成了 没有外界参与

无法打破的僵局。

例子：房间2把锁，同时有两个钥匙，才能打开。

A和B，每个人拿了一把锁，并且 绝不给别人。

乐观锁会死锁吗？不会，因为不占用资源。

四个条件：

分析产生某个事情的条件，然后根据条件，逐一解决，问题就解决了。

**互斥条件：钥匙，一个人拿了。另一个人不能拿。**

**不剥夺条件：也就是拿了锁的人，没人能让他放手。**

**请求和保持条件：有了一个资源，再请求另外一个资源。**

**循环等待条件：环状。**

解决它：最近线上项目经常死锁

打破其中一个即可。

打破互斥条件：经常发生在 写操作的时候。x锁，而此时，所以，资源很难共享。乐观锁 变相 处理。

打破不剥夺条件：要求：操作方在已获得资源的条件下，如果呀申请别的资源，如果申请失败，放手。

主动让出资源。

打破请求和保持条件：保持、请求，只留一个。

只请求不保持：只能持有一个资源，要请求其他资源，放手。

只保持不请求：媳妇和女朋友，同时拥有，别再去找其他资源了。

打破循环等待条件：按顺序来。排好序，葫芦娃救爷爷-----一个一个来。

一般打破，后三个条件。

目的：写代码：防止死锁，破防之后，在死锁发生时，尽快定位并解决。

定位：

程序死锁如何定位：看程序静止的点。分析线程dump文件定位死锁。（课：jvm调优实战）

数据库死锁如何定位：查看死锁日志。（课：mysql调优）

解决：

根据上面4个条件，找一个破掉。

# 读写分离

还可以用于：数据备份。

X锁，阻止了其他操作的 读和写。

S锁：可以并发读。

将 读写分开，保证读不受写的约束。还可以 将多个写 并发执行。

适用于：读多写少。

读少写多，提升效果并不明显。

路由：

select ---读库上

add update delete create ---写库上。

主从复制：

确保主从数据的一致性。

主：写

从：读。

将主库的数据同步到从库，避免：对 从库的写操作，不让从库上出现X锁。

binlog。

1。主库开启日志记录功能。写操作记录在binlog中。

2。binlog 被发到 从库中，写入从库的RelayLog。

3。从库解析RelayLog，重现数据。

STATEMENT:操作语句。如果一些 涉及到 当前时间的函数。now（），数据不一致了。

Row：记录。

mixed：操作和记录的混合。主要写入操作语句，必要的时候，记录。（大部分情况用这个模式）

主从复制的延迟问题：

日志：记录，传输，解析，

异步复制：降低写的响应时间，可能导致数据不一致或丢失。

半同步复制：写请求，写入主库后，日志至少被送到一个从库上，写才返回。

谷歌开发了mysql半同步的插件。

全同步复制：全写完，才响应用户。
