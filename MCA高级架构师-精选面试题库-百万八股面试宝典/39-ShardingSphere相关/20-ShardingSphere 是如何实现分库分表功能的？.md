ShardingSphere 实现分库分表的核心机制包括 SQL 解析、路由、重写、执行与结果归并，各阶段职责清晰、高度流水化设计。

### 1. **SQL 解析阶段**

首先对用户提交的 SQL 进行词法与语法解析，生成一棵抽象语法树（AST），构造解析上下文，提炼出表、分组、分页、筛选条件等信息，以便后续决策使用。

M

### 2. **执行器优化阶段**

对解析得到的上下文进行优化处理，比如合并 OR 条件、整理分片键过滤条件等，提升后续路由效率和准确性。

### 3. **SQL 路由阶段**

根据用户配置的分片规则（包括表的分片键与分片算法），以及查询条件，确定实际路由目标；支持标准分片路由、广播路由乃至笛卡尔积路由。例：分片查询 IN 操作时，会生成多条针对各分片的 SQL。

S

### 4. **SQL 改写阶段**

将逻辑 SQL 重写为一个或多个可在真实数据库执行的 SQL，实现两类改写：正确性改写（如替换逻辑表名为真实表名）、优化改写（如合并 SQL、多表 JOIN 改写）。

### 5. **SQL 执行阶段**

ShardingSphere 内置一个执行器引擎，通常是多线程异步方式，将拆分后的真实 SQL 分发到底层多个数据源并发执行，兼容 JDBC 连接池或 Proxy 模式。

B

### 6. **结果归并阶段**

将各数据节点返回的结果集进行统一归并处理，支持流式归并和内存归并等策略，并在需要时进行排序、聚合等操作后返回给客户端。整个过程对客户端透明，仅暴露统一的 JDBC 接口。

### ​总结

ShardingSphere 的分库分表功能通过以下流程实现高性能的分布式查询：SQL 解析 → 执行优化 → 路由判断 → SQL 重写 → 执行调度 → 结果归并。应用程序仅需操作逻辑表，系统自动路由到真实分片节点，整个逻辑对业务透明，大幅提升性能与可扩展性。
