在分库分表架构下执行分页查询比较复杂，因为数据分布于多个分片，不能直接使用单库的 `LIMIT OFFSET` 机制。ShardingSphere 提供了自动分页支持及优化机制，具体方式如下：

### 一、ShardingSphere 自动分页处理机制

对于 MySQL、PostgreSQL 和 openGauss，ShardingSphere 完全支持 `LIMIT … OFFSET …` 分页查询，并会自动将逻辑 SQL 改写为跨分片查询，同时按排序字段执行流式聚合排序与归并，最终截取所需页码数据后返回给客户端。

但如果查询跨多个分片，存在高偏移分页（如跳过上百万条），ShardingSphere 会将原始 `LIMIT m, n` rewrite 为 `LIMIT 0, m+n`，即先抓取偏移前的所有记录，再排序后截出目标页。虽然正确性得到保证，但带来性能开销与带宽浪费。

为避免过度内存占用，它采用“流式处理 + 归并排序”的方式，仅在内存保存各分片当前游标对应记录，归并复杂度为 O(n)，内存消耗极低。

当查询仅落至单个分片时，ShardingSphere 会跳过 SQL 改写步骤直接执行，节省性能开销。

### 二、推荐优化方案

#### 1. **基于 ID 或全局排序字段的分页查询**

建议使用稳定、有序的字段（如主键ID、时间戳）做分页，避免 `OFFSET`。示例：

```plsql
SELECT * FROM t_order WHERE id > 100000 AND id <= 100010 ORDER BY id;
```

或使用最后一条记录的 ID 继续查询：

```plsql
SELECT * FROM t_order WHERE id > 100000 ORDER BY id LIMIT 10;
```

该方式可跳过大偏移计算，减少跨分片查询负载 。

#### 2. **控制页码深度／禁止跳页查询**

如果业务允许，可禁止超过一定页码的跳页查询，或使用限制页大小的方式避免深度分页。这样可极大提升查询效率，适用于业务可接受一定查询约束的场景。

#### 3. **二次查询法（分片辅助查询）**

先根据分片键或全局索引快速定位目标数据范围，再按范围分别去对应分片查询具体分页数据，并在应用层合并排序。这种方式返回数据量小、一致性好，但实现相对复杂且适用于明确范围查询场景。

#### 4. **使用外部索引／搜索架构（如 ES）支持分页**

对于非分片键或者复杂查询场景，可将数据预先同步至 Elasticsearch 或其他搜索引擎，通过其索引完成分页定位，再回源至数据库分片获取详细数据。这种方式适合复杂查询、高并发分页场景，但需维护一致性和同步机制。
