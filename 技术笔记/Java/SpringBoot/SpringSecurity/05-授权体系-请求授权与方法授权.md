# 05-授权体系-请求授权与方法授权

## 1. 授权的两个层次

Spring Security 常用授权分两层：

1. 请求级授权：请求能不能进入某个 URL。
2. 方法级授权：当前用户能不能调用某个 Service 方法。

请求级授权适合粗粒度入口控制：

```java
http.authorizeHttpRequests(authorize -> authorize
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/orders/**").hasAuthority("order:read")
    .anyRequest().authenticated()
);
```

方法级授权适合业务语义更强的控制：

```java
@PreAuthorize("hasAuthority('order:refund')")
public void refund(Long orderId) {
    // 退款
}
```

实践建议：

1. URL 层拦住明显不该访问的入口。
2. Service 层保护关键业务动作。
3. 不要只依赖前端按钮隐藏。
4. 不要只在 Controller 判断权限，Service 被其他入口调用时也可能绕开。

## 2. authorizeHttpRequests

基础写法：

```java
@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests(authorize -> authorize
        .requestMatchers("/", "/login", "/assets/**").permitAll()
        .requestMatchers(HttpMethod.GET, "/api/articles/**").permitAll()
        .requestMatchers(HttpMethod.POST, "/api/articles/**").hasAuthority("article:create")
        .requestMatchers("/admin/**").hasRole("ADMIN")
        .anyRequest().authenticated()
    );

    return http.build();
}
```

规则含义：

| 规则 | 含义 |
|---|---|
| `permitAll()` | 公开访问 |
| `denyAll()` | 永远拒绝 |
| `authenticated()` | 需要登录 |
| `anonymous()` | 只允许匿名用户 |
| `hasRole("ADMIN")` | 需要 `ROLE_ADMIN` |
| `hasAuthority("article:create")` | 需要精确权限 |
| `hasAnyRole("ADMIN", "OPS")` | 满足任一角色 |
| `hasAnyAuthority(...)` | 满足任一权限 |
| `access(...)` | 自定义授权逻辑 |

## 3. requestMatchers 的匹配顺序

授权规则按顺序匹配，越具体越靠前。

正确：

```java
.authorizeHttpRequests(authorize -> authorize
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/**").authenticated()
    .anyRequest().permitAll()
)
```

错误：

```java
.authorizeHttpRequests(authorize -> authorize
    .requestMatchers("/api/**").authenticated()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .anyRequest().permitAll()
)
```

第二段里 `/api/admin/**` 会先被 `/api/**` 捕获，管理员规则失效。

## 4. Role 和 Authority

Spring Security 底层只认识 `GrantedAuthority`。

`Role` 是一种带前缀约定的 authority。

```java
.hasRole("ADMIN")
```

等价于检查：

```java
.hasAuthority("ROLE_ADMIN")
```

因此：

| 数据库存储 | 代码检查 |
|---|---|
| `ROLE_ADMIN` | `hasRole("ADMIN")` 或 `hasAuthority("ROLE_ADMIN")` |
| `ADMIN` | `hasAuthority("ADMIN")` |
| `user:read` | `hasAuthority("user:read")` |

建议：

1. 角色统一存 `ROLE_` 前缀，权限统一存 `resource:action`。
2. Controller 和 Service 里优先使用细粒度权限。
3. 后台菜单可以按权限过滤，但接口仍要后端校验。

## 5. 方法级授权

启用：

```java
@Configuration
@EnableMethodSecurity
public class MethodSecurityConfig {
}
```

Spring Boot starter security 不会默认启用方法级授权，需要自己加 `@EnableMethodSecurity`。

常用注解：

| 注解 | 作用 |
|---|---|
| `@PreAuthorize` | 方法执行前校验 |
| `@PostAuthorize` | 方法执行后根据返回值校验 |
| `@PreFilter` | 方法执行前过滤集合参数 |
| `@PostFilter` | 方法执行后过滤集合返回值 |
| `@Secured` | 老式角色注解，需显式启用 |
| `@RolesAllowed` | JSR-250 注解，需显式启用 |

最常用的是 `@PreAuthorize`。

## 6. @PreAuthorize 示例

按权限：

```java
@PreAuthorize("hasAuthority('user:read')")
public UserDetail getUser(Long id) {
    return userRepository.findDetail(id);
}
```

按角色：

```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long id) {
    userRepository.deleteById(id);
}
```

按参数：

```java
@PreAuthorize("#userId == authentication.principal.id or hasAuthority('user:read:any')")
public UserDetail getUser(Long userId) {
    return userRepository.findDetail(userId);
}
```

调用自定义权限服务：

```java
@PreAuthorize("@permissionService.canReadOrder(authentication, #orderId)")
public OrderDetail getOrder(Long orderId) {
    return orderRepository.findDetail(orderId);
}
```

权限服务：

```java
@Service
public class PermissionService {

    public boolean canReadOrder(Authentication authentication, Long orderId) {
        if (hasAuthority(authentication, "order:read:any")) {
            return true;
        }

        Long currentUserId = ((LoginUser) authentication.getPrincipal()).getId();
        return orderRepository.existsByIdAndUserId(orderId, currentUserId);
    }

    private boolean hasAuthority(Authentication authentication, String authority) {
        return authentication.getAuthorities().stream()
            .anyMatch(item -> item.getAuthority().equals(authority));
    }
}
```

## 7. @PostAuthorize

`@PostAuthorize` 可以根据返回对象判断。

```java
@PostAuthorize("returnObject.ownerId == authentication.principal.id")
public Document getDocument(Long id) {
    return documentRepository.findById(id).orElseThrow();
}
```

注意：

1. 方法已经执行完，数据已经查出来。
2. 不适合用于避免敏感查询本身。
3. 更适合对返回对象做兜底校验。

## 8. AuthorizationManager

现代 Spring Security 底层越来越多使用 `AuthorizationManager`。

自定义请求授权：

```java
@Component
public class TenantAuthorizationManager implements AuthorizationManager<RequestAuthorizationContext> {

    @Override
    public AuthorizationDecision check(
            Supplier<Authentication> authentication,
            RequestAuthorizationContext context
    ) {
        HttpServletRequest request = context.getRequest();
        String tenantId = request.getHeader("X-Tenant-Id");

        boolean granted = authentication.get().getAuthorities().stream()
            .anyMatch(authority -> authority.getAuthority().equals("tenant:" + tenantId + ":access"));

        return new AuthorizationDecision(granted);
    }
}
```

配置：

```java
http.authorizeHttpRequests(authorize -> authorize
    .requestMatchers("/tenant/**").access(tenantAuthorizationManager)
    .anyRequest().authenticated()
);
```

适合：

1. 多租户。
2. 复杂资源路径。
3. 动态权限策略。
4. 需要复用授权逻辑。

## 9. 权限表设计

典型 RBAC 表：

```text
sys_user
sys_role
sys_permission
sys_user_role
sys_role_permission
```

示例权限：

| 权限编码 | 含义 |
|---|---|
| `user:read` | 查看用户 |
| `user:create` | 新增用户 |
| `user:update` | 修改用户 |
| `user:delete` | 删除用户 |
| `order:read` | 查看订单 |
| `order:refund` | 订单退款 |

菜单和权限不要混为一谈：

1. 菜单是前端展示。
2. 权限是后端决策。
3. 一个权限可以对应多个按钮或接口。
4. 一个接口也可能需要多个权限之一。

## 10. 权限粒度怎么选

太粗：

```text
ADMIN
USER
```

问题：管理员权限越来越大，难以细分。

太细：

```text
button:user-page-left-top-create-click
```

问题：和 UI 绑定过深，维护困难。

推荐：

```text
资源:动作
user:read
user:create
user:update
user:delete
order:refund
report:export
```

## 11. 本章小结

授权设计原则：

1. 默认拒绝，明确放行。
2. 公开接口写清楚 `permitAll()`。
3. 业务接口至少 `authenticated()`。
4. 管理接口使用角色或权限。
5. 核心业务动作加方法级授权。
6. 权限编码稳定，不和页面文案强绑定。
7. 401 查认证，403 查权限和 CSRF。
