# 09-测试-排障与最佳实践

## 1. 测试依赖

Spring Security 提供测试支持：

```xml
<dependency>
    <groupId>org.springframework.security</groupId>
    <artifactId>spring-security-test</artifactId>
    <scope>test</scope>
</dependency>
```

常用于：

1. Mock 当前登录用户。
2. Mock 权限。
3. 测试 CSRF。
4. 测试 JWT。
5. 测试 URL 授权和方法授权。

## 2. MockMvc 测试 URL 权限

```java
@SpringBootTest
@AutoConfigureMockMvc
class UserApiSecurityTest {

    @Autowired
    MockMvc mockMvc;

    @Test
    void anonymousCannotAccessUserApi() throws Exception {
        mockMvc.perform(get("/api/users/me"))
            .andExpect(status().isUnauthorized());
    }

    @Test
    @WithMockUser(username = "alice", authorities = "user:read")
    void userWithPermissionCanAccess() throws Exception {
        mockMvc.perform(get("/api/users/me"))
            .andExpect(status().isOk());
    }

    @Test
    @WithMockUser(username = "bob", authorities = "order:read")
    void userWithoutPermissionCannotAccess() throws Exception {
        mockMvc.perform(get("/api/users/me"))
            .andExpect(status().isForbidden());
    }
}
```

## 3. @WithMockUser

角色：

```java
@WithMockUser(username = "admin", roles = "ADMIN")
```

等价于拥有：

```text
ROLE_ADMIN
```

权限：

```java
@WithMockUser(username = "alice", authorities = {"user:read", "order:create"})
```

注意：

1. `roles` 会自动加 `ROLE_`。
2. `authorities` 不会自动加前缀。
3. 同一个注解里不要混乱使用角色和权限。

## 4. 测试 CSRF

如果启用了 CSRF，POST 请求需要带 token：

```java
mockMvc.perform(post("/api/articles")
        .with(csrf())
        .contentType(MediaType.APPLICATION_JSON)
        .content("""
            {"title":"hello"}
            """))
    .andExpect(status().isOk());
```

无 CSRF token：

```java
mockMvc.perform(post("/api/articles")
        .contentType(MediaType.APPLICATION_JSON)
        .content("""
            {"title":"hello"}
            """))
    .andExpect(status().isForbidden());
```

如果你的项目是无状态 Bearer Token API 并关闭 CSRF，测试不需要 `.with(csrf())`。

## 5. 测试 JWT Resource Server

使用 `jwt()`：

```java
import static org.springframework.security.test.web.servlet.request.SecurityMockMvcRequestPostProcessors.jwt;

@Test
void jwtWithScopeCanAccess() throws Exception {
    mockMvc.perform(get("/api/messages")
            .with(jwt().authorities(new SimpleGrantedAuthority("SCOPE_message:read"))))
        .andExpect(status().isOk());
}
```

带 claim：

```java
mockMvc.perform(get("/api/me")
        .with(jwt().jwt(jwt -> jwt
            .subject("alice")
            .claim("permissions", List.of("user:read"))
        )))
    .andExpect(status().isOk());
```

如果你自定义了 `JwtAuthenticationConverter`，要确认测试中构造的 authorities 与生产转换结果一致。

## 6. 测试方法授权

```java
@SpringBootTest
class UserAdminServiceSecurityTest {

    @Autowired
    UserAdminService userAdminService;

    @Test
    @WithMockUser(authorities = "user:create")
    void createUserAllowed() {
        Long id = userAdminService.createUser(new CreateUserCommand("alice"));
        assertThat(id).isNotNull();
    }

    @Test
    @WithMockUser(authorities = "user:read")
    void createUserDenied() {
        assertThatThrownBy(() -> userAdminService.createUser(new CreateUserCommand("alice")))
            .isInstanceOf(AccessDeniedException.class);
    }
}
```

前提：配置了 `@EnableMethodSecurity`。

## 7. 打开调试日志

配置：

```yaml
logging:
  level:
    org.springframework.security: DEBUG
```

可以看到：

1. 请求匹配了哪些安全规则。
2. 认证过滤器是否执行。
3. 匿名认证是否生效。
4. 授权为什么拒绝。

也可以临时启用：

```java
@EnableWebSecurity(debug = true)
```

注意：debug 模式会输出较多信息，不建议生产开启。

## 8. 401 排查

401 表示未认证或认证失败。

排查清单：

1. 请求是否带了 Cookie 或 Authorization 头？
2. Bearer token 是否过期？
3. JWT 签名算法和密钥是否匹配？
4. `issuer-uri` 是否正确？
5. `aud`、`iss`、`exp`、`nbf` 是否校验失败？
6. Basic 账号密码是否正确？
7. 表单登录提交地址是否匹配 `loginProcessingUrl`？
8. 自定义登录成功后是否保存 Session 或返回 token？

典型现象：

| 现象 | 可能原因 |
|---|---|
| 浏览器跳登录页 | 当前是表单登录入口 |
| API 返回 HTML 登录页 | 没有自定义 `AuthenticationEntryPoint` |
| Bearer token 仍 401 | JWT 验证失败或 Resource Server 未配置 |
| 登录后仍未登录 | Session Cookie 没保存、跨域 Cookie 配置错、无状态配置导致不保存 |

## 9. 403 排查

403 表示认证了，但请求被拒绝。

排查清单：

1. 当前用户是否有目标权限？
2. `hasRole("ADMIN")` 实际需要 `ROLE_ADMIN`，数据库是否匹配？
3. 授权规则顺序是否写错？
4. POST/PUT/DELETE 是否缺 CSRF Token？
5. 方法级授权是否启用？
6. `@PreAuthorize` 表达式里的参数名是否能解析？
7. 自定义 `AuthorizationManager` 是否返回 false？

典型现象：

| 现象 | 可能原因 |
|---|---|
| GET 正常，POST 403 | 多半是 CSRF |
| admin 也 403 | 角色前缀或权限编码不匹配 |
| Controller 能进，Service 报 AccessDenied | 方法级授权拒绝 |
| Swagger 能打开，接口 403 | Swagger 静态资源放行了，但接口没权限 |

## 10. CORS 排查

浏览器控制台 CORS 报错时：

1. 看预检 `OPTIONS` 是否返回 200。
2. 看响应头是否有 `Access-Control-Allow-Origin`。
3. 带 Cookie 时看 `Access-Control-Allow-Credentials`。
4. `allowedOrigins` 是否是具体域名。
5. 请求头里是否有后端未允许的自定义头。
6. Spring Security 是否启用了 `http.cors()`。

不要只用 Postman 验证 CORS，因为 CORS 是浏览器安全模型。

## 11. 最佳实践

### 11.1 配置实践

1. 新项目使用 `SecurityFilterChain`，不要使用旧的 `WebSecurityConfigurerAdapter`。
2. 默认 `anyRequest().authenticated()`，明确公开接口。
3. API 项目自定义 JSON 401/403。
4. JWT API 设置 `STATELESS`。
5. 只对可信前端配置 CORS。
6. 不要为了调试长期关闭 CSRF、安全响应头和鉴权。

### 11.2 密码实践

1. 使用 `PasswordEncoderFactories.createDelegatingPasswordEncoder()` 或明确的强哈希算法。
2. 数据库存哈希值，不存明文。
3. 登录失败不要泄漏“用户名存在但密码错”这类可枚举信息。
4. 增加登录失败限流、验证码或账号锁定策略。

### 11.3 JWT 实践

1. Access Token 短有效期。
2. 不在 JWT 里放敏感隐私。
3. 使用非对称密钥时保护私钥，资源服务只拿公钥。
4. 设计密钥轮换。
5. 使用标准 `issuer-uri` 和 JWK Set。
6. 需要吊销时引入版本号、黑名单或刷新令牌存储。

### 11.4 授权实践

1. URL 层做入口控制。
2. Service 层做关键业务授权。
3. 权限编码稳定，使用 `resource:action`。
4. 菜单权限和接口权限分开设计。
5. 权限变化要考虑缓存失效。

### 11.5 运维实践

1. 安全日志记录登录成功、登录失败、登出、权限拒绝、token 刷新。
2. 敏感操作写审计日志。
3. 生产关闭详细错误堆栈。
4. Actuator 端点最小开放。
5. 全站 HTTPS。
6. Cookie 使用 `HttpOnly`、`Secure`、合理 `SameSite`。

## 12. 面试高频问题

### 12.1 Spring Security 的核心流程是什么

请求进入 Servlet 容器后，经 `DelegatingFilterProxy` 委托到 `FilterChainProxy`，匹配对应 `SecurityFilterChain`。过滤器链中认证过滤器尝试创建 `Authentication`，认证成功后放入 `SecurityContext`，授权过滤器根据请求和权限做决策，成功才进入 Controller。

### 12.2 认证和授权的区别

认证是确认“你是谁”，授权是确认“你能访问什么”。

### 12.3 hasRole 和 hasAuthority 的区别

`hasRole("ADMIN")` 默认检查 `ROLE_ADMIN`；`hasAuthority("ADMIN")` 精确检查 `ADMIN`。底层都是 `GrantedAuthority`。

### 12.4 为什么 POST 返回 403

可能是权限不足，也可能是 CSRF Token 缺失。若 GET 正常而 POST 403，优先查 CSRF。

### 12.5 JWT 为什么不能随便放敏感信息

JWT 默认只是编码和签名，不是加密。任何拿到 token 的人都能解码 payload。

### 12.6 Resource Server 做什么

Resource Server 保护 API，验证 Bearer Token 的签名、过期时间、签发者等，并把 token 中的 scope 或权限转换为 `GrantedAuthority`。

## 13. 本章小结

排障时按顺序问：

1. 请求是否进入了正确的 `SecurityFilterChain`？
2. 当前是否有 `Authentication`？
3. `Authentication` 是否已认证？
4. 权限集合是什么？
5. URL 规则是否匹配到预期规则？
6. 方法级授权是否拒绝？
7. 是 401、403、CSRF 还是 CORS？

能按这个顺序排查，Spring Security 的大部分问题都不会变成玄学。
