# 08-测试 调试 性能 与生产实践

> [!info] 本章抓什么
> WebFlux 能不能上线，关键不在“接口能跑”，而在测试、阻塞检测、超时、连接池、压测和可观测性。本章是从会写走向会维护。

## 1. Reactor 单元测试

`StepVerifier` 用来验证 `Mono` 或 `Flux` 的信号：

```java
@Test
void shouldReturnUser() {
    Mono<User> result = userService.getById(1L);

    StepVerifier.create(result)
            .expectNextMatches(user -> user.id().equals(1L))
            .verifyComplete();
}
```

验证异常：

```java
@Test
void shouldReturnErrorWhenUserMissing() {
    StepVerifier.create(userService.getById(999L))
            .expectErrorMatches(ex -> ex instanceof NotFoundException
                    && ex.getMessage().contains("用户不存在"))
            .verify();
}
```

验证时间：

```java
@Test
void shouldTimeout() {
    StepVerifier.withVirtualTime(() ->
            remoteService.slowCall().timeout(Duration.ofSeconds(3)))
            .thenAwait(Duration.ofSeconds(3))
            .expectError(TimeoutException.class)
            .verify();
}
```

## 2. WebTestClient 接口测试

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class UserControllerTest {
    @Autowired
    WebTestClient webTestClient;

    @Test
    void shouldGetUser() {
        webTestClient.get()
                .uri("/users/1")
                .exchange()
                .expectStatus().isOk()
                .expectBody()
                .jsonPath("$.id").isEqualTo(1)
                .jsonPath("$.name").isEqualTo("张三");
    }
}
```

测试 SSE：

```java
@Test
void shouldStreamEvents() {
    Flux<String> body = webTestClient.get()
            .uri("/events")
            .accept(MediaType.TEXT_EVENT_STREAM)
            .exchange()
            .expectStatus().isOk()
            .returnResult(String.class)
            .getResponseBody();

    StepVerifier.create(body.take(3))
            .expectNextCount(3)
            .verifyComplete();
}
```

## 3. 调试响应式链路

局部加日志：

```java
return userRepository.findById(id)
        .doOnSubscribe(s -> log.info("start query user id={}", id))
        .doOnNext(user -> log.info("found user={}", user.id()))
        .doOnError(ex -> log.error("query user failed id={}", id, ex))
        .doFinally(signal -> log.info("query user finished signal={}", signal));
```

链路命名：

```java
return userRepository.findById(id)
        .name("findUserById")
        .checkpoint("query user by id");
```

开发环境可开启更强调试，但生产慎用：

```java
Hooks.onOperatorDebug();
```

## 4. 阻塞检测

WebFlux 最大的线上坑是事件循环线程被阻塞。可以引入 BlockHound：

```xml
<dependency>
    <groupId>io.projectreactor.tools</groupId>
    <artifactId>blockhound</artifactId>
    <scope>test</scope>
</dependency>
```

测试中安装：

```java
@BeforeAll
static void setup() {
    BlockHound.install();
}
```

如果在非阻塞线程中调用 `Thread.sleep()`、JDBC、文件阻塞 I/O，BlockHound 能帮助发现。

> [!success] 测试建议
> 开发和测试环境尽早引入阻塞检测，比线上出现延迟尖刺后再排查要舒服很多。尤其是迁移老系统时，阻塞调用经常藏在工具类和第三方 SDK 里。

## 5. 生产性能原则

1. 事件循环线程不能阻塞。
2. 远程调用必须有超时。
3. `flatMap` 并发必须有上限。
4. 大集合不要随手 `collectList()`。
5. 连接池、线程池、队列长度都要可观测。
6. 限流、熔断、降级要放在服务边界。
7. 日志不要在高频流上逐条打 INFO。

> [!danger] 生产红线
> WebFlux 线上最常见的灾难组合是：事件循环阻塞、远程调用无超时、`flatMap` 无并发上限、重试无退避、响应体无大小控制。

## 6. 典型容量估算

假设接口每秒 2000 请求，每个请求平均调用 3 个下游，下游平均耗时 100ms。

阻塞模型粗估需要大量等待线程：

```text
并发等待数 = QPS * 平均等待时间
         = 2000 * 0.1 * 3
         = 600
```

WebFlux 不会让下游 100ms 变成 10ms，但它能避免 600 个工作线程一起阻塞等待。真正瓶颈会转移到：

1. 下游服务容量。
2. HTTP 连接池。
3. 数据库连接池。
4. CPU 序列化反序列化。
5. 内存缓冲。

## 7. 上线检查清单

| 检查项 | 要求 |
|---|---|
| starter | 不混入无意的 `spring-boot-starter-web` |
| 阻塞调用 | 明确识别并隔离到 `boundedElastic` |
| 远程调用 | 有超时、错误处理、连接池上限 |
| 重试 | 有最大次数、退避策略、幂等判断 |
| 数据访问 | 优先 R2DBC/Reactive MongoDB，JPA 不进入主链路 |
| 日志 | 有 traceId，错误有上下文 |
| 测试 | `StepVerifier` + `WebTestClient` 覆盖主流程 |
| 监控 | QPS、延迟、错误率、连接池、JVM、Netty 指标 |
| 压测 | 覆盖慢下游、大响应、连接耗尽、取消请求 |

## 8. 什么时候不要用 WebFlux

1. 团队刚接触 Spring Boot，业务只是普通 CRUD。
2. 数据访问主要依赖 JPA、MyBatis、阻塞式 JDBC。
3. 系统瓶颈是 CPU 计算，不是 I/O 等待。
4. 业务流程复杂但并发不高，响应式链路会降低可读性。
5. Java 21 虚拟线程已经足够解决当前阻塞并发问题。

成熟的技术判断不是“WebFlux 很高级所以全用”，而是“这条链路是不是 I/O 密集、非阻塞生态是不是完整、团队是否能维护”。

> [!quote] 架构判断
> 好的 WebFlux 项目不是“全链路都写成 Mono/Flux”，而是“在真正需要非阻塞和流式处理的地方，能清楚地控制资源、错误、背压和可观测性”。
