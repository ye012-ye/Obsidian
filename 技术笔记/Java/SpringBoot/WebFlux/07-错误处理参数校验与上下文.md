# 07-错误处理 参数校验 与上下文

> [!info] 本章抓什么
> WebFlux 里的异常不是同步调用栈上的普通抛出，它会沿响应式链路作为 `onError` 信号传播。本章重点是把错误、校验和 traceId 做成可观测、可维护的体系。

## 1. 错误也是信号

在 Reactor 中，一条流最终只能有三类终止方式：

1. 正常完成：`onComplete`
2. 异常失败：`onError`
3. 被取消：`cancel`

一旦出现 `onError`，这条流就结束了，不会继续发后续数据。

```java
return userRepository.findById(id)
        .switchIfEmpty(Mono.error(new NotFoundException("用户不存在")))
        .map(UserVO::from)
        .onErrorMap(DataAccessException.class,
                ex -> new BizException("数据库访问失败", ex));
```

## 2. 局部错误处理

降级：

```java
return remoteClient.getUser(id)
        .onErrorResume(ex -> localCache.getUser(id));
```

固定默认值：

```java
return configClient.getConfig()
        .onErrorReturn(Config.defaultConfig());
```

只记录日志，不吞异常：

```java
return orderService.create(command)
        .doOnError(ex -> log.error("创建订单失败, command={}", command, ex));
```

> [!warning] doOnError 提醒
> `doOnError` 只是旁路动作，适合打日志，不会把异常吃掉。要恢复流程用 `onErrorResume`，要转换异常用 `onErrorMap`。

## 3. 全局异常处理

Controller Advice：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(NotFoundException.class)
    public Mono<ResponseEntity<ApiError>> handleNotFound(NotFoundException ex) {
        ApiError body = new ApiError("NOT_FOUND", ex.getMessage());
        return Mono.just(ResponseEntity.status(HttpStatus.NOT_FOUND).body(body));
    }

    @ExceptionHandler(BizException.class)
    public Mono<ResponseEntity<ApiError>> handleBiz(BizException ex) {
        ApiError body = new ApiError("BIZ_ERROR", ex.getMessage());
        return Mono.just(ResponseEntity.badRequest().body(body));
    }
}
```

Spring Boot 3 可结合 `ProblemDetail`：

```java
@ExceptionHandler(NotFoundException.class)
public Mono<ProblemDetail> handleNotFound(NotFoundException ex) {
    ProblemDetail detail = ProblemDetail.forStatus(HttpStatus.NOT_FOUND);
    detail.setTitle("资源不存在");
    detail.setDetail(ex.getMessage());
    return Mono.just(detail);
}
```

## 4. 参数校验

请求 DTO：

```java
public record CreateUserRequest(
        @NotBlank(message = "姓名不能为空")
        String name,

        @Email(message = "邮箱格式不正确")
        String email
) {
}
```

Controller：

```java
@PostMapping("/users")
public Mono<UserVO> create(@Valid @RequestBody Mono<CreateUserRequest> body) {
    return body.flatMap(userService::create).map(UserVO::from);
}
```

如果 `@Valid @RequestBody Mono<T>` 的校验行为不符合预期，可以改成在链路中显式校验：

```java
@Component
public class ReactiveValidator {
    private final Validator validator;

    public ReactiveValidator(Validator validator) {
        this.validator = validator;
    }

    public <T> Mono<T> validate(T target) {
        Set<ConstraintViolation<T>> violations = validator.validate(target);
        if (violations.isEmpty()) {
            return Mono.just(target);
        }
        return Mono.error(new ConstraintViolationException(violations));
    }
}
```

使用：

```java
return body.flatMap(reactiveValidator::validate)
        .flatMap(userService::create);
```

## 5. Reactor Context

响应式链路不一定固定在同一个线程，传统 `ThreadLocal` 传递 traceId、用户信息会变得不可靠。Reactor 提供 `Context` 做链路上下文：

```java
public Mono<String> hello() {
    return Mono.deferContextual(ctx -> {
        String traceId = ctx.getOrDefault("traceId", "missing");
        return Mono.just("traceId=" + traceId);
    });
}
```

写入：

```java
return hello()
        .contextWrite(ctx -> ctx.put("traceId", UUID.randomUUID().toString()));
```

WebFilter 中写入：

```java
@Component
public class TraceWebFilter implements WebFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String traceId = Optional.ofNullable(exchange.getRequest().getHeaders().getFirst("X-Trace-Id"))
                .orElse(UUID.randomUUID().toString());

        return chain.filter(exchange)
                .contextWrite(ctx -> ctx.put("traceId", traceId));
    }
}
```

> [!success] 链路追踪建议
> WebFlux 项目里 traceId、租户、用户上下文尽量走 Reactor Context 或框架集成，不要只依赖裸 `ThreadLocal`。

## 6. 错误码设计

建议统一错误响应：

```java
public record ApiError(
        String code,
        String message,
        String traceId,
        Instant timestamp
) {
}
```

错误码分类：

| 前缀 | 含义 |
|---|---|
| `VALIDATION_` | 参数校验失败 |
| `AUTH_` | 认证授权失败 |
| `BIZ_` | 业务规则失败 |
| `REMOTE_` | 下游服务失败 |
| `DATA_` | 数据访问失败 |
| `SYSTEM_` | 未预期系统错误 |

## 7. 最佳实践

1. 业务可预期失败用明确异常，例如 `NotFoundException`、`BizException`。
2. 不要在 `doOnError` 里以为已经处理了异常。
3. 超时和重试要靠近远程调用边界。
4. 全局异常响应不要泄露堆栈、SQL、Token、内部地址。
5. traceId 要进入日志和错误响应，方便排查。
6. 响应式链路中跨线程上下文用 Reactor Context，不要裸用 ThreadLocal。

> [!danger] 错误响应红线
> 线上错误响应不要直接返回异常堆栈、SQL、内部 IP、Token、第三方密钥。对外给错误码和 traceId，对内用日志和链路追踪查细节。
