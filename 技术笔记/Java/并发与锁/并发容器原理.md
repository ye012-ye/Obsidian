# 并发容器原理

ConcurrentHashMap 是面试中**出现频率最高**的并发容器。

## HashMap 的线程安全问题

```mermaid

    A["HashMap 线程不安全"]
    A --> B["JDK 7: 并发扩容<br/>→ 链表成环 → 死循环 💀"]
    A --> C["JDK 8: 并发 put<br/>→ 数据覆盖丢失"]
    A --> D["size() 不准确"]
```

### JDK 7 死循环（头插法）

```mermaid
graph LR
    subgraph 扩容前["扩容前 bucket"]
        A["A"] --> B["B"] --> C["C"]
    end
    
    subgraph 扩容后头插法["并发扩容（头插法）"]
        B2["B"] --> A2["A"]
        A3["A"] --> B3["B"]
    end
    
    D["线程1和线程2同时扩容<br/>头插法导致 A→B→A 成环！<br/>get() 时死循环 💀"]
    
    style D fill:#ffcdd2
```

> JDK 8 改为**尾插法**，解决了死循环问题，但并发 put 仍可能数据丢失。

## ConcurrentHashMap

### JDK 7 实现：分段锁

```mermaid
graph TD
    subgraph ConcurrentHashMap_JDK7["JDK 7: Segment 分段锁"]
        S0["Segment[0]<br/>🔒 ReentrantLock"]
        S1["Segment[1]<br/>🔒 ReentrantLock"]
        S2["Segment[2]<br/>🔒 ReentrantLock"]
        S3["Segment[3]<br/>🔒 ReentrantLock"]
        
        S0 --> H0["HashEntry[]<br/>链表"]
        S1 --> H1["HashEntry[]<br/>链表"]
        S2 --> H2["HashEntry[]<br/>链表"]
        S3 --> H3["HashEntry[]<br/>链表"]
    end
    
    T1["线程1 操作 Segment[0]"] --> S0
    T2["线程2 操作 Segment[2]"] --> S2
    
    NOTE["不同 Segment 之间不互斥<br/>并发度 = Segment 个数（默认16）"]
    
    style NOTE fill:#fff9c4
```

**JDK 7 结构：**
- `Segment[]`（16个段，每个是 ReentrantLock）
- 每个 Segment 包含一个 `HashEntry[]`
- 操作不同 Segment 的线程互不影响
- **并发度 = Segment 个数 = 16**（固定，不可扩容）

### JDK 8 实现：CAS + synchronized（重点！）

```mermaid
graph TD
    subgraph ConcurrentHashMap_JDK8["JDK 8: Node 数组 + CAS + synchronized"]
        N0["Node[0]<br/>null"]
        N1["Node[1]<br/>链表头"]
        N2["Node[2]<br/>红黑树根"]
        N3["Node[3]<br/>null"]
        N4["Node[4]<br/>链表头"]
        
        N1 --> N1A["A"] --> N1B["B"] --> N1C["C"]
        N2 --> TREE["🌳 红黑树<br/>（链表长度≥8 且 数组≥64）"]
        N4 --> N4A["D"] --> N4B["E"]
    end
    
    T1["线程1 操作 Node[1]<br/>synchronized(Node[1]头节点)"] --> N1
    T2["线程2 操作 Node[4]<br/>synchronized(Node[4]头节点)"] --> N4
    
    NOTE2["锁粒度：单个桶（Node数组的一个位置）<br/>并发度 = 数组长度（可动态扩容！）"]
    
    style NOTE2 fill:#a5d6a7
```

### 关键细节

**为什么 put 时空桶用 CAS，非空桶用 synchronized？**

```mermaid

    D["桶不为空"] --> E["有头节点可以作为锁对象"]
    E --> F["synchronized(头节点)<br/>只锁当前桶，粒度最小"]

    style F fill:#c8e6c9
```

### size() 怎么保证准确？

```mermaid
graph TD
    A["ConcurrentHashMap 计数"]
    A --> B["类似 LongAdder 的思路"]
    B --> C["baseCount + CounterCell[]"]
    C --> D["put/remove 时<br/>CAS 更新 baseCount<br/>失败则更新 CounterCell"]
    D --> E["size() = baseCount + ΣCounterCell"]
```

### JDK 7 vs JDK 8 对比

| 对比 | JDK 7 | JDK 8 |
|------|-------|-------|
| **数据结构** | Segment[] + HashEntry[] + 链表 | Node[] + 链表/红黑树 |
| **锁机制** | ReentrantLock（Segment 锁） | **CAS + synchronized（桶锁）** |
| **锁粒度** | Segment（16个段） | **单个桶（数组每个位置）** |
| **并发度** | 固定 16 | **等于数组长度（动态增长）** |
| **查询复杂度** | O(n)（链表） | O(log n)（红黑树） |
| **扩容** | Segment 内单线程 | **多线程协助扩容** |
| **计数** | 分段统计 | baseCount + CounterCell |

## BlockingQueue（阻塞队列）

### 阻塞特性

```mermaid
graph TD
    subgraph 阻塞队列
        Q["BlockingQueue"]
    end
    
    P["生产者"] -->|"put(e)<br/>队列满时阻塞"| Q
    Q -->|"take()<br/>队列空时阻塞"| C["消费者"]
    
    style Q fill:#e3f2fd
```

| 操作方式 | 抛异常 | 返回特殊值 | 阻塞 | 超时 |
|----------|--------|-----------|------|------|
| **插入** | add(e) | offer(e) | **put(e)** | offer(e, time) |
| **移除** | remove() | poll() | **take()** | poll(time) |
| **检查** | element() | peek() | - | - |

### 常用实现

```mermaid
graph TD
    A["BlockingQueue 实现"]
    A --> B["ArrayBlockingQueue<br/>有界数组，一把锁"]
    A --> C["LinkedBlockingQueue<br/>有界链表，两把锁（读写分离）"]
    A --> D["SynchronousQueue<br/>零容量，直接传递"]
    A --> E["PriorityBlockingQueue<br/>优先级排序，无界"]
    A --> F["DelayQueue<br/>延迟队列"]
```

### ArrayBlockingQueue vs LinkedBlockingQueue

| 特性 | ArrayBlockingQueue | LinkedBlockingQueue |
|------|-------------------|---------------------|
| **存储** | 数组（固定大小） | 链表 |
| **有界** | 必须指定容量 | 默认 Integer.MAX_VALUE |
| **锁** | **一把锁**（put/take 互斥） | **两把锁**（putLock + takeLock） |
| **并发** | 低 | **高**（读写分离） |
| **内存** | 预分配 | 动态分配 |

## CopyOnWriteArrayList

```mermaid
graph TD
    A["CopyOnWriteArrayList"]
    A --> B["读操作：直接读<br/>不加锁 ⚡"]
    A --> C["写操作：复制整个数组<br/>在副本上修改<br/>然后替换原数组"]
    
    C --> D["写时复制<br/>Copy-On-Write"]
    
    style B fill:#a5d6a7
    style C fill:#ffcc80
```

```mermaid
sequenceDiagram
    participant W as 写线程
    participant R as 读线程
    participant ARR as 数组引用

    Note over ARR: array = [A, B, C]
    
    R->>ARR: 读取 array[1] = B（无锁 ✅）
    
    W->>W: 1. 复制数组 newArr = [A, B, C]
    W->>W: 2. 在 newArr 上修改 newArr[1] = X
    W->>ARR: 3. array = newArr（原子替换引用）
    
    R->>ARR: 仍然可能读到旧值 B（最终一致性）
```

**适用场景：读多写少**（如配置列表、白名单）。
**不适用：写多、数据量大（每次写都复制整个数组）。**

## 面试高频问题

### Q1：ConcurrentHashMap JDK 7 和 8 的区别？

JDK 7 用 Segment 分段锁（ReentrantLock），并发度固定 16。JDK 8 改为 CAS + synchronized 锁桶，并发度等于数组长度，还支持链表转红黑树和多线程协助扩容。

### Q2：ConcurrentHashMap 的 put 流程（JDK 8）？

1. hash 计算定位桶
2. 桶为空 → CAS 插入
3. 正在扩容 → 帮助扩容
4. 桶不为空 → synchronized(头节点) 后遍历链表/红黑树插入
5. 链表长度 ≥ 8 且数组 ≥ 64 → 转红黑树

### Q3：ConcurrentHashMap 为什么用 synchronized 而不是 ReentrantLock？

1. JDK 6+ synchronized 经过大量优化（锁升级），性能接近 ReentrantLock
2. synchronized 由 JVM 实现，能被 JIT 进一步优化（锁消除/粗化）
3. 内存占用更小（不需要额外的 AQS Node 对象）

### Q4：ConcurrentHashMap 的 key 和 value 能不能为 null？

**都不能！** 与 HashMap 不同。
原因：多线程环境下无法区分"key 不存在"和"key 的 value 是 null"的情况。`get(key) == null` 到底是没有这个 key 还是 value 就是 null？有二义性。
