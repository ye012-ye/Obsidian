# 性能调优、监控与生产避坑

> [!tip] 本章目标
> 你要能判断 Netty 服务到底慢在哪里：线程、内存、网络、协议、业务还是系统参数。

## 生产红线

> [!danger] 这些事很危险
> - EventLoop 中执行阻塞调用。
> - ByteBuf 消费后不释放。
> - 没有限制单连接读写速率。
> - 没有限制最大帧长度。
> - 写缓冲积压还继续无限写。
> - 客户端断线重连没有退避。
> - 所有连接共用一个有状态 Handler。
> - TLS、压缩、序列化都在 I/O 线程里做重活。

## 线程调优

默认 `NioEventLoopGroup()` 会按 CPU 核数生成线程。一般先用默认，再通过压测调整。

业务耗时任务要丢到业务线程池：

```java
DefaultEventExecutorGroup businessGroup = new DefaultEventExecutorGroup(16);
pipeline.addLast(businessGroup, "business", new BusinessHandler());
```

> [!warning] 不是线程越多越快
> EventLoop 线程太多会增加上下文切换。业务线程太多会打爆下游数据库或缓存。先量化瓶颈再调。

## 写缓冲水位

```java
bootstrap.childOption(ChannelOption.WRITE_BUFFER_WATER_MARK,
        new WriteBufferWaterMark(32 * 1024, 64 * 1024));
```

当写缓冲超过高水位，Channel 会变成不可写。

```java
if (ctx.channel().isWritable()) {
    ctx.writeAndFlush(msg);
}
```

## native transport

Linux 上可以考虑：

1. `netty-transport-native-epoll`
2. `netty-transport-native-io_uring`

Netty 4.2 迁移指南提到，io_uring transport 已从 incubator 毕业为一等支持模块。

## 4.2 迁移提醒

Netty 4.2 迁移重点：

1. 推荐先升级到最新 4.1.x。
2. 使用 BOM 管理版本，避免 4.1 和 4.2 混在 classpath。
3. 客户端 TLS 主机名校验默认变化，要显式确认。
4. 默认 allocator 变更为 adaptive，升级时监控内存和 GC。

> [!success] 稳妥升级路径
> 先依赖收敛，再灰度，再压测，再看 direct memory、连接数、P99、错误率。底层网络库升级不要赌运气。

## 监控指标

建议看：

1. 当前连接数。
2. 每秒读写字节。
3. 每秒消息数。
4. EventLoop 任务积压。
5. direct memory 使用。
6. ByteBuf 泄漏日志。
7. 写缓冲不可写次数。
8. 重连次数。
9. P95/P99 处理耗时。

## 本章小结

> [!info] 性能心法
> Netty 的性能来自“少阻塞、少拷贝、少锁、可控内存、可控连接”。任何一个不可控，都会把高性能变成高风险。
