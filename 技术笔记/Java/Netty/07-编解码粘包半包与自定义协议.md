# 编解码、粘包半包与自定义协议

> [!tip] 本章目标
> 你要知道 TCP 没有“消息边界”，所以应用层协议必须自己定义边界。

## TCP 是字节流

你发送：

```text
hello
world
```

服务端可能收到：

```text
helloworld
```

也可能收到：

```text
he
llowo
rld
```

这就是粘包和半包。

> [!danger] 粘包不是 Netty bug
> TCP 本来就是流协议，不保证一次 write 对应一次 read。只要写 TCP 服务，就必须处理消息边界。

## 常见拆包方案

| 方案 | Netty 支持 | 适合 |
|---|---|---|
| 固定长度 | `FixedLengthFrameDecoder` | 固定报文 |
| 分隔符 | `DelimiterBasedFrameDecoder` / `LineBasedFrameDecoder` | 文本协议 |
| 长度字段 | `LengthFieldBasedFrameDecoder` | 自定义二进制协议 |

## 推荐自定义协议

```text
magic      4 bytes  魔数
version    1 byte   协议版本
type       1 byte   消息类型
requestId  8 bytes  请求 ID
length     4 bytes  body 长度
body       N bytes  JSON / Protobuf / MessagePack
```

## LengthFieldBasedFrameDecoder

```java
new LengthFieldBasedFrameDecoder(
        1024 * 1024,
        14,
        4,
        0,
        0
);
```

参数含义：

1. 最大帧长度。
2. 长度字段偏移。
3. 长度字段占几个字节。
4. lengthAdjustment。
5. initialBytesToStrip。

> [!warning] 这个类很强，也很容易配错
> 配错偏移或剥离长度，会出现解码错位。写自定义协议时先画出字节布局，再算参数。

## 自定义 Decoder 思路

```java
public class MessageDecoder extends ByteToMessageDecoder {
    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < 18) {
            return;
        }

        in.markReaderIndex();
        int magic = in.readInt();
        byte version = in.readByte();
        byte type = in.readByte();
        long requestId = in.readLong();
        int length = in.readInt();

        if (in.readableBytes() < length) {
            in.resetReaderIndex();
            return;
        }

        byte[] body = new byte[length];
        in.readBytes(body);
        out.add(new Message(version, type, requestId, body));
    }
}
```

## 本章小结

> [!success] 协议设计心法
> 先解决边界，再解决语义。没有可靠拆包，后面的登录、心跳、消息推送都是空中楼阁。
