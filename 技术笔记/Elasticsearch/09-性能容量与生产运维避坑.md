# 性能、容量与生产运维避坑

> [!tip] 本章目标
> 你要知道 ES 为什么在线上变慢、爆内存、写不进去、查不动，以及怎么提前设计。

## 常见事故清单

| 事故 | 表现 | 常见根因 |
|---|---|---|
| 搜索突然很慢 | P95/P99 飙升 | 深分页、复杂 wildcard、字段太多 |
| 写入积压 | Bulk 失败、队列堆积 | bulk 太大、refresh 太频繁、磁盘压力 |
| 内存上升 | GC 频繁 | 聚合字段不当、fielddata、请求太重 |
| shard 热点 | 单节点压力高 | 路由不均、分片设计不合理 |
| mapping 爆炸 | 集群状态变大 | 动态字段无限增长 |
| 搜不到新数据 | 延迟可见 | refresh 机制导致近实时 |

## refresh 不是 commit

ES 是近实时搜索。写入后默认不是立刻一定能搜到，而是等 refresh 后可见。

> [!warning] 不要每条都强制 refresh
> `refresh=true` 会让数据更快可见，但频繁使用会伤写入性能。除非测试或强需求，否则别在高频写入链路滥用。

## 深分页

```text
from = 100000
size = 20
```

这不是“只查 20 条”，而是要处理前面大量候选再丢掉。

> [!success] 替代方案
> 普通页码：`from + size`。  
> 无限滚动：`search_after`。  
> 大批量导出：根据版本和场景选择 scroll / point in time / search_after。

## Bulk 调优

Bulk 关注三个指标：

1. 每批文档数。
2. 每批请求体大小。
3. 失败率和重试策略。

> [!example] 调优方法
> 从小批量开始压测，比如 500 条或 5MB 一批；观察 CPU、heap、磁盘 IO、写入延迟和 reject；再逐步增大，不要直接上来 5 万条一批。

## 索引别名与重建索引

Mapping 改错后，经常要重建索引：

```mermaid
graph LR
    A["products_v1"] --> B["创建 products_v2"]
    B --> C["全量 reindex / 业务重放"]
    C --> D["校验数量和搜索结果"]
    D --> E["别名 products_current 切到 v2"]
    E --> F["保留 v1 回滚窗口"]
```

> [!success] 推荐命名
> 物理索引：`products_v1`、`products_v2`。  
> 读写别名：`products_current`。  
> 业务代码尽量访问别名，不直接绑死物理索引。

## Java 项目里的稳定性守护

1. 客户端设置超时。
2. 搜索参数限制最大 `size`。
3. 禁止用户直接传任意 DSL。
4. 对 Bulk 失败做局部重试和死信记录。
5. 对 ES 异常返回友好降级。
6. 记录搜索关键词、耗时、命中数、错误。

> [!danger] ES 不是垃圾桶
> 不要把所有 JSON 原样扔进 ES。字段爆炸、无用字段过多、超大文档、动态 mapping 失控，都会让集群慢慢变成难维护的泥潭。

## 本章小结

> [!info] 生产心法
> ES 快不是魔法，是索引设计、查询约束、批量写入、资源容量、监控告警共同换来的。
