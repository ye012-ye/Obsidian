# 面试题与大神修炼清单

> [!tip] 本章目标
> 不只背答案，而是形成“概念 -> 原理 -> 业务场景 -> 生产风险”的回答框架。

## 高频面试题

### 1. ES 为什么搜索快？

回答框架：

1. 基于 Lucene。
2. 使用倒排索引。
3. 文本会经过 analyzer 分词。
4. 查询时从 term 找文档集合，而不是全表扫描。
5. 分布式分片可以并行处理数据。

### 2. text 和 keyword 有什么区别？

`text` 会分词，适合全文搜索；`keyword` 不分词，适合精确匹配、过滤、排序和聚合。

### 3. match 和 term 有什么区别？

`match` 会分析查询文本，适合全文搜索；`term` 不分析查询值，适合精确字段。

### 4. query 和 filter 有什么区别？

`query` 参与相关性评分；`filter` 不参与评分，常用于精确过滤，适合状态、分类、价格区间、权限。

### 5. 为什么不建议深分页？

`from + size` 越深，需要跳过的候选越多，内存和 CPU 成本越高。可以用 `search_after` 等方式处理滚动翻页。

### 6. 为什么 ES 不适合作主库？

ES 主要面向搜索和分析，不擅长复杂事务、强一致、多表关系约束。常见架构是数据库做事实来源，ES 做搜索读模型。

### 7. Mapping 设计错了怎么办？

不能指望所有字段都能原地修改。常见做法是新建 `v2` 索引，重建数据，校验后通过别名切换。

### 8. Bulk 写入要注意什么？

控制批大小，处理局部失败，避免无限重试，监控 reject、耗时、失败率，必要时死信和补偿。

### 9. 中文搜索怎么优化？

从分词、同义词、字段权重、拼音、业务排序、搜索日志反馈几个方向优化。不要只靠一个 analyzer 解决所有问题。

### 10. ES 写入后为什么马上搜不到？

ES 是近实时搜索，写入后要等 refresh 才对搜索可见。频繁强制 refresh 会影响写入性能。

## 避坑清单

> [!danger] 线上红线
> - 不限制用户传入的 `size`。
> - 让前端直接传任意 DSL。
> - 对 `text` 字段做排序聚合。
> - 随意开启动态字段，导致 mapping 爆炸。
> - Bulk 失败不看 item 级错误。
> - 把 ES 当主库。
> - 修改 analyzer 却不重建索引。
> - 只看平均耗时，不看 P95/P99。

## 从入门到大神的路线

第一阶段：会用。

1. CRUD。
2. Bulk。
3. match、term、bool、range。
4. 分页、排序、高亮、聚合。

第二阶段：会设计。

1. Mapping。
2. Analyzer。
3. 索引别名。
4. 同步链路。
5. 业务排序。

第三阶段：会排障。

1. 慢查询。
2. 写入失败。
3. 深分页。
4. refresh 延迟。
5. shard 和容量问题。

第四阶段：会做搜索产品。

1. 搜索日志。
2. 无结果词分析。
3. 相关性调优。
4. 同义词迭代。
5. 个性化排序。
