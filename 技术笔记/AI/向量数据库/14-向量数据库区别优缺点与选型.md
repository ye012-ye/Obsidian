# 向量数据库区别优缺点与选型

> [!info] 模块导航
> 上级：[[常用向量数据库使用方法与最佳实践]]；对比选型：[[向量数据库区别优缺点与选型]]。

## 17.1 一句话选型

| 数据库 | 一句话定位 |
|---|---|
| Pinecone | 托管优先，快速上生产，少运维 |
| Milvus / Zilliz | 大规模向量检索，开源自建或托管都可 |
| Qdrant | 工程友好，payload 过滤清晰，RAG 很顺手 |
| Weaviate | 对象模型和 hybrid search 友好 |
| pgvector | PostgreSQL 用户的低引入成本方案 |
| Elasticsearch | 已有 ES 搜索体系时加入语义搜索 |
| OpenSearch | 开源搜索生态里的向量检索 |
| Redis | 热数据、低延迟、内存型向量检索 |
| Chroma | 本地开发、原型、轻量 RAG |
| LanceDB | 嵌入式、多模态、数据湖和本地优先 |
| MongoDB Atlas Vector Search | MongoDB Atlas 文档数据就地向量搜索 |
| FAISS | 高性能向量检索库，不是完整数据库 |

## 17.2 优缺点对比

| 方案 | 优点 | 缺点 |
|---|---|---|
| Pinecone | 托管省心，API 简单，生产化能力强 | 成本和底层可控性受云服务限制 |
| Milvus | 大规模能力强，索引类型丰富，开源生态成熟 | 自建运维复杂，小项目偏重 |
| Qdrant | API 直观，payload filter 强，部署相对轻 | SQL 和复杂分析能力弱于关系数据库 |
| Weaviate | hybrid search 强，对象模型清晰，RAG 生态好 | schema 和模块体系需要学习 |
| pgvector | 与 PostgreSQL 事务和 SQL 结合好，引入成本低 | 超大规模和高 QPS 不如专用向量库 |
| Elasticsearch | BM25、过滤、聚合、搜索生态成熟 | 集群运维重，纯向量场景可能成本高 |
| OpenSearch | 开源搜索生态，AWS 生态友好 | 调优复杂，版本和插件差异要关注 |
| Redis | 低延迟，适合热数据，和缓存生态结合好 | 内存成本高，不适合大量冷数据 |
| Chroma | 上手快，适合本地 RAG 和 demo | 大规模生产治理能力要谨慎评估 |
| LanceDB | 嵌入式简单，多模态和数据湖友好 | 在线高并发服务形态需要额外评估 |
| MongoDB Atlas | 文档和向量同库，适合 MongoDB 应用 | 依赖 Atlas，纯向量专项能力不是重点 |
| FAISS | 性能强，索引算法丰富，适合评测和自研 | 没有数据库治理能力，需要自己封装 |

## 17.3 按场景推荐

| 场景 | 优先考虑 |
|---|---|
| 快速上线企业 RAG | Pinecone、Qdrant Cloud、Weaviate Cloud |
| 私有化大规模向量检索 | Milvus、Qdrant、OpenSearch |
| 已有 PostgreSQL | pgvector |
| 已有 Elasticsearch / OpenSearch | Elasticsearch Vector Search、OpenSearch k-NN |
| 已有 MongoDB Atlas | MongoDB Atlas Vector Search |
| 热数据低延迟推荐 | Redis Vector Search |
| 本地开发和 demo | Chroma、LanceDB、FAISS |
| 多模态数据湖 | LanceDB、Milvus |
| 研究评测和自研索引 | FAISS |
| 需要强 metadata filter | Qdrant、Pinecone、Elasticsearch、MongoDB Atlas |
| 需要 hybrid search | Weaviate、Elasticsearch、OpenSearch、Pinecone、Chroma |

## 18. 生产落地检查清单

## 18.1 数据入库前

1. embedding 模型、维度、metric 是否固定。
2. chunk 策略是否经过人工抽样检查。
3. `doc_id`、`chunk_id`、`tenant_id`、`source` 是否必填。
4. 是否能根据 `doc_id` 删除和重建。
5. 是否保留原文位置和引用链接。
6. 是否有重复 chunk 去重策略。
7. 是否有模型版本字段。

## 18.2 查询阶段

1. 是否做了租户和权限过滤。
2. topK 是否足够大。
3. 是否需要 hybrid search。
4. 是否使用 reranker。
5. 是否记录 query、召回文档、score、最终答案。
6. 是否能回放线上 bad case。
7. 是否限制返回字段，避免网络开销。

## 18.3 运维阶段

1. 是否有备份和恢复演练。
2. 是否有索引构建监控。
3. 是否监控 QPS、P95、P99、错误率、内存、磁盘。
4. 是否监控召回质量，不只监控系统指标。
5. 是否有模型升级灰度方案。
6. 是否有数据删除和权限变更同步机制。
7. 是否有成本预算和容量规划。

## 19. 常见反模式

## 19.1 只建向量库，不建评测集

没有评测集就不知道召回质量有没有变好。

至少准备：

1. 50 到 200 个真实用户问题。
2. 每个问题的理想答案或相关文档。
3. topK 命中率。
4. rerank 后命中率。
5. LLM 最终答案评分。

## 19.2 把权限交给大模型

错误做法：

```text
先召回所有租户文档，再告诉 LLM 不要回答无权限内容
```

正确做法：

```text
向量检索阶段就用 tenant_id / acl filter 限制候选集
```

## 19.3 文档更新后只更新原文，不更新向量

原文和向量必须有同步机制。

建议：

1. 原文更新触发异步 re-embedding。
2. 旧 chunk 标记版本或删除。
3. 新 chunk 使用稳定 id 或新版本 id。
4. 检索时过滤 `status = active`。

## 19.4 一开始就追求复杂架构

小规模知识库可以从 Chroma、LanceDB、pgvector 起步。等数据量、并发、隔离、安全、监控要求变高，再迁移到 Qdrant、Milvus、Pinecone 等更专业方案。

不要因为“以后可能很大”一开始就引入过重集群。

## 19.5 只看向量库 benchmark

真实 RAG 效果由多环节共同决定：

1. 文档解析。
2. chunk 策略。
3. embedding 模型。
4. 向量索引。
5. metadata filter。
6. hybrid search。
7. reranker。
8. prompt 拼接。
9. LLM 生成。

向量库性能好，不代表最终答案一定好。
